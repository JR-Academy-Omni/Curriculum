前两章我们聊了单个 Agent 是怎么跑起来的（Agent Fundamentals），也聊了它怎么调工具（Agent Tool Use）。这一章往前一步：当一个 Agent 扛不动一整个任务的时候，你怎么把它拆成一群 Agent 协作。

我先说结论，省得你读到最后才发现——**多 Agent 不是越多越好**。它是把一个大任务切开、分给几个专职小弟去干，然后有个头儿把结果拼回来。听着很美，代价也实打实。什么时候该拆、怎么拆、拆了怎么收场，这章讲清楚。

## 为什么要懂 / 不懂会怎样

2026 年你手上的工具——Claude Code、Cursor、Codex CLI——早就不是「你问一句它答一句」了。它们能自己开子任务、自己调工具、自己跑一长串活。你迟早会遇到一个任务大到单个 Agent 上下文塞不下，或者它干着干着就跑偏、把前面做的事忘了。

不懂多 Agent 编排的人，遇到这种情况只会一个劲儿往一个 Agent 里塞更多指令、更多上下文，结果就是它越来越糊涂、输出越来越水。懂的人知道：这时候该把活拆开，让每个 Agent 只盯自己那一小块。

反过来也一样。有人学了这套就上头，什么破事都要拆五个 Agent，结果 token 烧成三倍、协调乱成一团、还多出好几个能出错的地方。这也是不懂——不懂什么时候**不该**拆。

## 编排者 + 工作者：核心模式

最常见、最好用的一个模式，叫**编排者 + 工作者**（Orchestrator + Workers）。

打个生活里的比方。你开一家小餐馆，来了个大订单。主厨（编排者）不亲自炒每一道菜，他干的是：看订单、把菜拆开、凉菜交给凉菜师傅、热菜交给炒锅、甜点交给点心师，最后菜都好了他检查装盘、一起出。每个师傅只管自己那口锅，不用管别人在忙啥。

放到 Agent 上就是这样：

```
                  ┌─────────────────┐
                  │   编排者 Agent    │
                  │  Orchestrator    │
                  │ ① 拆任务          │
                  │ ② 分派            │
                  │ ④ 汇总/收尾       │
                  └───┬────┬────┬────┘
             分派 ↓    ↓    ↓
        ┌─────────┐┌─────────┐┌─────────┐
        │ Worker A ││ Worker B ││ Worker C │
        │ 调研     ││ 写方案   ││ 审校     │
        │(专职+专用工具)         │         │
        └────┬────┘└────┬────┘└────┬────┘
             └──────────┴──────────┘
                   结果回传 ③
                        ↓
                  编排者汇总成最终产物
```

编排者本身不干具体的脏活，它的价值全在**拆、派、拼**这三件事上。工作者只干一件事，干完把结果甩回去。

**术语：编排者（Orchestrator）**
- 一个负责拆任务、分派、收结果的 Agent，自己不做具体执行。
- 像餐馆主厨——不炒菜，但决定谁炒哪道、最后统一装盘。
- 工作中：给它一个复杂目标，让它自己判断切成几块、每块派给谁。
- 最常踩的坑：让编排者又拆活又亲自下场干活，角色混了，上下文照样炸。

**术语：工作者（Worker）**
- 只负责一个专职子任务的 Agent，有自己的 system prompt 和自己那套工具。
- 像专职师傅——凉菜师傅不碰炒锅，只把凉菜做到最好。
- 工作中：给它一块界限清楚的活 + 只给它需要的工具，别把全套工具都塞给它。
- 最常踩的坑：给的任务边界模糊，工作者不知道自己该管到哪，越界去动别人的活。

## 拆多 Agent 到底图什么

值得拆的三个真实理由：

**1. 上下文隔离。** 每个工作者只带自己那一摊的上下文，互不污染。调研 Agent 的一堆搜索结果，不会挤进写方案 Agent 的脑子里干扰它。这是拆分最大的好处——你等于给每个子任务开了一块干净的白板。单 Agent 干到后面上下文一团乱，就是因为所有东西都堆在一块了。

**2. 专职专用。** 不同工作者可以有完全不同的 system prompt 和工具集。审校 Agent 的 prompt 可以专门写「你是个挑刺的编辑，只找逻辑漏洞和事实错误」，还只给它读文件的工具、不给它写文件的权限。这种专门化，单个万能 Agent 做不到——你没法让一个 prompt 同时既是热情的调研员又是刻薄的审校。

**3. 可并行。** 互不依赖的子任务可以同时跑，省时间。三个 Worker 一起动，墙上时钟只走一份活的时间。

## 串行 vs 并行：哪些能同时跑

这是拆活时最关键的判断：**子任务之间有没有依赖。**

- **能并行**：任务 A 和任务 B 谁都不用等谁。比如「调研竞品 X」和「调研竞品 Y」——两个 Worker 同时开跑，互不相干。
- **必须串行**：后一个任务的输入，是前一个任务的输出。比如「写方案」必须等「调研」出结果才能开始，「审校」又必须等「方案」写完。这条线只能一个接一个来。

```
串行（有依赖，只能排队）：
  调研 ──→ 写方案 ──→ 审校 ──→ 交付

并行（无依赖，可同时）：
  ┌─→ 调研竞品 A ─┐
  ├─→ 调研竞品 B ─┤──→ 编排者汇总
  └─→ 调研竞品 C ─┘
```

现实里往往是混着的：先并行调研三家竞品，汇总后再串行地写方案、审校。编排者要做的就是画出这张依赖图，能并的并、该串的串。

## 代价与取舍：先别急着拆

现在说重话。多 Agent 不是免费的，每多一个 Agent 你就多付三笔账：

- **多一份 token 成本**——每个 Agent 都要吃自己的 system prompt、自己的上下文，编排者还要花 token 去拆去拼。三个 Worker + 一个编排者，token 账单可能比单 Agent 干完翻好几倍。
- **多一层协调复杂度**——谁传给谁、结果什么格式、怎么合并，这些都得你设计好，出问题也更难排查。
- **多一个出错点**——四个 Agent 就有四个可能失败的地方，任何一个抽风都会拖垮整条链。

所以关键那句话——**先问：这些子任务能独立完成吗？**

如果子任务耦合得很紧、彼此要共享一大堆上下文，那拆开反而更糟：你得把同一份上下文复制给每个 Worker（token 更贵），Worker 之间还要来回传信息对齐（更慢更容易错）。这种时候，**单个 Agent 一口气干完，往往又快又省**。

一句糙话记住：**能一个人顺手干完的活，别硬拉三个人开会。**

## 交接、汇总，和某个 Worker 挂了怎么办

**交接（handoff）**：Worker 把结果甩回编排者时，格式要固定。别让它随便回一大段散文，最好约定好结构——比如「调研 Worker 只回一个要点列表」。这样编排者拼起来才不费劲。

**汇总**：编排者拿到几份结果后，不是简单拼接，而是要做整合——去重、对齐口径、把矛盾的地方标出来。这一步做不好，多 Agent 的产出会显得像几个人各写各的、拼在一起很分裂。

**Worker 失败怎么办**：这是新手最容易漏的。真实世界里 Worker 会挂——搜不到东西、超时、输出跑偏。编排者要有预案：

- **能重试的重试**：短暂失败（超时、限流）就让它再跑一次。
- **能降级的降级**：三家竞品调研挂了一家，先拿另外两家的结果往下走，标注「竞品 C 缺失」，别整条链卡死。
- **必须叫停的叫停**：如果挂的是关键前置（调研全挂了，方案没法写），就停下来报错，别硬着头皮让下游 Worker 凭空编。

## 落地：用 Claude Code sub-agents 拆一个任务

2026 年这套东西你不用自己从零搭，Claude Code 的 sub-agents 就是现成的编排者 + 工作者。你（或主 Agent）当编排者，派子任务给 sub-agent，它们各自带独立上下文去跑，跑完把结果回给你。

拿一个典型任务：「调研某个技术方案 + 写一份选型建议 + 审校」。你可以这样拆：

```
主 Agent（编排者）
  │
  ├─ 派 sub-agent「调研员」
  │     system prompt: 只搜集事实和数据，不下结论
  │     工具: 搜索、读网页
  │     产出: 一份要点清单
  │
  ├─（等调研回来，串行）派 sub-agent「方案作者」
  │     输入: 调研清单
  │     产出: 选型建议草稿
  │
  └─（等草稿，串行）派 sub-agent「审校」
        system prompt: 挑逻辑漏洞和没依据的断言
        工具: 只读
        产出: 修改意见
  ↓
主 Agent 汇总三方结果，出最终建议
```

注意这里三个子任务是**串行**的（每个都依赖上一个）。但如果你要调研的是五个方案，那「调研这五个」就可以并行开五个 sub-agent 同时跑，全回来了再进入串行的写作阶段。

关键设计动作就三个：给每个 sub-agent **界限清楚的活**、**只给它需要的工具**、**约定好回传格式**。做到这三点，编排就顺了。

## 常见坑

- **为拆而拆**：任务本来单 Agent 五分钟搞定，非要拆四个，结果慢一倍贵三倍。拆之前先问「能独立吗」。
- **编排者亲自下场干活**：角色混了，上下文照样爆。编排者就管拆派拼，别抢 Worker 的活。
- **工作者边界模糊**：没说清 Worker 该管到哪，它要么越界要么漏活。
- **不处理失败**：默认每个 Worker 都会成功——真上线一个 Worker 挂了整条链崩，还找不到是谁挂的。
- **交接格式随意**：Worker 回一大段散文，编排者拼得七荤八素。回传格式要提前约定。

## 小结

多 Agent 编排的核心就一句：**把一个大任务拆成几个专职小活，用编排者管拆派拼，用工作者各干一摊。**

拆的三个理由是上下文隔离、专职专用、可并行；代价是更贵、更复杂、更多出错点。所以每次动手前，先问那句灵魂拷问——**这些子任务能独立完成吗？** 能，就拆，享受隔离和并行的好处；不能（耦合紧、要共享大量上下文），就老老实实用单 Agent，反而更快更省。

依赖决定串并行：谁不等谁就并行，谁要吃谁的输出就串行。别忘了给 Worker 失败留预案——能重试重试、能降级降级、该叫停叫停。落到手上的活，用 Claude Code sub-agents 就能直接练：界限清楚、工具精简、格式约定好，三件事做到，一个人也能指挥一支 Agent 小队。
