# Agent Tool Use — 让 Agent 用对工具

上一章「Agent Fundamentals」里，我们把 Agent 拆成了「感知—思考—行动」的循环。这章我专门讲那个「行动」环节：**工具调用（tool use / function calling）**。

我自己最早写 Agent 的时候，脑子里想的是「模型很聪明，我把工具丢给它，它自然会用」。结果第一个 demo 就翻车了：我给了它 12 个工具，它查天气去调了发邮件的工具，还把城市名传进了收件人字段。不是模型笨，是我根本没告诉清楚每个工具是干嘛的、参数长什么样。

工具用得好不好，八成不取决于模型，取决于**你怎么描述工具**。这章就是教你把工具描述清楚、把边界设死。

---

## 为什么要懂这个 / 不懂会怎样

Claude Code、Cursor、Codex CLI 这些 2026 年天天在用的编码 Agent，本质都是「一个会调工具的模型」。它们能读文件、跑命令、改代码，靠的全是背后一套工具定义。

如果你自己写 Agent 却不懂工具设计，会踩这些坑：
- 模型总选错工具，你以为是模型蠢，其实是 description 写得含糊；
- 参数经常传错格式，程序崩了模型还一脸懵，因为你返回的错误它读不懂；
- 一个「删除数据库」的工具你设成能直接调用，某次模型误判，生产数据没了。

反过来，工具设计对了，一个中等能力的模型也能干得又稳又准。

---

## 一个工具调用的完整回合

先看清楚「模型调一次工具」到底发生了什么。它不是一步，是一个**闭环**：

1. 模型看到你的问题 + 可用工具列表，决定「我要调 `get_weather`，参数是 `{city: "Melbourne"}`」；
2. **运行时**（你的代码，不是模型）真正去执行这个函数，拿到结果；
3. 结果作为一条 `tool` 角色的消息**回填**给模型；
4. 模型基于这个结果继续——要么再调别的工具，要么给出最终答案。

关键点：**模型自己不会执行任何东西**。它只输出「我想调什么、传什么参数」，真正跑代码的是你的运行时。跑完把结果喂回去，它才能接着想。

用 JSON 看一遍这个来回：

```json
// 第 1 步：模型的输出——它决定调工具
{
  "role": "assistant",
  "content": [
    {
      "type": "tool_use",
      "id": "toolu_01A",
      "name": "get_weather",
      "input": { "city": "Melbourne" }
    }
  ]
}

// 第 2 步：你的运行时执行 get_weather("Melbourne")，拿到结果
// 第 3 步：把结果作为 tool_result 回填
{
  "role": "user",
  "content": [
    {
      "type": "tool_result",
      "tool_use_id": "toolu_01A",
      "content": "{\"temp\": 14, \"condition\": \"多云\"}"
    }
  ]
}

// 第 4 步：模型基于结果继续，这次给最终答案
{
  "role": "assistant",
  "content": [
    { "type": "text", "text": "墨尔本现在 14 度，多云，出门带件外套。" }
  ]
}
```

一个 Agent 任务往往是这个回合转好几圈：调工具、看结果、再调、再看，直到能回答为止。

## 模型靠什么选对工具

模型面对一堆工具，凭三样东西决定用哪个：

**术语：工具 description（工具描述）**
- 一句话解释：告诉模型「这个工具是干什么的、什么时候该用它」的自然语言说明。
- 形象比喻：像超市货架上的标签。标签写「进口零食」你找薯片就走过去了；标签只写「食品」，你站货架前一脸茫然。
- 工作中怎么用：写清楚**用途 + 触发场景 + 边界**，比如「查询某城市当前天气。仅用于实时天气，历史天气请用 `get_weather_history`」。
- 最常踩的坑：description 写得含糊（比如只写「天气工具」），这是**模型选错工具的头号原因**。模型不是读你的代码，它只读这行字。

第二样是**参数 schema**。它用 JSON Schema 约束每个参数的类型、格式、哪些必填。schema 越明确，模型越不容易传错。第三样是**工具数量**——工具太多会稀释模型的判断力，一次给它 30 个工具，它挑错的概率远高于给 5 个相关的。按当前任务场景只暴露相关工具，别一股脑全塞进去。

看一个把三样都写清楚的工具定义：

```json
{
  "name": "get_weather",
  "description": "查询某个城市的『当前实时』天气。适用于用户问『现在/今天天气怎么样』。历史天气或天气预报请勿使用此工具。",
  "input_schema": {
    "type": "object",
    "properties": {
      "city": {
        "type": "string",
        "description": "城市英文名，如 'Melbourne'、'Sydney'"
      },
      "unit": {
        "type": "string",
        "enum": ["celsius", "fahrenheit"],
        "description": "温度单位，默认 celsius"
      }
    },
    "required": ["city"]
  }
}
```

注意 `enum` 把 `unit` 限死在两个合法值里，`required` 标明 `city` 必须传。这些约束是在帮模型少犯错。

## ReAct：想一步、做一步、看一步

Agent 调工具不是盲目乱调，一个经典且好用的思路是 **ReAct**：**Reason（想）→ Act（调工具）→ Observe（看结果）**，循环往复。

**术语：ReAct**
- 一句话解释：让模型「先说出推理、再调工具、再观察结果」交替进行的一种 Agent 模式。
- 形象比喻：像修水管的师傅。他先想「应该是这段漏了」（Reason），拧开阀门试一下（Act），看水停没停（Observe），没停就换个判断再试。
- 工作中怎么用：让模型在每次调工具前先输出一句「我为什么调这个」，你调试时能一眼看出它哪步判断错了。
- 最常踩的坑：跳过 Observe——工具返回了结果，但你没把结果好好回填，模型就等于闭着眼睛在猜下一步。

一个查航班的例子，思路是这样走的：

```
Reason:  用户要订最便宜的墨尔本飞悉尼的票，我得先查航班列表。
Act:     调 search_flights({from: "MEL", to: "SYD", date: "2026-08-01"})
Observe: 返回 3 个航班，最便宜的是 JQ501，$89。
Reason:  找到最便宜的了，现在需要用户确认再下单。
Act:     调 confirm_with_user({message: "JQ501 $89，确认下单？"})
```

每一圈都「想清楚—动手—看结果」，错了也能在下一圈纠回来。

## 常见坑

**坑一：工具没做幂等，重试就重复副作用。** 网络抖一下、模型重试一次，`create_order` 被调了两遍，用户下了两个单。给有副作用的工具加个幂等键（比如 `idempotency_key`），同一个 key 只生效一次。

**坑二：失败只返回 `"error"`。** 模型收到光秃秃一个 error，完全不知道该怎么改，只能瞎试或者放弃。要返回**可行动的错误**——告诉它错在哪、怎么改。

**坑三：高风险操作能被直接调用。** 删除、支付、部署这类不可逆操作，如果模型能直接调，一旦它判断错就是事故。这类工具必须卡一道人工确认，或者拆成「先预演、再执行」两步。

对比一下坏错误和好错误：

```json
// ❌ 模型读完一脸懵，不知道怎么办
{ "error": "invalid input" }

// ✅ 可行动的错误：错在哪 + 怎么改
{
  "error": "invalid_date_format",
  "message": "date 必须是 YYYY-MM-DD 格式，你传的是 '2026/8/1'",
  "fix": "改成 '2026-08-01' 再试一次"
}
```

第二种错误里，模型直接知道下一步改 `date` 的格式重调就行——这就是「可自我修正的错误返回」。

---

## 给工具设计防错边界

把上面的坑反过来，就是三条工具设计准则：

1. **执行前确认**：高风险操作返回一个「待确认」状态，等人工点头（或再一轮模型确认）才真执行。
2. **参数校验**：在函数真正跑之前先校验参数，不合法就返回结构化的可行动错误，别让脏参数进到核心逻辑。
3. **可自我修正的错误返回**：错误里带上 `message` 和 `fix`，让模型下一轮能自己改对，而不是卡死。

这三条不是锦上添花，是让 Agent 从「demo 能跑」走向「敢放上生产」的分界线。

## 动手：为一个 Agent 设计 3 个工具

给一个「个人日程助手」Agent 设计 3 个工具，每个都写清 description + schema，并想清楚它的防错边界。我给你打个样，第一个我写完整，后两个你照着补：

```json
[
  {
    "name": "list_events",
    "description": "查询指定日期范围内的日程。用于用户问『我明天有什么安排』这类只读查询。",
    "input_schema": {
      "type": "object",
      "properties": {
        "start_date": { "type": "string", "description": "起始日期 YYYY-MM-DD" },
        "end_date":   { "type": "string", "description": "结束日期 YYYY-MM-DD" }
      },
      "required": ["start_date", "end_date"]
    }
  }
  // TODO 你来补：
  // 2. create_event —— 有副作用，记得想幂等 + 参数校验
  // 3. delete_event —— 高风险，记得设执行前确认
]
```

补的时候问自己三个问题：description 够不够让模型「一眼知道什么时候用它」？schema 有没有把非法参数挡在门外？有副作用/高风险的工具，防错边界设了没？

## 小结

- 一次工具调用是个闭环：**模型出意图 → 运行时执行 → 结果回填 → 模型继续**，模型自己从不执行代码。
- 模型选对工具靠三样：**清晰的 description、明确的参数 schema、克制的工具数量**。含糊的 description 是选错工具的头号原因。
- **ReAct**（想—做—看）让 Agent 每步都可追、错了能纠。
- 三个必防的坑：**没做幂等**（重试重复副作用）、**只返回 error**（模型没法改）、**高风险操作能直调**（该加人工确认）。
- 工具设计的三条边界：**执行前确认、参数校验、可自我修正的错误返回**——这是从 demo 到生产的分界线。

下一步，试着把你现在用的某个 Agent（哪怕是 Claude Code 里的一个自定义工具）的 description 拿出来读一遍：如果你自己都要猜它什么时候该被调用，那模型也在猜。
