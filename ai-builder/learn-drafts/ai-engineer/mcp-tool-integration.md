前两章我们聊过 MCP 是什么、怎么把一个 MCP 服务器配进 Claude Code 或 Cursor。这章我要解决的是最后一公里，也是最容易翻车的一段：**工具接进去了，模型到底怎么找到它、怎么决定用它、怎么传参数、拿到结果又怎么接着往下走。**

我踩过一个很典型的坑。我给团队写了个查订单的 MCP tool，本地测服务器跑得好好的，`tools/list` 也能列出来。结果一到实战，模型要么根本不调，要么调了传个乱七八糟的参数。查了半天发现问题不在代码，在我那句敷衍的 description。所以这章我想让你明白一件事：**MCP 集成的成败，八成不在服务器写得对不对，而在你给模型的「说明书」写得清不清楚。**

## 集成的四个环节：模型是怎么用上你工具的

先把整个链路拆开。模型用一个工具，永远是这四步，缺一不可：

1. **发现（tools/list）** —— 客户端启动时向 MCP 服务器要一份工具清单。服务器返回每个工具的 name、description、inputSchema。这份清单会被塞进模型的上下文，模型这时才「知道」有哪些工具可用。
2. **选择** —— 模型根据你的自然语言需求，对照每个工具的 description，判断该用哪个（或者一个都不用）。**它靠的就是 description 那几句话，没有别的信息。**
3. **调用（tools/call）** —— 模型决定用某个工具后，按 inputSchema 生成一段参数 JSON，客户端把「工具名 + 参数」发给服务器执行。
4. **回填** —— 服务器返回结构化结果，客户端把结果塞回模型上下文。模型拿到结果后，要么直接给你答案，要么发现还得再调一次（比如先查用户 ID、再拿 ID 查订单）。

这四步里，第 1、3、4 步基本是协议和代码在跑，你控制不了太多。**你真正能左右模型行为的，是第 2 步——而它完全取决于第 1 步你喂进去的 description 和 schema。** 这就是为什么后面我会反复强调「写清楚」。

## Tool 和 Resource：一个是动手，一个是查资料

MCP 里有两类东西经常被搞混，但边界其实很清楚：

**术语：Tool（工具）**
- 一句话解释：模型主动调用、会产生副作用（改数据、发请求、触发动作）的能力。
- 形象比喻：像给助理一双手——「去把这单退款」「去建个日历事件」。
- 工作中怎么用：查订单、发邮件、创建工单、部署服务，这些都是 Tool。
- 最常踩的坑：把只读查询也当 Tool 无所谓，但把有副作用的写操作当成「顺手做做」——没加确认，模型一激动就真删了数据。

**术语：Resource（资源）**
- 一句话解释：只读的上下文数据，模型可以引用但不会改变任何东西。
- 形象比喻：像桌上摊开的参考手册——随便看，但看了不会把手册改掉。
- 工作中怎么用：一份产品文档、一段数据库 schema、一个配置文件的当前内容。
- 最常踩的坑：把「查订单」这种带查询参数、每次结果不同的动作硬塞成 Resource——它其实更适合做 Tool，因为它需要模型传参、主动触发。

简单判断法：**这个操作会不会「改变世界」或者「需要模型传参数去执行」？会 → Tool；只是把已有内容摊给模型看 → Resource。** 这章我们聚焦 Tool，因为集成的复杂度基本都在这。

## 一个 Tool 定义长什么样

MCP 的 tool 定义就三个核心字段：`name`、`description`、`inputSchema`。inputSchema 是标准的 JSON Schema。给你一个完整例子——查订单：

```json
{
  "name": "get_order",
  "description": "根据订单号查询订单的当前状态、金额和物流信息。用于回答'我的订单到哪了''这单多少钱''发货了没'这类问题。只读，不会修改订单。如果订单号不存在会返回明确错误。",
  "inputSchema": {
    "type": "object",
    "properties": {
      "order_id": {
        "type": "string",
        "description": "订单号，格式为 ORD- 开头加 8 位数字，例如 ORD-20260615。区分大小写。"
      },
      "include_items": {
        "type": "boolean",
        "description": "是否返回订单内的商品明细。默认 false，只有用户明确问'买了哪些东西'时才设为 true。",
        "default": false
      }
    },
    "required": ["order_id"],
    "additionalProperties": false
  }
}
```

注意几个细节：

- **description 里说清了「什么时候用」和「用于回答哪类问题」**，不是干巴巴一句「查询订单」。这直接决定了第 2 步模型选不选它。
- **每个参数自己也有 description**，连格式（`ORD-` 开头）都写了。模型据此才能从你「帮我查下 ORD-20260615」里正确提取参数。
- **`required` 明确必填**，`additionalProperties: false` 防止模型瞎编字段。
- **`default` 给了保守默认值**——不该多返回的默认不返回。

## 为什么 description 要懂：含糊的说明是选错工具的头号原因

这是这章我最想让你记住的一条。模型选工具时，它不看你的代码、不看你的注释，**它眼里只有那份 tools/list 里的 description。** 你写得含糊，它就选错。

举个真实对比。假设你有两个工具：`get_order`（查单个订单）和 `list_orders`（查某用户的所有订单）。如果两个 description 都只写「查询订单」，模型面对「张三最近下了几单」时根本分不清该用哪个,大概率乱选。但如果分别写成：

- `get_order`: 「根据**单个订单号**查询该订单详情」
- `list_orders`: 「根据**用户 ID**查询该用户名下**所有**订单的列表」

模型立刻就知道「查某人所有单」该用后者。**description 的价值不在于描述工具本身，而在于帮模型把它和别的工具区分开。** 写的时候多想一句：「模型凭这句话，能不能和我其他工具区分开？」

**不懂会怎样**：你会遇到那种玄学 bug——工具明明能用，模型就是不调，或者调错。你去查代码查半天，其实改一句 description 就好了。这种时间浪费我见得太多了。

## 错误处理：返回「能自我修正的信息」，不是一句 error

模型有个很好的特性：只要你告诉它哪里错了、怎么改，它能自己重试修正。但前提是你的错误信息得**可行动**。

反面教材：

```json
{ "error": "failed" }
```

模型看到这个只能干瞪眼，它不知道是订单号错了、没权限、还是超时，只能放弃或者原样重试。

正确做法是按错误类型返回具体、可修正的信息：

```json
{
  "error": "invalid_order_id",
  "message": "订单号 'ORD-123' 格式不对，正确格式是 ORD- 加 8 位数字（如 ORD-20260615）。请向用户确认完整订单号。"
}
```

分场景该返回什么：

- **参数错误** → 说清哪个参数错了、正确格式是什么、让模型去跟用户确认。模型会重新问用户拿正确的值。
- **无权限** → 明确「当前身份无权查此订单，需要工单权限」。模型就不会傻乎乎重试，而是转告用户「这需要更高权限」。
- **超时 / 服务不可用** → 返回「上游服务暂时超时，可稍后重试」。模型知道这是临时故障，可能会重试一次或告诉用户稍等。

**核心心法**：把错误信息当成「写给模型看的下一步指令」，而不是「写给日志的技术堆栈」。

## 幂等和危险操作：写操作要能重复，删除要人来点头

这里有两条工程纪律，直接关系到会不会出事故。

**术语：幂等（Idempotent）**
- 一句话解释：同样的请求执行一次和执行十次，结果一样，不会重复产生副作用。
- 形象比喻：像电梯按钮，你连按五下，电梯也只来一次。
- 工作中怎么用：写操作（创建、更新）尽量带一个 `idempotency_key`（幂等键），服务器发现同 key 已处理过就返回上次结果，而不是再执行一遍。
- 最常踩的坑：模型因为超时没收到结果，会自动重试。如果你的创建订单接口不幂等，用户就会莫名其妙多出一单。

为什么模型场景下幂等特别重要？因为**模型重试是常态**——网络抖一下、结果没回填成功，它就再调一次。人手动操作重试概率低，模型高得多。所以凡是写操作，默认就要往幂等上设计。

至于**删除、支付、部署**这类不可逆或高风险动作，别指望幂等兜底，直接上人工确认。两种常见做法：

1. 工具本身分两步——先返回「即将删除 X，确认请再调一次带 `confirmed: true`」，模型会把这句转达给用户，等用户点头才真执行。
2. 客户端层面配置这类工具需要人工审批（Claude Code 里危险工具默认会弹确认）。

原则一句话：**只读随便调，可逆写操作做幂等，不可逆 / 花钱 / 上线的操作一律人来拍板。**

## 动手：把内部查订单 API 包成一个 MCP tool

假设你们后端有个现成接口 `GET /internal/orders/{id}`，返回订单 JSON。把它包成 MCP tool，核心就是「把上面那个 tool 定义注册进去 + 写好 handler」。用 TypeScript SDK 大概长这样：

```typescript
server.tool(
  "get_order",
  "根据订单号查询订单状态、金额、物流。只读。订单号格式 ORD- 加 8 位数字。",
  {
    order_id: z.string().describe("订单号，如 ORD-20260615"),
    include_items: z.boolean().default(false).describe("是否返回商品明细"),
  },
  async ({ order_id, include_items }) => {
    // 1. 参数校验，错了返回可修正信息
    if (!/^ORD-\d{8}$/.test(order_id)) {
      return {
        content: [{ type: "text", text:
          `订单号 '${order_id}' 格式不对，应为 ORD- 加 8 位数字。请向用户确认。` }],
        isError: true,
      };
    }
    // 2. 调真实内部 API
    const res = await fetch(`${API_BASE}/internal/orders/${order_id}`, {
      headers: { Authorization: `Bearer ${process.env.INTERNAL_TOKEN}` },
    });
    if (res.status === 404) {
      return { content: [{ type: "text", text:
        `订单 ${order_id} 不存在，请让用户核对订单号。` }], isError: true };
    }
    if (!res.ok) {
      return { content: [{ type: "text", text:
        `上游服务返回 ${res.status}，暂时不可用，可稍后重试。` }], isError: true };
    }
    // 3. 结构化回填，按 include_items 裁剪
    const order = await res.json();
    const payload = include_items ? order : { ...order, items: undefined };
    return { content: [{ type: "text", text: JSON.stringify(payload) }] };
  }
);
```

对着前面讲的检查一遍：description 说清了用途和格式（选对）、参数带 describe（传对）、三种错误分别返回可行动信息（自我修正）、只读所以不用管幂等。这就是一个能被模型「用对」的 tool。token 不进代码、走环境变量——这是红线。

## 常见坑

- **description 写给人看，不是写给模型看**：写成「订单查询工具 v2」这种没用，要写「什么时候该用、用来回答什么问题」。
- **一个工具塞太多功能**：又能查又能改又能删的「万能订单工具」，模型永远搞不清该传什么。拆成职责单一的小工具，模型选得准。
- **错误只返回 error 或者直接抛异常**：模型接不住，只能放弃。永远返回结构化、可行动的错误文本。
- **写操作没幂等，靠祈祷模型不重试**：模型一定会重试，早晚出重复数据。
- **危险操作不加确认**：删除 / 支付 / 部署裸奔，迟早出事故。

## 小结

MCP 集成的四个环节——发现、选择、调用、回填——里，**你唯一能大幅影响模型行为的杠杆就是那份工具说明**：清晰的 description 决定模型选不选、选不选对；精确的 inputSchema（每个参数都带 describe）决定它传不传对参数；可行动的错误信息让它能自我修正；幂等和人工确认则守住写操作和危险操作的安全底线。把这几条落到你写的每个 tool 上，你的工具就从「能列出来但不好使」变成「模型拿起来就用对」。下一步，去挑一个你们团队真实的内部只读 API，照着上面的模板包一个 tool，在 Claude Code 里试试它到底选不选得对。
