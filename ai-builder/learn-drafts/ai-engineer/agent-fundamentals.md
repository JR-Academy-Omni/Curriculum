## 先说人话：Agent 不是「更聪明的聊天」

我第一次用 Claude Code 的时候，脑子里的模型是错的。我以为它就是个「懂代码的 ChatGPT」——我问一句，它答一句，只不过答得比网页版准。

用了两天我才反应过来：完全不是这么回事。

我让它「把这个项目里所有用了旧版日期库的地方换成新的」，然后它自己去 `grep`、自己打开文件、自己改、改完自己跑测试、测试挂了它自己看报错再改回来。我全程没插一句话。这就不是「聊天」了——聊天是我说一句它说一句，而它是**自己在一个循环里，决定下一步干什么**。

这个「自己决定下一步」就是 Agent 和普通 LLM 调用最本质的区别。

一个最好记的公式：

**术语：Agent（智能体）**
- 一句话解释：LLM + 工具 + 循环 + 记忆，让模型在循环里自己决定下一个动作，直到任务完成。
- 形象比喻：普通 LLM 是「你问路，路人给你指个方向」；Agent 是「你把地址给司机，他自己看导航、自己拐弯、堵车了自己绕路，把你送到」。
- 工作中怎么用：交给它路径不确定的活——比如「修好这个 bug」「把这块代码从 A 框架迁到 B 框架」，你不用告诉它每一步怎么走。
- 最常踩的坑：把它当搜索引擎用，一次只问一个小问题，那你就浪费了它「自己跑循环」的能力，也白白多花了钱。

## 核心：感知 → 推理 → 行动 → 观察，循环起来

Agent 的心脏就是一个循环。每转一圈做四件事：

1. **感知（Perceive）**：看当前上下文——任务是什么、上一步干了啥、结果如何。
2. **推理（Reason）**：模型想一下，决定下一个动作该干嘛。
3. **行动（Act）**：真的去调工具——跑命令、读文件、改代码、搜网页。
4. **观察（Observe）**：看行动的结果，把结果塞回上下文，进入下一圈。

一直转，直到任务完成，或者撞上你设的停止条件。

用伪代码写出来，其实特别简单：

```python
def run_agent(task, tools, max_steps=15, budget=2.0):
    context = [task]          # 记忆：一开始只有任务本身
    steps = 0
    cost = 0.0

    while steps < max_steps and cost < budget:
        # 1. 感知 + 2. 推理：模型看着当前上下文，决定下一步
        decision = llm.decide(context, tools)
        cost += decision.token_cost

        # 模型自己判断：任务做完了吗？
        if decision.is_done:
            return decision.final_answer

        # 3. 行动：执行模型选的工具
        result = run_tool(decision.tool, decision.args)

        # 4. 观察：把结果写回记忆，进入下一圈
        context.append(decision.action)
        context.append(result)
        steps += 1

    return "触发停止条件（步数或预算超了），交还给人处理"
```

注意看：**决定用哪个工具的是模型，不是你写死的 `if-else`**。你只提供了工具箱（`tools`）和护栏（`max_steps`、`budget`），路怎么走是它自己选的。这就是「自治」。

## Agent vs Workflow vs 单次 Prompt

这三个东西天天被混着说，但用错场景代价很大。我用一个「点外卖」的类比来分：

- **单次 Prompt**：你直接跟朋友说「帮我拿桌上那杯咖啡」。一句话说清，一步到位，不需要中途反馈。
- **Workflow（工作流）**：你写好一张纸条——「先烧水 → 再放咖啡粉 → 倒水 → 搅拌」。步骤固定，谁来做、做几次，路径都一样，结果可预测。
- **Agent**：你说「我想喝点提神的东西」。它得自己看家里有啥（感知）、决定做美式还是泡茶（推理）、发现没咖啡豆了跑去买（行动）、尝一口不够浓再补（观察）。路径事先不知道。

**术语：Workflow（工作流）**
- 一句话解释：把任务拆成固定的几步，按预先定好的路径一步步执行，路径可预测。
- 形象比喻：像工厂流水线，每个工位干什么、下一步传给谁，都是提前排好的。
- 工作中怎么用：重复性、结构化的任务——比如「每天早上把这份 JSON 转成报表并发邮件」，步骤永远一样。
- 最常踩的坑：明明是固定流程，非要套个 Agent 去「智能」处理，结果又慢又贵还时好时坏。

一句话记牢：**Workflow 的步骤是你（工程师）定的；Agent 的步骤是模型（运行时）定的。**

| | 谁决定步骤 | 路径 | 适合 |
|---|---|---|---|
| 单次 Prompt | 你，一句话 | 无 | 一步能说清的活 |
| Workflow | 你，写死流程 | 固定、可预测 | 重复、结构化任务 |
| Agent | 模型，动态决定 | 不确定 | debug、迁移、跨模块改造 |

## 什么时候「不该」用 Agent

这条特别重要，因为现在满世界都在喊「上 Agent」，很多人上错了。

**只要满足这几个条件，就别用 Agent，用 Workflow 更好：**

- 步骤是固定的，每次都一样；
- 任务能一次说清楚，不需要模型中途拿主意；
- 你不需要看到中间结果再决定下一步。

举个我踩过的坑：我要「把 200 个 markdown 文件的 frontmatter 里加一个字段」。这活我一开始扔给 Agent，它一个个文件读、一个个想、一个个改，跑了十几分钟还花了不少 token。后来我用一个 20 行的脚本（Workflow），2 秒跑完，一分钱没花，而且每次结果完全一样。

**Agent 的自治是有成本的**：

- **更慢**——每一圈都要等模型推理一次；
- **更贵**——循环几圈，token 就翻几倍；
- **更难复现**——同样的任务跑两遍，走的路径可能不一样，出 bug 你都难 debug。

所以选择标准很简单：**路径确定 → Workflow；路径不确定、需要模型随机应变 → Agent。** 别为了「显得先进」硬上 Agent。

## 完成条件 vs 停止条件：别让它跑成死循环

Agent 最大的风险是**转不停**。它以为还没做完，一圈接一圈地跑，或者两个动作来回横跳（改了又改回来），把你的钱包和 API 额度烧穿。

所以必须给它两种「刹车」：

- **完成条件（Done）**：任务真的达成了——测试全过、目标文件都改完、问题解决了。这是「正常下班」。
- **停止条件（Stop / 护栏）**：还没做完但必须叫停——步数超上限、花费超预算、连续几步没进展。这是「拉闸保命」。

回头看上面伪代码里的 `max_steps` 和 `budget`，就是停止条件。**上线任何 Agent 之前，这两个护栏必须设。** 我的习惯是：先把 `max_steps` 设得偏小（比如 10 步）跑几次，看它一般几步能搞定，再往上调。

## Agent 到底需要哪几个部件

拆开看，一个能跑的 Agent 就三样核心部件：

1. **工具（Tools）**：Agent 的手脚。读写文件、执行 shell、搜网页、调 API……没有工具，模型只能「说」不能「做」。工具描述写得越清楚，模型选得越准。
2. **记忆 / 上下文（Memory / Context）**：Agent 的记事本。每一圈的动作和结果都要写回上下文，它才知道「我已经试过什么、现在到哪了」。上下文太长会爆窗口，所以现在很多做法是压缩历史或分段总结。
3. **停止条件（Stop）**：Agent 的刹车。上面刚讲过，防死循环、控成本。

2026 年这些部件很多已经被工具链封装好了。比如 Claude Code 的 **sub-agents**：主 Agent 可以派一个子 Agent 去专门啃某个子任务（比如「你去把测试全跑一遍并总结哪些挂了」），子 Agent 有自己独立的上下文，跑完把结论汇报回来。这本质上就是把「记忆」做了分层——主 Agent 的上下文不会被一堆测试日志塞爆。你不用自己从零实现循环，但**理解循环怎么转、护栏怎么设，你才知道它为什么慢、为什么绕、什么时候该拦它**。

## 为什么要懂这个 / 不懂会怎样

不懂 Agent 的循环机制，你会犯两类错：

- **该用 Workflow 的地方硬上 Agent**——固定任务被跑得又慢又贵还不稳定，你还纳闷「AI 怎么这么笨」。其实是你选错工具了。
- **不设护栏就放它跑**——某天它卡在一个死循环里，一晚上烧掉一大笔 API 费用，或者把不该动的文件改了一堆，你早上起来收拾残局。

反过来，懂了之后你会很自然地做判断：这个任务路径确定吗？确定就写脚本；不确定就配好工具和停止条件，交给 Agent。

## 常见坑

- **把 Agent 当聊天用**：一次只问一个小问题，浪费了它自己跑循环的能力。想让它发挥价值，就给它一个「有终点、路径不明」的完整任务。
- **不设 `max_steps` / 预算**：这是新手最容易漏的，直接后果就是死循环烧钱。任何 Agent 上线前先设护栏。
- **工具描述太糊**：工具是模型的手，描述写不清它就选错工具。写工具描述要像写给一个聪明但没上下文的新同事看。
- **上下文无限堆**：每圈结果都往里塞，迟早爆窗口。长任务要考虑压缩历史或用 sub-agent 分担。

## 小结

- **Agent = LLM + 工具 + 循环 + 记忆**，核心是模型在循环里**自己决定下一步**，不是更聪明的聊天。
- 循环四步：**感知 → 推理 → 行动 → 观察**，转到完成或撞停止条件为止。
- **Workflow 步骤你定、路径固定**；**Agent 步骤模型定、路径不确定**。重复任务用 Workflow，debug / 迁移 / 跨模块改造这类路径不明的活才用 Agent。
- 自治有成本（更慢、更贵、更难复现），所以路径确定时别硬上 Agent。
- 上线前必设**停止条件**（步数 / 预算上限），防死循环。
- 三个核心部件：**工具、记忆、停止条件**。2026 年像 Claude Code sub-agents 已经帮你把循环和记忆分层封装好了，但机制你得懂，才知道什么时候该拦它。
