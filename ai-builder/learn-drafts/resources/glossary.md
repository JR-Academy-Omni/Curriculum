# Vibe Coding 大师课 · 术语表

> 用法：按主题分组，遇到看不懂的词直接 Ctrl+F 搜。每条 = 一句人话解释 + 一个例子/类比。中英对照。

---

## 一、基础

**Vibe Coding（氛围编程）** — 主要用自然语言向 AI 描述意图、由 AI 生成代码，你负责判断和把关，而不是逐行手写。*例：你说"给我做一个能上传 CSV 并画折线图的页面"，AI 直接写出代码，你跑起来看对不对。* 注意：不是"不看代码只看感觉"就交付——你仍要 review、测试、担责。

**Prompt（提示词）** — 你发给模型的那段输入指令/问题，决定它输出什么。*类比：给外包同事的需求文档，写得越清楚、约束越明确，交付越靠谱。*

**Context Engineering（上下文工程）** — 有意识地设计"喂给模型的整包信息"（指令 + 相关代码 + 文档 + 历史 + 工具结果），让模型在有限窗口里拿到最相关的内容。*例：与其把整个 repo 塞进去，不如只放要改的那 3 个文件 + 项目规范。* 比"写好一句 Prompt"更大——它管的是整个信息环境。

---

## 二、Prompt 与项目约定

**Cursor Rules / `.cursorrules`** — 放在项目里的规则文件，告诉 Cursor 编辑器"这个项目一贯怎么写代码"，每次生成都自动带上。*例：写"用 TypeScript、不用 any、组件用函数式"，之后 AI 生成都遵守。* 新版 Cursor 也支持 `.cursor/rules/` 目录形式。

**`CLAUDE.md`** — 放在项目根目录的说明文件，Claude Code 每次进项目自动读取，用来记住项目结构、命令、约定和红线。*类比：给新同事的入职手册，AI 一进门就先看它。* 和 `.cursorrules` 是同类东西，各家工具的项目记忆文件。

**Slash Command（斜杠命令）** — 在 AI 编程工具里用 `/名字` 触发的预设动作/流程。*例：打 `/review` 让它审当前 diff，`/deploy` 触发发布流程。* 本质是把一段常用指令封装成一个可复用的快捷入口。

**Skill（技能）** — 一份打包好的专项能力说明（做某类任务的 SOP + 可带脚本/资源），AI 在合适场景自动加载或你手动调用。*类比：一个"专科顾问"，需要写小红书就叫写小红书那个，需要审课程就叫审课程那个。* 比单个 Slash Command 更完整，通常含判断何时用、怎么做、红线。

## 三、Agent 与编排

**Agent（智能体）** — 能自己拆解目标、调用工具、根据结果决定下一步、循环推进直到完成任务的 AI，而不是问一句答一句。*例：让它"修好这个 bug"，它自己读代码→改→跑测试→看报错→再改，直到测试通过。*

**Agentic Coding（智能体式编程）** — 让 Agent 自主完成一整段编码任务的工作方式（读文件、改多处、跑命令、验证），你从"写代码"退到"给目标 + 验收"。*和 Vibe Coding 的区别：Vibe Coding 强调"用自然语言编程"这件事，Agentic Coding 强调"AI 自己成环地干活"这种自主度。*

**Tool Use / Function Calling（工具调用 / 函数调用）** — 模型不只输出文字，还能按约定格式请求调用外部函数/工具（查数据库、发请求、跑代码），拿到结果再继续。*例：你问"北京现在几点"，模型调用一个 `get_time` 工具拿到真实时间再回答。* 两个词基本同义，Function Calling 是早期叫法。

**ReAct（Reason + Act，推理-行动）** — 一种 Agent 套路：让模型先"想一步"（Reason）再"做一步"（Act），拿到观察结果后再想再做，交替循环。*类比：一边出声念叨思路一边动手，而不是闷头一次做完。*

**Sub-agent（子智能体）** — 主 Agent 派生出来专门干某个子任务的 Agent，干完把结果交回主 Agent。*例：主 Agent 要重构代码，先派一个 sub-agent 专门"把整个 repo 里相关调用点找全"，找完再继续。* 好处是子任务用独立上下文，不污染主线。

**Orchestrator / Worker（编排者 / 工作者）** — 一种多 Agent 分工模式：Orchestrator 负责拆任务、分派、汇总；Worker 负责各自埋头执行一块。*类比：项目经理（Orchestrator）拆活派给几个工程师（Worker），最后汇总交付。*

**Workflow（工作流）** — 预先写死的固定步骤序列（步骤和顺序是人定的），每步可能调用模型。*和 Agent 的区别：Workflow 路径是固定的（照剧本走）；Agent 自己决定下一步（临场发挥）。* 很多真实系统是两者混用。

**Schedule（定时/调度）** — 让某个任务/Agent 按时间自动触发（每天几点、每隔多久、某时刻一次）。*例：每天早上 9 点自动跑一遍"检查部署状态"。*

## 四、MCP（模型上下文协议）

**MCP（Model Context Protocol，模型上下文协议）** — 一个开放标准，规定 AI 应用怎么统一接入外部工具和数据源，让"一个工具写一次、各家 AI 都能用"。*类比：AI 世界的 USB-C 接口——不用为每个模型单独适配。*

**MCP Client / Server（MCP 客户端 / 服务端）** — Client 是发起方（AI 应用/编辑器那边），Server 是提供能力的一方（暴露某个工具/数据源）。*例：Cursor 是 Client，一个"查你数据库"的 MCP Server 提供查询能力，Cursor 连上就能用。*

**Tool vs Resource（工具 vs 资源）** — MCP Server 提供的两类东西：Tool 是"能执行的动作"（会产生副作用，如发邮件、写文件）；Resource 是"可读取的数据"（如一份文件内容、一条记录）。*记忆：Tool = 动词（做事），Resource = 名词（读料）。*

**Transport（传输方式：stdio / SSE）** — Client 和 Server 之间怎么通信。**stdio** 走本地标准输入输出（Server 和应用在同一台机器上、进程间通信）；**SSE**（Server-Sent Events，也常见 Streamable HTTP）走网络、适合远程 Server。*例：本地跑的文件工具用 stdio；托管在云上的服务用 SSE/HTTP。*

## 五、模型与优化

**Token（词元）** — 模型处理文本的最小计费/计量单位，一个 token 约等于英文几个字符或中文一两个字，不完全等于"一个词"。*例：一段 500 字的中文大约几百到上千 token；API 按 token 计费和限长。*

**上下文窗口（Context Window）** — 模型一次能"看进去"的 token 上限（输入 + 输出合计）。*类比：模型的工作台面大小，放不下的内容它就看不到。* 超了要么报错、要么旧内容被挤掉。

**中间遗忘（Lost in the Middle）** — 当上下文很长时，模型对开头和结尾的信息记得清楚，对夹在中间的信息容易忽略。*应对：把最关键的指令/信息放开头或结尾，别埋在长文中段。*

**Temperature（温度）** — 控制输出随机性的参数：低（接近 0）更确定、更保守、更可复现；高更发散、更有创意也更容易跑偏。*例：写代码/抽结构化数据用低温；头脑风暴文案用高一点。*

**Top-P（核采样，Nucleus Sampling）** — 另一个控制随机性的参数：只从累计概率达到 P 的那批最可能候选里采样。*例：Top-P=0.9 表示只在"最靠谱的前 90% 概率质量"里选词。* 一般 Temperature 和 Top-P 调一个就够，别同时大改。

## 六、Prompt 技巧与常见坑

**Zero-shot（零样本）** — 不给示例，直接下指令让模型做。*例：直接说"把这段话翻成英文"。*

**Few-shot（少样本）** — 在 Prompt 里先给几个"输入→输出"示例，再让模型照做。*例：给 3 个"公司名→行业标签"的例子，再让它标第 4 个。* 适合让输出格式/风格对齐。

**Chain-of-Thought（CoT，思维链）** — 引导模型先分步推理再给答案（如"一步步想"），提升复杂推理的正确率。*类比：让它把演算过程写出来，而不是直接报答案。*

**幻觉（Hallucination）** — 模型一本正经地编出看似合理、实则不存在或错误的内容（假引用、假 API、假事实）。*应对：关键事实要核实、让它给可验证来源、代码要真跑一遍。*

**Slopsquatting（幻觉依赖包）** — 一种供应链攻击：模型幻觉出一个不存在的库名，攻击者抢注这个名字发恶意包，你照抄安装就中招。*应对：装依赖前先确认这个包真实存在、来源可信，别盲信 AI 给的 import。* 名字混着"slop（AI 垃圾产出）"和"typosquatting（错拼抢注）"。

## 七、检索、数据与工程安全

**RAG（Retrieval-Augmented Generation，检索增强生成）** — 回答前先从你的知识库检索相关片段，塞进上下文再让模型基于这些内容作答。*例：客服机器人先从公司文档里搜到相关条款，再据此回答，而不是凭记忆。* 主要用来减少幻觉、让模型用上"训练时没见过的私有/最新资料"。

**向量数据库（Vector Database）** — 专门存"文本的语义向量（embedding）"并做相似度检索的数据库，是 RAG 的常用底座。*类比：按"意思接近"来找资料，而不是按关键词精确匹配。* 例：Pinecone、Milvus、pgvector。

**幂等（Idempotent）** — 同一个操作执行一次和执行多次，结果一样、不会重复产生副作用。*例：设计一个"创建订单"接口，重试时不会重复下单。* 在 Agent/自动化里很重要——重试和重放不能把事情做两遍。

**PII（Personally Identifiable Information，个人身份信息）** — 能识别到具体个人的数据：姓名、手机号、邮箱、身份证、地址、简历等。*红线：喂给外部模型、写进日志或 git 前要格外小心，密钥更是任何情况都不进 git。*

**脱敏（Data Masking / Anonymization）** — 在保留数据可用性的前提下，去掉或替换掉敏感信息。*例：把手机号打码成 `138****1234`、把真实姓名换成"学员 A"，再拿去做案例分享或喂模型。*
