## 先讲个我差点翻车的下午

上周三下午，线上有个诡异的 500，我盯着日志看了半天没头绪。顺手把一整段错误日志复制进 Claude Code，让它帮我分析堆栈。粘完的那一秒我心里咯噔一下——那段日志里有几行是完整的请求体，里面躺着一个客户的邮箱、手机号，还有半截 authorization header。

我当时的第一反应不是"糟了违规了"，而是更朴素的一个问题：**这段东西现在到底去哪了？会不会被记下来？会不会哪天变成模型训练数据的一部分？** 我答不上来。

这就是这一章想解决的事。你天天在用 Claude Code、Cursor、Codex CLI，把代码、日志、报错、配置一股脑喂进去，效率是真高。但"喂什么"这件事，大部分人从来没认真想过边界在哪。

**不懂会怎样？** 轻的是你把公司还没公开的业务逻辑、内部架构喂给了一个你不清楚数据流向的服务；重的是客户 PII、密钥、内部机密流出去，等出了事，追责追到你头上，而你连"我当时喂了什么、它存了没有"都说不清。这不是吓唬——这是每个用 AI 写代码的人每天都在做、但很少复盘的动作。

好消息是：这件事的规则其实很朴素，一张清单就能覆盖 90% 的日常判断。我们来把它讲清楚。

## 一、红黄绿：喂之前先过一遍这张表

不用背复杂的分类法，你只要在按下回车前，问自己一句：**"这段东西如果被记下来、被别人看到，我慌不慌？"** 然后对号入座。

### 🟢 绿灯——基本可以放心喂

- 公开的开源代码、公开文档、Stack Overflow 上就能搜到的东西
- 你自己写的、不含任何机密逻辑的通用工具函数、算法练习
- 已经开源或计划开源的项目代码
- 造出来的假数据、示例数据、公开 API 的示例响应

这类东西本来就在公共领域，喂进去没有额外风险。

### 🟡 黄灯——脱敏 / 处理后才能喂

- 生产日志（几乎一定夹着真实数据，见开头那个坑）
- 含真实数据库 schema、表结构、字段命名的东西（字段名本身可能泄露业务）
- 内部但不算最高机密的代码（把里面的真实域名、内部服务地址、员工信息换掉再喂）
- 带真实用户样本的测试数据

黄灯的意思不是"不能喂"，是"喂之前先动手把敏感部分替换掉"。下面第四节专门讲怎么脱敏。

### 🔴 红灯——绝对不喂，任何情况

- **密钥、token、密码、私钥、连接串**——这是铁律，任何 AI、任何 git、任何截图都不进
- **客户 / 用户的个人信息（PII）**：真实姓名、身份证、邮箱、手机、地址、支付信息
- **公司核心机密**：未公开的算法、定价模型、合同条款、财务数据、并购信息
- **受合规约束的敏感数据**：医疗、金融、政府类数据（这类往往有硬性法规管着）
- **未公开的业务逻辑**：那些"我们凭这个吃饭"的核心 know-how

红灯的东西，哪怕只是"让 AI 帮我看一眼"，也不要。

> 一个简单的心法：**绿灯 = 本来就公开的；红灯 = 泄露了拿不回来的；黄灯 = 处理一下就能降到绿灯的。** 拿不准的一律当黄灯处理，脱敏后再说。

## 二、同一个工具，不同版本，数据待遇可能完全不同

这是最多人踩的认知坑：**"我用的是 Claude Code，我同事也用 Claude Code，那我俩的数据处理方式一样"——不一定。**

同一个 AI 工具，个人版和企业版、免费和付费、不同订阅层级，在这三件事上可能有本质区别：

1. **你的输入会不会被用于改进 / 训练模型**——有的版本默认会用，有的默认不会，有的可以在设置里关。
2. **数据存在哪、存多久**——有的按区域存储，有的有明确的保留期和删除承诺。
3. **数据驻留（data residency）在哪个区域**——你的数据落在澳洲、美国还是别处，对企业合规可能是硬指标。

### 术语四行模板：数据驻留（Data Residency）

- **一句话**：你的数据物理上存放在哪个国家 / 区域。
- **为什么你要在意**：有些企业、有些行业要求数据必须留在澳洲境内，跨境存储可能直接违反公司 policy。
- **常见误区**：以为"云服务嘛数据在哪都行"——对合规敏感的公司，区域是能一票否决的硬条件。
- **怎么查**：看你所用工具那个具体版本的官方条款里"data residency / storage location"章节，别猜。

**关键提醒：我不替你下结论。** 我不会告诉你"Cursor 的免费版会不会拿你代码训练"或者"Codex CLI 数据存几天"——因为这些条款会变，各家写法不同，各订阅层不同，我编一个具体数字给你反而害了你。

**你该做的是**：打开你正在用的那个工具、那个具体版本的官方隐私条款 / 数据处理说明，重点看三段——(1) 会不会用于训练；(2) 存储区域；(3) 保留期。花十分钟看一次，比背任何清单都管用。企业环境里，这一步往往你们 IT / 法务已经审过了，直接去问他们要结论。

## 三、澳洲 / 企业环境：多一层意识，但别自己当律师

如果你在澳洲、或者在给企业客户做项目，客户数据这件事要额外上心。

定性地说，你需要有这么几条基本意识：

- **客户数据不是你的**——你只是替客户处理它，把它喂给第三方 AI 服务，本质上是把客户数据交给了另一个你可能没授权的地方。
- **PII 要格外小心**——澳洲对个人信息的处理是有明确监管的，企业客户往往在合同里就写了数据怎么用、能不能出境。
- **合同 / NDA 可能直接禁止**——很多企业项目的合同里有条款限制数据流向第三方工具，你把客户代码喂 AI 之前，这条得先确认。

### 但是——这里划一条硬边界

**我不做法律建议，这一章也不给你任何具体法规条款、罚款数字或公司案例。** 原因很简单：合规是真金白银的事，条款会更新、行业有差异、你的具体情形我不了解，我编一个"根据某某法第几条罚多少"给你，你照着做出了事，责任担不起。

所以涉及合规，标准答案永远是这一句：

> **"具体能不能这么用，问你自己公司的 policy 和法务 / 合规团队。"**

这不是推卸，这是对的做法。在企业里，"我用 AI 之前先确认了公司 AI 使用 policy" 本身就是专业度的体现。很多公司已经有明确的《AI 工具使用规范》，你第一件事应该是去找它、读它。

## 四、脱敏：把黄灯降成绿灯的手艺

大部分"想喂但不敢喂"的场景，脱敏都能解决。核心思路一句话：**把真实的敏感信息，换成结构一样、内容假的占位，再喂给 AI。**

### 术语四行模板：脱敏（Data Masking / Sanitization）

- **一句话**：把数据里的敏感部分替换成假的、无法反推真实值的内容，但保留它的结构和格式。
- **为什么你要在意**：AI 需要的是"结构和逻辑"来帮你 debug，它并不需要真实的客户邮箱——所以真实值完全可以换掉。
- **常见误区**：以为"改几个字符"就行——`zhang@abc.com` 改成 `zhang@xyz.com` 还是能认出人，脱敏要换成完全无意义的 `user@example.com`。
- **怎么做**：见下面这张对照。

| 真实数据 | 脱敏后 |
|---|---|
| `sk-live-9f3a...`（密钥） | `sk-xxxxxxxx`（占位，甚至整段删掉） |
| `zhang.wei@client.com.au` | `user@example.com` |
| `+61 4xx xxx xxx`（真实手机） | `+61 400 000 000` |
| 真实客户名 `Acme 矿业` | `ClientCo` |
| 内部服务地址 `internal-api.corp.local` | `api.example.internal` |
| 生产数据库真实记录 | 手工造 2-3 条假记录，格式一样 |

**实操习惯**：日志类的东西，先在本地文本编辑器里用查找替换把敏感字段清一遍，再复制进 AI。别嫌麻烦——开头那个下午如果我先做这一步，就不会有那声"咯噔"。

有一个反直觉但重要的点：**脱敏之后，AI 帮你解决问题的能力几乎不打折。** 它需要的是"有个字段格式不对""这里空指针了"这种结构信息，真实的客户是谁它根本不关心。所以脱敏不是"为了合规牺牲效率"，是"几乎零代价拿到安全"。

## 五、什么时候该考虑私有 / 本地部署

如果你的项目敏感到"连脱敏后都不放心往外发"，那就该考虑把 AI 拉到你自己可控的环境里。定性地讲，有这么几档选择：

### 术语四行模板：私有部署（Private / Self-hosted Deployment）

- **一句话**：把模型跑在你自己的服务器 / 内网 / 本地机器上，数据不出你的边界。
- **为什么你要在意**：对最敏感的项目，"数据不离开公司网络"可能是唯一能过合规的方案。
- **常见误区**：以为"本地跑的模型一定更笨"——现在能本地跑的开源模型对很多编码任务已经够用，敏感项目值得这个取舍。
- **怎么选**：跟你们 IT / 平台团队谈，这通常不是个人能拍板的事。

几种典型形态（由轻到重）：

- **企业网关 / 代理**：公司统一接一个企业版入口，走审计、走脱敏管道，员工不直接接触外部服务。企业里最常见。
- **自托管开源模型**：在公司自己的机器上跑开源模型，数据完全不出内网。
- **敏感项目断网用法**：极端场景下，某些项目直接规定"在隔离环境 / 断网机器上用本地模型"，一点数据都不外流。

个人开发者一般用不到最重的那档，但你至少要知道"有这条路"——当遇到一个"这数据真的一点都不能出去"的项目时，答案不是"那就别用 AI 了"，而是"换成本地 / 私有部署"。具体怎么落地，交给你们平台 / IT 团队，别自己硬上。

## 常见坑

- **"内部代码应该没事吧"**——内部 ≠ 可以外发。内部代码里常藏着真实域名、内部服务名、账号，甚至注释里的业务机密。喂之前当黄灯处理。
- **"我删掉密钥再发"结果删漏了**——密钥经常不止一处（配置里一个、日志里又一个、注释里还有一个）。养成"密钥类整段清空"的习惯，别逐个找。
- **把工具的版本混为一谈**——个人版的数据处理方式，不能想当然套到企业版上，反之亦然。用哪个版本就查哪个版本的条款。
- **拿 AI 当法务**——包括我在内的任何 AI，都不该是你的合规依据。合规问题回到公司 policy 和法务。
- **脱敏改一半**——把 `client.com` 改成 `client2.com` 不叫脱敏，还是能认出来。要换成彻底无意义的占位。
- **截图也是外发**——不只是复制粘贴文本，把带敏感信息的截图丢给多模态 AI 一样是把数据送出去了。

## 小结

- 按下回车前先问一句："这东西泄露了我慌不慌？" 慌 → 红灯或黄灯。
- **红灯铁律**：密钥、PII、核心机密，任何情况都不喂、不进 git、不截图。
- **黄灯常态**：日志、内部代码、真实测试数据——脱敏后再喂，成本极低、效果几乎不打折。
- 同一工具不同版本的数据待遇可能完全不同，用哪个就去读哪个的官方条款（训练 / 存储区域 / 保留期这三段）。
- 合规问题一律回到**公司 policy 和法务**，不靠 AI、不靠记忆里的法条。
- 敏感到脱敏都不够 → 考虑企业网关 / 私有 / 本地部署，交给 IT 团队落地。

**今天就动手的一件事**：给你自己或团队写一张《AI 使用数据红线》清单，三档就够：

```
绝不进 AI：密钥/token/密码、客户PII、公司核心机密、合规敏感数据
脱敏后可进：生产日志、内部代码、真实schema、真实测试数据（换假值再喂）
随便用：公开代码、开源库、假数据、已开源项目
```

把它贴到团队 wiki，新人第一天就发给他。有了这张表，"能不能喂 AI" 这个问题，团队里每个人都能在三秒内自己答出来——这比任何事后的追责都值钱。
