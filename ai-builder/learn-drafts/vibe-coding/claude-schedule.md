我第一次给 Claude Code 配定时任务，是想让它每天早上帮我查一下线上课程大纲和本地 outline 有没有对不上。结果第一版我写得太随意，只说了一句「帮我同步一下」。它真的去 `git push` 了——把一份还没审的改动推到了 main。那天早上我一边喝咖啡一边看到通知，手一抖差点把杯子扣键盘上。

从那以后我才明白：定时任务不是「设个闹钟让 AI 聊天」，是**把一段能自己判断、能动手的流程,交给一个没人盯着的 agent 在固定时间跑**。它能干多少活,取决于你给它划的边界。这一章就讲怎么把这份边界说明书写清楚。

## Claude Schedule 到底是什么

先把概念摆正,不然后面全是误会。

**Claude Schedule（定时 Agent）：**
- 一句话解释：让一个带完整上下文的 AI agent 在你指定的时间点,自动跑一段**可验证**的工作流,而不是等你手动敲命令。
- 形象比喻：像给你雇了个夜班同事——你下班前把「几点做什么、能碰什么、做完怎么交」写在便利贴上,他半夜按点上工,早上你来看他留的记录。
- 工作中怎么用：内容巡检、drift 检查、依赖安全更新、日报生成、失败复盘,这类「每天/每周都要做一遍、步骤固定、结果能对错」的活。
- 最常踩的坑：以为它是「定时聊天机器人」。它不是来陪你唠嗑的,它是来**执行 + 交作业**的。你不给它明确的验收标准,它就会自己发挥,发挥就容易翻车。

它和你平时在 Claude Code / Cursor 里边聊边改最大的区别是:**没人在旁边确认**。你手动改代码时,每一步都能喊停;定时任务跑起来的时候你可能在睡觉。所以这份说明书必须替「当时不在场的你」把关。

## 什么活适合交给定时任务

判断标准就一条:**这件事出错了,能不能自动发现、能不能轻松回退。** 能的就适合,不能的就别。

适合的:

- **内容巡检**——每天扫一遍已发布的文章/页面,查死链、查过期日期、查 SEO 标题有没有被谁改乱。纯只读,最多出个报告。
- **drift 检查**——比对 production 的课程数据和本地 `outline.json`,哪节课线上线下不一致列出来。也是只读 + 报告。
- **依赖安全更新**——扫 `package.json` 里有没有安全公告的包,升版本、跑测试,**但只出 PR,不自动 merge、不自动部署**。人第二天来点一下 merge。
- **报告生成**——把昨天的 git 提交、数据、进展汇总成一份日报,写进文件或发通知。
- **失败任务复盘**——上一个定时任务挂了,让它读 log、总结失败原因、给建议。

不适合的(这些必须人工确认):

- 删数据库记录、删文件——不可逆。
- 任何涉及**支付、退款、扣费**的操作。
- 直接 **部署到生产**——哪怕测试过了,部署时机也该是人决定的。
- 给用户群发消息/邮件——发出去收不回。

**判断口诀:凡是「做错了要打电话找人求救」的操作,不进定时任务。**

## 怎么写一份可上线的任务说明书

一份能上线的说明书,我固定写五块。缺一块,这任务我就不敢挂上去。

### 1. 任务目标(一句话说清「干什么、算成功」)

别写「检查一下课程」。要写「比对 production 的 `ai-builder` 课程 lesson 列表和本地 `curriculum/ai-builder/public/outline.json`,把标题/顺序/数量不一致的 lesson 列成一张表」。目标里必须带**验收标准**——你怎么知道它成功了。

### 2. 执行边界(最关键的一块)

这是我那次翻车之后加的。把权限**分成四档,明确写死这次能到哪一档:**

**执行边界(四档权限):**
- 一句话解释:把 agent 能做的事,从「只能看」到「能改能推能上线」分成递进的四级,这次任务只授权到必要的那一档。
- 形象比喻:像给保洁的门禁卡——发个只能进大堂的,还是能进机房的,你得想清楚。给多了出事,给少了干不成。
- 工作中怎么用:在说明书里明写一行,比如「本任务权限:**可提交**(能 git commit + 开 PR),**禁止部署**」。
- 最常踩的坑:图省事直接给最高权限。默认就该给最低的那档,不够用了再往上加。

四档是这么分的:

| 档位 | 能做什么 | 典型任务 |
|------|---------|---------|
| 只读 | 读文件、跑查询、生成报告 | 内容巡检、drift 检查 |
| 可写 | 改本地文件,不碰 git | 批量改文案、格式化 |
| 可提交 | git commit + 开 PR,**不合并** | 依赖更新、自动化改稿 |
| 可部署 | 直接推 main / 上线 | ⚠️ 基本不给定时任务 |

写在说明书里就是明确一行:

```
权限档位: 可提交
- 允许: 读文件、改本地文件、git commit、git push 到新分支、gh pr create
- 禁止: git push origin main、merge PR、任何部署命令(deploy/scp/ci trigger)
```

### 3. 输入来源

它从哪拿数据?写清楚具体路径/接口。比如「本地 `outline.json` + production admin API 的 `/lessons?course=ai-builder`」。别让它自己去猜该读哪个文件。

### 4. 验证方式

跑完怎么证明结果是对的,不是瞎编的。常用四种:

- **diff**——改动前后做对比,只报真正变了的地方。
- **测试**——依赖升级后跑一遍 `npm test`,绿了才算数。
- **计数**——「本地 42 节课,线上 40 节,差 2 节」,数字对得上。
- **schema 校验**——生成的 JSON 用 schema 验一遍格式,不合规就报错别提交。

### 5. 输出方式

结果去哪。报告写进哪个文件、PR 开到哪、通知发到哪个渠道。**一次任务只对应一种主输出**,别让它又发通知又直接改线上。

## 「默认保守」原则

这四个字是我这套东西的地基:**先报告,先 diff,先让人确认。**

意思是,任何有风险的定时任务,第一版都别让它真动手。先让它**只生成报告**——「我打算改这 5 个地方,你看对不对」。跑几天你确认它每次判断都靠谱了,再把权限往上放一档,让它开 PR。再稳一阵,才考虑是不是让它 commit。

给依赖更新任务的说明书,保守版长这样:

```
1. 扫 package.json 找有安全公告的包
2. 逐个升级到最近的 patch 版本
3. 跑 npm test —— 挂了就停,不提交,写进失败报告
4. 全绿 → git checkout -b deps/auto-update-{date}
5. git commit + gh pr create(标题带 [自动] 前缀)
6. ⛔ 到此为止。不 merge、不部署。等人 review。
```

你看第 3 步和第 6 步——一个是「不确定就停」,一个是「到边界就收手」。这两道闸门是定时任务和手动操作最大的差别:它得替不在场的你踩刹车。

## 失败报告怎么写

定时任务一定会失败——网断了、接口变了、测试挂了。关键是它失败之后留下的信息,能不能让你三分钟看懂、知道下一步干嘛。

一份合格的失败报告固定三块:

```
❌ 任务失败:依赖安全更新 2026-07-02 06:00

失败命令:
  npm test

错误原因:
  lodash@4.17.22 升级后,src/utils/format.test.ts 3 个用例挂了
  TypeError: _.pick is not a function

下一步建议:
  1. lodash 4.17.22 移除了 _.pick,改用 _.pickBy 或原生解构
  2. 已回滚到 4.17.21,未提交任何改动
  3. 需人工确认要不要改调用方代码后再升级
```

**失败命令**告诉你它卡在哪一步,**错误原因**给原始报错别润色,**下一步建议**给人一个可执行的起点。最后那句「未提交任何改动」尤其重要——它在告诉你现场是干净的,你不用先去收拾残局。

## 为什么要懂这个 / 不懂会怎样

懂了,你能把每天重复的巡检、比对、更新交出去,早上打开电脑看一份现成报告,像上面雇了个夜班同事。

不懂,两种结局:要么你根本不敢用,重复劳动继续手动干;要么你像我那样图省事给了满权限,某天半夜它「自作主张」把没审的东西推上线,甚至删了不该删的。前者浪费时间,后者是真事故。

## 常见坑

- **给了满权限图省事**——最常见也最致命。默认最低档,不够再加。
- **没有验证**——它说「同步完成了」,你也不知道到底对不对。没有 diff/测试/计数,报告等于零。
- **失败悄无声息**——任务挂了没通知,你三天后才发现巡检早就没跑了。失败也要有输出。
- **一个任务干太多事**——又检查又修改又部署又通知。拆成几个小任务,每个只干一件、只有一种输出。
- **把它当聊天**——写成「帮我看看有没有问题」这种开放式提问。定时任务的指令必须是**闭合的、有验收标准的**。

## 小结

Claude Schedule 是让 agent 在固定时间跑一段可验证工作流,不是定时聊天。写一份能上线的说明书,五块缺一不可:**任务目标(带验收)、执行边界(只读/可写/可提交/可部署分四档)、输入来源、验证方式(diff/测试/计数/schema)、输出方式**。适合的活是能自动发现错、能轻松回退的——巡检、drift 检查、只出 PR 的依赖更新、报告、复盘;不适合的是删除/支付/部署这类不可逆操作,一律人工确认。贯穿始终一个原则:**默认保守——先报告、先 diff、先让人确认**,权限一档一档往上放。最后别忘了配好失败报告(失败命令 + 错误原因 + 下一步建议),让不在场的你也能三分钟接手。
