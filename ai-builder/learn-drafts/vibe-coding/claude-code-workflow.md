# Claude Workflow 模板与执行清单

我带过不少刚上手 Claude Code 的同学，最常见的翻车姿势是：打开终端，敲一句「帮我把用户表加个 phone 字段」，然后回车，等着看结果。运气好改对了；运气不好，它改了 model、忘了改 migration，或者顺手把你没让它动的一个测试文件也重写了。你盯着一堆 diff，不知道从哪看起。

问题不在于模型笨，在于你没给它一个**可复现的 workflow**。同样一句话，今天让它做和明天让它做，结果可能完全不同——因为每次的上下文、你补充的信息、它自己临时的判断都在变。真正能落地的做法，是把「让 AI 干活」这件事变成一条固定流程：输入什么、中间怎么走、输出长什么样，全都有章可循。这一章就讲怎么搭这条流程。

## 为什么要懂 workflow，不懂会怎样

一句话 Prompt 能用，但它是**赌博式**的。任务小、你自己心里有数、错了也能一眼看出来——OK，赌一把没问题。可一旦任务碰到多个文件、有隐含约束（数据库要同步改、接口有调用方、代码风格有规矩），一句话就压不住了。你会陷进「它改一版 → 我发现漏了 → 再补一句 → 它又改坏别的」的死循环，比自己写还慢。

懂 workflow 的人不一样：他知道每个任务开始前先把「目标、范围、不许动什么、怎么算成功」写清楚，结束时让 AI 交出「改了哪些、测试过没、还有什么风险」的账。这样即使 AI 中途犯浑，你也能在检查点拦住它，而不是等到全跑完才发现满盘皆输。

## Workflow 的三层：从随手一句到自动 Agent

先建立一个分层的直觉。让 AI 干活，从轻到重有三种形态，别用大炮打蚊子，也别拿弹弓打坦克。

- **一次性 Prompt**：一句话，一个来回。适合改个错别字、写个正则、解释一段代码。不值得搞流程。
- **静态 Workflow**：你预先定好固定的几步（读文件 → 改代码 → 跑测试 → 报告），每次任务都走同样的轨道。适合**重复出现、步骤稳定**的活，比如「加字段」「加一个 API 端点」——这类任务八成时间是同一套动作。
- **动态 Agent Workflow**：你只给目标和工具，让 Claude Code 自己决定下一步做什么、调哪个工具、要不要回头重试。适合**探索性、边界不清**的活，比如「排查这个偶发的线上报错」——你自己都不知道要读哪几个文件，只能让它自己找。

这一章重点讲**静态 Workflow**，因为日常开发九成任务都属于这一层，把它固化下来收益最大。

**术语：静态 Workflow**
- 一句话解释：预先定死步骤顺序、每次任务都照同一套轨道走的执行流程。
- 形象比喻：像麦当劳后厨的操作卡——不管谁上手，做汉堡的动作顺序都一样，出品才稳定。
- 工作中怎么用：把「加字段/修 bug/加接口」这类高频任务写成固定四段模板，每次填空即可。
- 最常踩的坑：把探索性任务（不知道要动哪些文件）硬塞进静态模板，结果 AI 被框死、找不到真正的问题源头。

## 一个可复现 workflow 的四个固定件

不管什么任务，一个能复现的 workflow 都由四段组成。记住这四个词：**入口、上下文、执行、出口**。

- **入口**：目标是什么、范围到哪、**禁止动什么**、怎么算成功。禁止事项是最容易漏的一件——不写清楚，AI 就容易「顺手」改到你不想它碰的地方。
- **上下文**：相关文件路径、要遵守的约束（命名、风格、依赖）、历史上为什么这么设计的决策。给足上下文，AI 才不会瞎猜。
- **执行**：先出计划、再调工具读代码、然后改、最后跑验证命令。计划先行是关键，让它说清楚要动哪些文件、你先审一眼。
- **出口**：diff 摘要、测试结果、剩余风险、下一步建议。这是它给你交的账，让你不用逐行读也能判断做得对不对。

**术语：禁止事项（Constraints / Guardrails）**
- 一句话解释：明确告诉 AI「这些地方不许碰」的边界清单。
- 形象比喻：像给装修工划的红线——这面墙是承重墙，怎么都不能砸。
- 工作中怎么用：写在入口段，例如「只改 user 模块，不动 auth、不动测试快照、不加新依赖」。
- 最常踩的坑：只说要做什么、不说不能做什么，AI 就会自作主张扩大改动范围，diff 一大片没法审。

## 完整走一遍：给用户表新增一个字段

假设需求是：给用户加一个 `phone`（手机号）字段，前后端都要能存能取。看我怎么用四段式把它变成一个 workflow。

**第一步，写入口。** 我不会直接说「加个 phone 字段」，我会给 Claude Code 这样一段：

```
目标：给 User 增加 phone 字段（string，可空），后端能存取、API 能返回。
范围：只改 user 相关的 model、migration、schema、序列化。
禁止：不动 auth 逻辑；不改任何测试快照；不加新依赖；不动前端样式。
成功标准：迁移能跑通；新增字段在创建/查询用户的接口里能读写；已有测试全绿。
```

**第二步，喂上下文。** 让它先摸清现状，别急着写：

```bash
# 让 Claude Code 先读相关文件，别急着改
# 你可以直接说：先读 user model、对应 migration 目录、API schema，
# 告诉我现在字段是怎么定义的、迁移是怎么组织的，先别改代码。
```

它读完会回你「model 在 src/models/user.ts，迁移在 migrations/ 用时间戳命名，接口 schema 在 src/schemas/user.ts」之类。这一步是让它把上下文装进脑子，也让你确认它找对了地方。

**第三步，执行——先计划后动手。** 明确要求它先出计划：

```
先给我一个改动计划：列出要改哪几个文件、每个文件改什么，先别写代码。
```

它会列：`1. user model 加 phone 字段 2. 新建一个 migration 3. user schema 加 phone 4. 序列化时带上 phone`。你扫一眼，觉得没漏没多，说「按这个计划改」。它才开始动代码。

**第四步，验证。** 改完立刻跑命令验证，别信「我改好了」这句话：

```bash
# 迁移能不能跑通
npm run migrate

# 相关测试有没有被改绿/改红
npm test -- user

# 类型检查
npm run typecheck
```

**第五步，收出口。** 让它交账：

```
给我一个总结：改了哪些文件（diff 摘要）、上面命令的结果、还有什么风险、建议的下一步。
```

它回你「改了 4 个文件、migrate 通过、user 测试 12 个全绿、typecheck 无报错；风险：老数据的 phone 是 NULL，如果前端假设它一定有值会出问题；下一步：前端表单加上 phone 输入框」。到这，你对这次改动的全貌一清二楚——这就是可复现的意义。

修 bug 也是同一套模板，只是入口段变成「复现步骤 + 期望行为 + 实际行为」，执行段多一步「先写一个能复现这个 bug 的失败测试」，验证段看那个测试从红变绿。骨架完全一样。

## 检查点与失败回退

四段之间有三个**检查点**，每个都对应一种失败和回退方式。

- **计划检查点**（执行前）：计划里出现了你禁止的文件、或漏了关键改动 → 别让它动手，直接把计划打回：「你不能碰 auth，去掉那条；migration 你漏了，补上」。这是最便宜的拦截点，改一句话的事。
- **验证检查点**（改完后）：测试红了、迁移报错 → 把**完整报错**贴回去让它修，别自己脑补。修两三轮还在原地打转，说明它跑偏了 → `git checkout .` 回到干净状态，重开一个来回，把出错的线索写进入口段（「上次它误改了 X，这次禁止」）。
- **出口检查点**（交账后）：它列的剩余风险你不接受 → 当成一个新的小 workflow 继续，别在同一个乱糟糟的对话里硬缝。

一条铁律：**动手前先 commit**。有 git 兜底，任何一步搞砸了都是 `git checkout .` 一键回到原点，代价为零。没有这层保险，你才会舍不得推翻重来、结果越缝越烂。

## 把 workflow 固化成可复用资产

上面这套四段式，你不该每次手打。把它沉淀成资产，自己省事，全组也能用。有三个层次：

- **CLAUDE.md**：放项目根目录，Claude Code 每次自动读。把「本项目的通用约束」写进去——技术栈、命名规矩、不许动的目录、跑测试的命令。相当于给所有任务预置了上下文和禁止事项，你就不用每次重复。
- **Slash command**：把某类任务的完整四段模板存成一条自定义命令。比如写一个 `/add-field`，正文就是上面那套「目标/范围/禁止/成功标准 + 先计划后改 + 跑这几个验证命令」的模板，以后加字段直接 `/add-field phone`。Cursor 和 Codex CLI 也有各自的自定义命令 / 规则文件机制，思路一致：把重复的流程存下来复用。
- **Skill**：比 slash command 更完整的一包能力（带说明、步骤、示例文件），适合封装那种「有固定 SOP、还带参考素材」的复杂流程。团队里把它 commit 进仓库，所有人 pull 下来就能用同一套标准。

**团队怎么共享**：这三样都是**纯文本、进 git**。你把 CLAUDE.md、`.claude/commands/` 下的 slash command、skill 目录一起提交，同事拉下来就自动获得同一套 workflow。等于把「组里最会用 AI 那个人的干活习惯」变成了全组的默认配置——这才是 workflow 真正的复利。

## 小结

- 让 AI 干活别赌一句话，按任务复杂度选层：随手改 → 一次性 Prompt；重复稳定的活 → 静态 Workflow；探索性的活 → 动态 Agent。
- 任何可复现 workflow 都是四段：**入口**（目标/范围/禁止/成功标准）、**上下文**（文件/约束/历史决策）、**执行**（先计划后改、跑验证命令）、**出口**（diff 摘要/测试结果/剩余风险/下一步）。
- 三个检查点拦住三种失败：计划检查点打回错误计划、验证检查点贴报错重修或回退、出口检查点另起新流程。动手前先 commit，回退零成本。
- 把这套模板沉淀成 CLAUDE.md / slash command / skill，进 git 全组共享，让团队用同一套标准跟 AI 协作。
