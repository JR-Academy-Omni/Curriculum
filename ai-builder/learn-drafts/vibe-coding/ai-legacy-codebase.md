## 先讲个我踩过的坑

我第一次在公司老项目上用 Claude Code，信心满满。项目是个跑了三年的后端，二十几万行。我让它"给订单接口加一个折扣字段"，听起来是个五分钟的活。

它确实动手了——很快。然后我一看 diff：它顺手把一个共用的日期格式化函数从 `dayjs` 换成了 `date-fns`，理由是"date-fns 更现代"。这个函数被四十多个地方引用，其中有几个隐晦的时区处理逻辑。测试当时没覆盖到那几处，CI 绿了，我也 merge 了。结果第二天财务那边报账单时间全错了一小时。

问题不在 AI 笨。问题在于**我把它当成在绿地上写新代码的助手，而它面对的其实是一片雷区**。这一章就是讲怎么避免这种事。

## 绿地和遗留，根本不是一回事

你在 YouTube、在大部分教程里看到的 AI 编程，几乎全是**绿地（greenfield）**——空文件夹，从零开始，"帮我做个 Todo App"。AI 在这种场景下表现惊艳，因为没有任何历史包袱，它想怎么写就怎么写。

但你上班第一天打开的，几乎一定是**遗留（legacy）代码库**——已经有人写了几个月甚至几年、正在生产上跑、你只懂其中一小块。两者的区别是本质性的：

| | 绿地项目 | 遗留项目 |
|---|---|---|
| 上下文 | 几乎没有，你说了算 | 巨大，且大半没写在文档里 |
| 约定 | 你现定 | 已经定死，且经常是隐性的 |
| 改动影响 | 影响你自己 | 牵一发动全身 |
| 能不能大改 | 随便重构 | 不能，改坏了是线上事故 |

**这就是学校/教程和工作之间最大的落差。** 教程教你怎么让 AI 从零造东西，工作却要你让 AI 在一个它（和你）都没完全读懂的系统里做精准的小手术。同样一句"帮我加个功能"，在这两种场景下是完全不同难度的任务。

## 关键一步：给 AI 喂对上下文

AI 不是不聪明，是**它默认对你的项目一无所知**。你不告诉它这个项目的架构和雷区，它只能靠猜——猜就会引入新花样。

### 术语：上下文窗口 / 中间遗忘

- **是什么**：模型一次能"看进眼里"的文字量就是上下文窗口。塞太多，中间那段它会记不清，这叫"lost in the middle / 中间遗忘"。
- **为什么你要懂**：很多人第一反应是把整个 repo 塞给 AI，觉得"给得越全越好"——恰恰相反，塞满了它反而抓不住重点。
- **不懂会怎样**：AI 前面读的约定，改到后面自己就忘了，风格前后不一。
- **一句话**：给 AI 的不是越多越好，是**越相关越好**。

所以喂上下文的正确姿势不是"全量灌"，是**精准 + 分层**：

**1. 用 `CLAUDE.md` 沉淀项目知识。** 这是最重要的一招。在项目根目录放一个 `CLAUDE.md`，把 AI 每次都该知道的东西写进去，它会自动读。别写废话，写它猜不到的：

```markdown
# 项目约定（AI 必读）

## 架构
- 后端 NestJS，DDD 分层：controller → service → repository，禁止跨层调用
- 数据库改动一律走 migration，禁止直接改 schema

## 雷区（改这些之前先问我）
- src/utils/datetime.ts：全站时区逻辑，被 40+ 处引用，不要动实现
- src/legacy/billing/：老计费模块，无测试覆盖，只读不改

## 风格约定
- 日期统一用 dayjs，不要引入 date-fns / moment
- 错误处理走 AppException，不要裸 throw new Error
```

### 术语：CLAUDE.md 项目记忆

- **是什么**：放在项目里的一份 markdown，AI 每次进这个目录都会先读，相当于给它的"入职手册"。
- **为什么你要懂**：它是你把"隐性约定"变成"显性规则"的唯一低成本方式，一次写，之后每次省心。
- **不懂会怎样**：你每次对话都要重复交代"我们用 dayjs 别用别的"，累且容易漏。
- **一句话**：项目的隐性知识，一次沉淀进 `CLAUDE.md`，AI 天天记得。

**2. 用 `@` 引用关键文件，而不是全量。** 需要改哪块，就 `@` 点名相关的几个文件，让 AI 聚焦：

```
帮我在订单里加折扣字段。相关文件：
@src/order/order.service.ts
@src/order/dto/create-order.dto.ts
先读这两个文件，别的地方不要碰。
```

**3. 让它先读懂再动手。** 大项目里，动手前的"复述"这一步能救命：

```
先别写代码。用你自己的话说一遍：
折扣字段应该加在哪一层、会影响哪些调用方、
现有的价格计算逻辑在哪个函数里。我确认后你再改。
```

它复述对了，你才放心；复述错了，说明上下文没喂够，这时候补比改完再返工便宜得多。

**4. 分模块喂，善用 sub-agent（2026）。** Claude Code 现在能开 sub-agent（子代理）帮你分担——比如让一个 sub-agent 专门去"读懂计费模块的调用关系"，把总结带回来，主对话不用被这些细节撑爆上下文窗口。大项目里这比一股脑全塞进主对话干净得多。

## 小步安全改造：把 AI 关进笼子里

上下文喂对了，接下来是控制它的**动作范围**。核心就一句：别让它一次改太多。

**动手前先 commit。** 这是最便宜的保险。让 AI 改之前先存档，改坏了一个 `git checkout` 就回来了：

```bash
git add -A && git commit -m "chore: checkpoint before AI change"
```

**明确划定禁区。** 在指令里把"能碰什么、不能碰什么"说死：

```
只改 order.service.ts 里的 createOrder 方法。
不要改函数签名，不要动别的文件，不要加新依赖，
不要重构任何现有代码——哪怕你觉得它写得不好。
```

那句"哪怕你觉得它写得不好"很关键——AI 特别爱"顺手优化"，这正是我开头那个坑的来源。

**每一步都跑测试。** 别攒一大堆改动再验证。改一小块 → 跑测试 → 绿了再下一步：

```bash
npm test -- order.service.spec.ts
```

改动小，出问题时定位范围就小，回滚也不心疼。

## 常见坑

- **把整个 repo 塞给 AI**：以为给得越全越好，结果触发中间遗忘，它反而抓不住重点。给相关的几个文件就够。
- **不写 `CLAUDE.md`，靠每次对话临时交代**：隐性约定漏一条，AI 就引入一个新依赖。规则该沉淀就沉淀。
- **让它"顺便重构一下"**：在遗留项目里，"顺便"两个字是事故之源。加功能就只加功能，重构单独开一次、单独 review。
- **CI 绿了就 merge**：遗留项目测试覆盖往往有洞（我那次就是），AI 改到没测试覆盖的隐晦逻辑，CI 照样绿。改到雷区文件时人要亲自看 diff。
- **让 AI 自作主张换技术栈**：它会推荐"更现代"的库。老项目要的是一致，不是新潮。`CLAUDE.md` 里把技术选型写死。

## 小结

绿地和遗留是两种活。教程只教前者，工作里全是后者，这个落差你得自己补上。补的方法就三件事：

1. **喂对上下文**——`CLAUDE.md` 沉淀项目知识，`@` 引用相关文件，分模块喂、必要时上 sub-agent，别全量灌导致中间遗忘。
2. **让它先读懂再动手**——动手前先让 AI 复述理解，错了当场补，不等改完返工。
3. **小步安全推进**——先 commit，划死禁区（不重构、不换库、不动签名），改一小块就跑一次测试。

一句话记住：在别人的老项目里，AI 是个能力很强但完全不了解情况的新人。你的活不是放手让它跑，是**给它足够的背景、划清边界、盯着每一步**。做到这三点，它就是最好的帮手；做不到，它就是那个把 dayjs 换成 date-fns 的实习生。

## 实操：给一个现有仓库配 AI，安全加一个小功能

找一个你手上真实的、已有一定规模的项目（不是新建的 demo），跟着走一遍：

**Step 1 · 写 `CLAUDE.md`。** 在项目根目录建 `CLAUDE.md`，至少写清三段：架构分层、雷区文件（点名那些"被很多地方引用、不要乱动"的）、风格约定（用哪些库、不用哪些库）。

**Step 2 · 起一个 checkpoint。**

```bash
git add -A && git commit -m "chore: checkpoint before AI feature"
```

**Step 3 · 让 AI 先读懂。** 挑一个真实的小需求（加个字段、加个校验），`@` 引用相关的 1-3 个文件，让它**先复述**改哪层、影响谁，你确认。

**Step 4 · 划禁区、动手。** 指令里写死：只改哪个文件哪个方法、不重构、不换库、不动签名。让它改。

**Step 5 · 跑测试 + 亲自看 diff。**

```bash
npm test
git diff
```

重点扫一眼：它有没有碰你没让它碰的文件？有没有偷偷加依赖？确认干净了再 commit。

走完这一遍，你就有了在任何遗留项目里安全用 AI 的完整套路——这套路比"会写 prompt"值钱得多，因为它决定了你敢不敢在真实生产项目上放手用 AI。
