## 安装 Cursor 与基础配置

先说句掏心窝的：这一章看起来只是"装个软件"，很多人想跳过。但我带过的学员里，后面卡得最惨的那批，十有八九都是环境没配好——模型登不上、快捷键不会按、项目索引没建起来，于是每写一行代码都要跟工具较劲，AI 明明能帮你，你却在跟安装界面搏斗。第一步这 20 分钟花扎实了，后面几十节课都顺；这里偷懒，后面天天返工。所以别嫌它基础，跟我一步步来。

Cursor 是最流行的 AI 编程 IDE 之一，基于 VS Code，集成 Claude / GPT 模型。装好环境后面学得更顺。你可以把它理解成"一个会写代码的 VS Code"——外壳、快捷键、插件生态几乎和 VS Code 一模一样，但它把 AI 缝进了编辑器的每一个角落，你选中一段代码就能让它改，不用再复制到网页版聊天框里来回倒腾。

### 下载与安装

1. 打开官网 https://cursor.com
2. 下载对应系统版本（macOS / Windows / Linux）
3. 运行安装；首次启动选 "Import VS Code Settings" 继承已有配置

这里补一句：如果你之前用过 VS Code，"Import VS Code Settings" 一定要点。它会把你原来的主题、插件、快捷键一整套搬过来，你打开的第一眼就是熟悉的样子，几乎零学习成本。如果你从没用过 VS Code，全新安装也没关系，跟着默认配置走就行，后面我们再慢慢调。

### 登录与模型选择

- 用 Google / GitHub 登录领试用额度
- 推荐模型：**Claude Fable 5 / Opus 4.8 质量最高**，写复杂逻辑、跨文件重构、需要"想清楚再下笔"的活交给它们最稳；**GPT-5.4 mini 更快出草稿**，写点小函数、快速试个想法、要它秒回的时候用它更爽
- 初期保持默认，之后按延迟和成本调

模型这块多讲两句，因为选错了会直接影响你的体验。质量高的模型（Claude Fable 5 / Opus 4.8）胜在"理解力"——你给它一段乱七八糟的需求，它能读懂你真正想要什么，改出来的代码也更少埋坑；代价是稍慢一点、成本高一点。快模型（GPT-5.4 mini）胜在"手速"，适合你已经想清楚要什么、只是懒得手敲的场景。我的习惯是：默认挂质量高的，遇到"我就要个循环""帮我改个变量名"这种小活临时切快的。你不用一开始就纠结，保持默认用一周，自己就有手感了。

> **术语：模型（Model）**
> 一句话解释：就是背后帮你写代码的那个 AI 大脑，Cursor 只是它的"身体"（编辑器界面）。
> 形象比喻：同一个 IDE 好比同一辆车，换模型就像换发动机——有的猛有的省油。
> 工作中怎么用：右下角或命令面板里能随时切换，一个项目里不同任务用不同模型很正常。
> 最常踩的坑：以为"Cursor 卡"，其实是选了个又慢又贵的模型在写简单活，换个快模型立刻顺。

### 必备快捷键

- Cmd/Ctrl + K：在光标处生成/修改代码（最常用）
- Cmd/Ctrl + L：打开 chat 侧栏多轮讨论
- Cmd/Ctrl + Shift + I：打开 Composer 跨文件重构
- Cmd/Ctrl + Shift + P：命令面板，切模型/开关功能

这四个快捷键是你每天要按几百次的东西，我一个个讲透，顺便告诉你"什么时候该按哪个"，因为按错了会很别扭。

**Cmd/Ctrl + K —— 就地生成/改代码。** 这是最高频的一个。你把光标放在某一行，或者选中一段代码，按 K，弹出一个小输入框，你用中文说"给这个函数加上参数校验"，它就在原地改给你看。使用场景：你正在写一个函数，写到一半卡住了，选中已有的几行按 K 说"帮我补全剩下的逻辑"——不用离开当前位置，改动直接内联显示，你按回车接受、按 Esc 拒绝。它的特点是"点状、精准"，只动你选中的地方。

**Cmd/Ctrl + L —— 打开右侧聊天栏讨论。** 这个是"聊天"，不是"直接改代码"。使用场景：你拿到一段别人写的、看不懂的代码，选中它按 L，在侧栏问"这段在干嘛？为什么用递归？"——它会解释，你可以追问，来回聊好几轮。或者你还没想清楚怎么写，先按 L 跟它讨论方案、让它给几个思路，聊明白了再回编辑器动手。K 是"动手"，L 是"动脑"，这是两者最大的区别。

**Cmd/Ctrl + Shift + I —— 打开 Composer 跨文件重构。** 这是重头戏，也是 Cursor 比原生 VS Code 强的地方。使用场景：你要做的事横跨好几个文件——比如"把整个项目里的 `getUser` 重命名成 `fetchUser`，相关的 import 和调用都一起改"，或者"新加一个用户注册功能，要动路由、控制器、数据库模型三个文件"。这种活用 K 一个文件一个文件改会累死，Composer 能一次性理解多个文件的关系、成批改动、最后给你一个统一的 diff 让你审。

> **术语：Composer 跨文件重构**
> 一句话解释：让 AI 一次性在多个文件里协同改动的"大工程"模式。
> 形象比喻：K 是拿手术刀点一下，Composer 是叫来一个施工队，同时开好几个房间的工。
> 工作中怎么用：做加新功能、大范围重命名、拆分模块这类"牵一发动全身"的活时用它。
> 最常踩的坑：什么都想用 Composer，结果它一次改太多，你根本审不过来——小改动老老实实用 K。

**Cmd/Ctrl + Shift + P —— 命令面板。** 这是 VS Code 原生就有的万能入口，Cursor 也保留了。按下去弹出一个搜索框，你输关键词就能找到几乎所有功能：切换模型、开关某个设置、格式化文件、装插件……记不住某个功能的入口时，Shift + P 搜一下最快。

### 配置建议

- 设置里开 "Auto Suggest" 及时补全
- 常用目录加入 Workspace 便于索引；大项目先跑一次全局搜索预热索引
- 团队协作配好 Git 凭证，保证 git status 正常

再补几条我踩过坑之后总结的：

**Auto Suggest（自动补全）一定要开。** 开了之后你敲代码时，AI 会像"读心术"一样在光标后面用灰字预判你接下来要写什么，按 Tab 就接受。它跟按 K 的区别是：K 是你主动喊它来改，Auto Suggest 是它在你打字过程中默默帮你续写。刚开始你可能觉得灰字有点晃眼，用两天就离不开了。

**关于索引。** Cursor 要先把你的项目"读一遍、建个目录"，之后你问它"用户登录的逻辑在哪个文件"它才答得上来。大项目第一次打开时，让它在后台跑一会儿把索引建完，别急着提问，不然它对项目还"一无所知"，答出来的东西张冠李戴。

> **术语：索引（Codebase Indexing）**
> 一句话解释：Cursor 把你整个项目扫一遍、建立"哪段代码在哪"的地图，好让 AI 能全局理解你的工程。
> 形象比喻：像图书管理员先把书全部登记进系统，之后你要哪本它才能秒定位。
> 工作中怎么用：新项目打开后等它索引建完再提整体性问题（"这个功能怎么实现的"）。
> 最常踩的坑：项目刚打开索引没建完就急着问，AI 答得驴唇不对马嘴，你还以为它笨。

**Git 凭证配好。** Cursor 很多协作和版本相关的能力都依赖底层的 Git 正常工作，配好凭证、保证在项目里跑 `git status` 不报错，后面提交代码、看改动历史才顺。快速自检一下：

```bash
# 在你的项目目录里跑，能正常显示状态就说明 Git 通了
git status

# 顺手确认一下身份配好了（提交代码要用）
git config user.name
git config user.email
```

### Cursor 和原生 VS Code 有啥不一样

既然 Cursor 是"套壳 VS Code"，很多人问那我干嘛不直接用免费的 VS Code 装个插件。差别主要在"AI 是不是长在骨子里"：

- **Composer 跨文件协作**是 Cursor 的招牌能力，原生 VS Code 加插件很难做到这种多文件协同改动的顺滑度。
- **`@` 引用上下文**：在聊天或 K 里打 `@`，可以精准把某个文件、某个函数、甚至某个文档喂给 AI 当参考，让它"看着这个来改"，而不是瞎猜。这是 Cursor 里非常好用的一招。
- **深度的项目索引**让 AI 能全局理解你的代码，回答"这个功能散落在哪几个文件"这类问题。

> **术语：@ 引用上下文**
> 一句话解释：在对话里用 `@` 手动指定"参考哪个文件/函数"，把准确的上下文塞给 AI。
> 形象比喻：像开会时你把某份文件推到对方面前说"照这个来"，省得他自己乱翻。
> 工作中怎么用：改 A 文件但逻辑要参照 B 文件时，`@B文件` 一提，AI 就不会两头对不上。
> 最常踩的坑：不用 @ 直接让它改，它凭空脑补相关代码，结果对不上你项目的真实结构。

### 隐私与联网注意点

用 AI IDE 绕不开一个问题：你的代码会不会被传出去。定性地说几条，别当成绝对结论、以你所在公司/项目的合规要求为准：

- **`.cursorignore` 文件**：类似 `.gitignore`，你可以在项目根目录建一个 `.cursorignore`，把不想让 AI 碰、不想被索引的文件/目录写进去（比如含密钥的配置、敏感数据）。这是你控制"哪些内容参与 AI 处理"的第一道闸。
- **联网**：AI 补全和对话本质上要把相关上下文发到模型服务端处理，所以是需要联网的。断网时 AI 功能基本用不了。
- **敏感信息**：不管工具怎么设置，最稳的做法永远是——**真实密钥、token、密码绝对不要写进代码文件**（走环境变量），这样就算内容参与了处理，泄露的也不是活凭证。

（以上是原则性说明，Cursor 具体的数据处理策略、企业版隐私模式等，以官网最新条款为准，我这里不编具体数字。）

### 常见坑

- **登录后模型用不了 / 一直转圈**：多半是网络问题或试用额度设置没生效，先在命令面板（Shift + P）里确认当前选的是哪个模型，换一个再试。
- **AI 回答对不上项目**：十有八九是索引没建完，或者没用 `@` 把相关文件喂给它——大项目先等索引，改动前先 `@` 引用。
- **什么都用 Composer**：小改动用 Composer 反而慢、diff 大到审不过来，记住"点状改动用 K，跨文件工程用 Composer"。
- **灰字补全乱按 Tab**：Auto Suggest 很好用，但别闭眼接受，看清楚它续写的是不是你要的再按 Tab，不然会引入你没注意到的逻辑。
- **Git 没配就开干**：`git status` 报错还硬写，后面提交、看历史全乱，第一步就把 Git 凭证配好。

### 小结

这一章我们做了三件事：把 Cursor 装好并继承了原有配置；搞清楚了 2026 年该用哪个模型（质量优先 Claude Fable 5 / Opus 4.8，求快用 GPT-5.4 mini）；把四个每天要用几百次的快捷键——K（就地改）、L（聊天讨论）、Composer（跨文件重构）、Shift + P（命令面板）——的用途和使用场景过了一遍。再加上索引、Auto Suggest、`@` 引用、`.cursorignore` 这几个概念，你现在手里的这台 IDE 已经是一个"会写代码的搭档"，而不只是个文本编辑器。环境这一关过了，后面学起来会轻松很多。

### 练习

- 新建 hello.ts，按 Cmd/Ctrl + K 输入"写一个打印 Hello JR Academy 的函数并导出"
- 观察生成代码，改 prompt 加参数校验，注意响应速度

做完上面两步，再多练三个动作，把这章学到的都串起来：

1. **试 L 的讨论能力**：选中刚生成的函数，按 Cmd/Ctrl + L，在侧栏问"如果我想让这个函数支持传入名字参数，该怎么改？"——先聊清楚思路，再回编辑器动手，体会 L（动脑）和 K（动手）的分工。
2. **试 `@` 引用**：再建一个 `config.ts`，随便写个常量导出。回到 hello.ts 里按 K，用 `@config.ts` 引用它，让 AI"用 config 里的常量作为默认名字"，感受一下有没有 `@` 引用的区别。
3. **对比两个模型的手感**：把同一个 prompt（比如"给函数加参数校验并抛出友好错误"）分别用 Claude Fable 5 / Opus 4.8 和 GPT-5.4 mini 各跑一次，注意两点——**响应速度**谁快，**改出来的代码**谁更周到。跑完你就明白前面说的"质量 vs 速度"到底是什么感觉了。

```typescript
// hello.ts —— 你可能会得到类似这样的结果（加了参数校验之后）
export function sayHello(name: string): void {
  if (!name || name.trim() === "") {
    throw new Error("name 不能为空");
  }
  console.log(`Hello ${name}, welcome to JR Academy`);
}
```
