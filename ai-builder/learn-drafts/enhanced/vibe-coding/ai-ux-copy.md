## AI 辅助 UX 文案与交互

先说个我自己踩过的坑。我第一次独立做一个登录页，功能全通了：账号密码能提交、校验能拦、接口能连。我很得意，发给设计出身的同事看。她只看了三秒就说：「你这个按钮写『提交』，错误提示写『Error: invalid credentials』，用户看到会懵。」

我当时的反应是：这不都是小事吗？功能才是重点。

后来我才明白，用户根本不关心我后端写得多干净。他们只看得见界面上那几行字。工程师写的文案有个通病——又干又冷、只描述系统状态、不管人的感受。「提交」「Error 400」「暂无数据」，这些字对机器很准确，对人很冰冷。而这些字，恰恰是用户对你产品的第一印象、也常常是唯一印象。

好消息是：AI 不只是帮你写代码，它现在是你身边那个「随叫随到的产品文案 + UX 顾问」。界面文案、空状态、可访问性提示、微交互脚本——这些以前要么靠自己硬憋、要么要拉设计师排期的活，现在都能用 AI 加速几倍。这一章就讲怎么把它用透。

先立一个观念——不懂这些会怎样：

- 按钮文案干巴巴 → 用户不敢点，转化掉一截。
- 空状态一片空白 → 新用户第一次进来看到啥都没有，以为产品坏了，直接走人。
- a11y（可访问性）全靠猜 → 用屏幕阅读器的用户、只能用键盘的用户，你的产品对他们等于不存在，严重的还违反当地无障碍法规。
- 加载 / 成功 / 失败没有反馈 → 用户点完不知道发生了啥，反复乱点。

下面几个术语你会反复遇到，先用「四行模板」把它们钉死。

**术语：空状态 empty state**
- 一句话解释：界面上「本该有内容、但现在还没有内容」时展示的那个页面（比如收件箱一封邮件都没有、搜索无结果、新用户还没建任何项目）。
- 形象比喻：像你搬进新家的第一晚，屋里空荡荡。你需要的不是一间空房，而是一张纸条告诉你「插座在这、水电已通、第一步先干嘛」。
- 工作中怎么用：任何列表 / 数据展示区都要单独设计空状态，写清「为什么是空的 + 现在能做什么（放个 CTA 按钮）」。
- 最常踩的坑：直接留白，或者只写一句冷冰冰的「暂无数据」。用户会以为是 bug，不知道下一步该干嘛。

**术语：可访问性 a11y**
- 一句话解释：让有视觉 / 听觉 / 行动障碍的人也能正常用你的产品（a11y = accessibility，中间 11 个字母的缩写）。
- 形象比喻：像给建筑修无障碍坡道。坡道不是给残障人士的「特殊照顾」，推婴儿车的、拖行李箱的所有人都受益。
- 工作中怎么用：给图片加 alt、给表单控件加 label、保证能用键盘 Tab 走完整个流程、错误提示要能被屏幕阅读器读出来。
- 最常踩的坑：只用颜色传递信息（比如「红色框代表错」），色盲用户完全看不出来；还有只顾着鼠标点，忘了很多人只用键盘。

### 生成一致的文案语气

原始 prompt 是这样的：

> 你是产品文案，为登录页生成 标题/副标题/按钮文案/错误提示。品牌语气：温暖、清晰、有行动感。约束：不超过30字；错误提示包含解决方案。让 AI 输出多个备选并标注差异。

我们把它拆开讲，为什么每一句都不能省：

- **「你是产品文案」**——给 AI 定角色。不定角色，它会像写技术文档一样输出，干、准、冷。定成产品文案，它才会站在用户情绪的角度写。
- **「品牌语气：温暖、清晰、有行动感」**——这是最关键的约束。没有它，AI 每次的语气都飘。有了它，你后面所有页面的文案才能统一成一个「人格」。（进阶做法：把这段语气描述存成一个文件，比如 `docs/tone.md`，以后每次让 AI 写文案都 `@docs/tone.md` 带上，全站语气就锁死了。）
- **「不超过30字 / 错误提示包含解决方案」**——具体的可验证约束。「不超过30字」逼它别啰嗦；「包含解决方案」直接把工程师最容易忘的事变成硬要求。
- **「输出多个备选并标注差异」**——不要让 AI 一锤定音。让它给 3 个版本、并说清每个版本偏向哪种感觉，你来选。你比 AI 更懂你的用户。

怎么判断 AI 给的好不好？看一个对比就懂了。

烂文案（工程师默认会写的）：

```
标题：登录
副标题：请输入您的凭据
按钮：提交
错误：Error: Authentication failed (401)
```

好文案（带情绪、带出路）：

```
标题：欢迎回来
副标题：登录继续你上次的进度
按钮：登录我的账号
错误：邮箱或密码不对。忘记密码了？点这里重置 →
```

差别在哪？好文案里，每一行都在「帮用户往前走一步」，而不是「向用户汇报系统状态」。尤其错误提示——烂的那条只告诉你「错了」，好的那条告诉你「哪里错了 + 现在怎么办」。这就是「包含解决方案」这个约束的价值。

在 Cursor 或 Claude Code 里，你甚至可以让它直接改进现有组件，而不是只给纯文本：

> @components/LoginForm.tsx 把里面所有面向用户的文案改成「温暖、清晰、有行动感」的语气，错误提示要包含解决方案，每条不超过30字。只改文案字符串，不动逻辑。改完给我 diff。

最后那句「只改文案不动逻辑、给我 diff」很重要——它把 AI 的改动范围框死，你 review 起来一目了然，不会一不小心把校验逻辑也给你「优化」没了。

### 设计空状态与引导

原文的做法保留并讲透：

- 要求 AI 输出空态文案 + 示例插图描述 + CTA 链接
- 补充「触发条件」和「成功指标」便于验证

为什么要 AI 顺带输出「示例插图描述」？因为空状态光有文字还是空。一句「你还没有任何项目」配一张友好的小插画，感受完全不同。你不一定要自己画，但让 AI 描述出来（比如「一个打开的空盒子，暖色调，旁边一支笔」），你就能拿这段描述去找现成图库、或喂给出图工具。

为什么要「触发条件 + 成功指标」？因为空状态有好几种，别混成一个：

- 新用户第一次来（还没建过东西）→ 引导他建第一个。
- 搜索 / 筛选后没结果 → 提示他换个关键词，别让他以为产品没内容。
- 全删光了（曾经有、现在清空了）→ 语气可以轻松点，「都清完啦，休息一下？」

一个更完整的例子。假设做一个待办应用的空收件箱：

```jsx
function EmptyInbox() {
  return (
    <div role="status" style={{ textAlign: 'center', padding: '48px' }}>
      {/* 插图描述：一只打勾的信封，暖橙色，轻松风格 */}
      <img src="/empty-inbox.svg" alt="收件箱已清空" width={120} />
      <h3>收件箱清空了 🎉</h3>
      <p>所有待办都处理完了。要不要现在加一条新的？</p>
      <button onClick={onAddTask}>+ 新建待办</button>
    </div>
  );
}
```

注意里面的 `role="status"` 和 `alt`——这就是把 a11y 顺手带进去了，屏幕阅读器会读出「收件箱已清空」。这也引出下一节。

### 可访问性辅助

原始 prompt：

> 审查 @components/Form.tsx 的可访问性：label/aria/键盘导航/错误提示。输出问题列表和修复 patch。

让 AI 生成 a11y 检查表和 patch。

这个 prompt 好在两点：一是 `@components/Form.tsx` 直接把真实文件喂给它，它审的是你的真代码不是泛泛而谈；二是明确列了四个检查维度（label / aria / 键盘导航 / 错误提示），AI 就不会漏。

它一般会返回这样的东西：

```
问题清单：
1. [严重] <input> 没有关联 <label>，屏幕阅读器读不出这是什么字段。
2. [中] 错误提示是纯红色文字，无 aria-live，屏幕阅读器不会主动播报。
3. [中] 自定义下拉框用 <div> 做的，不能用键盘 Tab / 方向键操作。
4. [轻] 提交按钮 loading 时没有 aria-busy。
```

配套的 patch 可能长这样：

```jsx
// 修复 1 + 2：label 关联 + 错误可被朗读
<label htmlFor="email">邮箱</label>
<input id="email" type="email" aria-describedby="email-err" />
{error && (
  <p id="email-err" role="alert" style={{ color: '#c00' }}>
    {error}
  </p>
)}
```

怎么判断这个 patch 好不好？一个笨办法特别管用：改完后，把鼠标丢一边，只用键盘（Tab 切换、Enter 确认）走一遍你的表单。如果能顺利走完、每一步焦点都看得清、报错能听见——就成了。走不通，就把卡住的地方再喂回给 AI：「我用键盘 Tab 到下拉框后按方向键没反应，帮我修」。AI 审 a11y 很强，但最终验证要靠你亲自用一遍。

**术语：微交互 microinteraction**
- 一句话解释：用户做一个小动作时，界面给的那一点点即时反馈（按钮按下去有个下沉、点赞时爱心弹一下、加载时转圈）。
- 形象比喻：像你按电梯按钮，它亮一下灯。灯不影响电梯来不来，但没有这个灯你会一直怀疑「我到底按上了没」，忍不住狂按。
- 工作中怎么用：给每个关键操作配「进行中 / 成功 / 失败」三态反馈，让用户永远知道系统在干嘛。
- 最常踩的坑：只做了「成功」态，忘了「加载中」和「失败」态。用户点完按钮，界面半天没动静，就开始重复点、或者以为坏了走人。

### 微交互与动效脚本

原文保留：

- 让 AI 设计「加载、成功、失败」的动画或状态机描述，再生成代码
- 要求给出「终端/移动端」差异处理

为什么要 AI 先给「状态机描述」再给代码？因为微交互最容易漏状态。你脑子里往往只想着「点了就成功」，但真实世界里还有「正在请求」「请求失败」「网慢超时」。让 AI 先把状态画出来，你一眼就能看出漏没漏：

```
idle（初始）
  → 用户点击 → loading（转圈，按钮禁用防重复点）
      → 成功 → success（打勾，2 秒后回 idle）
      → 失败 → error（抖一下 + 显示可重试的错误提示）
```

有了这张图，再让它生成代码就稳了：

```jsx
const [state, setState] = useState('idle');

async function handleSubmit() {
  setState('loading');
  try {
    await save();
    setState('success');
    setTimeout(() => setState('idle'), 2000);
  } catch {
    setState('error'); // 保留在 error，让用户能看到并重试
  }
}

// 按钮根据 state 变样子
<button disabled={state === 'loading'} aria-busy={state === 'loading'}>
  {state === 'loading' ? '保存中…' :
   state === 'success' ? '已保存 ✓' :
   state === 'error'   ? '失败，重试' : '保存'}
</button>
```

「终端 / 移动端差异处理」也别忘让 AI 提醒你：手机上没有鼠标 hover，所以别把关键反馈只挂在 hover 上；手机屏小，动效要更克制，别让页面乱抖影响阅读。

### 常见坑

- **让 AI 一次给一个「最终版」文案就直接用**。永远让它给多个备选、标差异，你来选。你比 AI 懂你的真实用户。
- **语气不统一**。这个页面 AI 写得活泼、那个页面写得严肃，全站像好几个人写的。解决办法：把品牌语气写进一个文件，每次 `@` 带上。
- **只做成功态，不做加载和失败态**。这是工程师最常见的 UX 债。让 AI 输出状态机时，主动问它「失败和超时怎么办」。
- **a11y 只看 AI 报告不亲自验**。AI 能揪出大部分问题，但「键盘走一遍、屏幕阅读器听一遍」这种真人验证不能省。
- **空状态留白或只写「暂无数据」**。至少要有一句话解释 + 一个能点的下一步。
- **把 AI 当设计师全权托管**。AI 是加速器，不是拍板人。它给方案，你做判断——尤其涉及品牌调性和真实用户的地方。

### 小结

这一章的核心就一句话：**界面上给用户看的每一个字、每一个反馈，都是产品的一部分，值得像写代码一样认真对待——而 AI 能让你做这件事快很多。**

具体记住这几点：
- 文案定角色 + 定品牌语气 + 给可验证约束（字数、必含解决方案），并要多个备选。
- 空状态要分清是「新用户 / 无搜索结果 / 已清空」哪一种，各写各的引导 + CTA。
- a11y 让 AI 审 label / aria / 键盘 / 错误提示，但最后自己用键盘走一遍。
- 微交互先让 AI 画状态机（idle / loading / success / error），再生成代码，别漏状态。
- AI 是你的 UX 副驾，不是自动驾驶——它出方案，你拍板。

### 练习

（在原练习基础上做，先完成原三步，再加做拓展。）

**基础（原练习，必做）**：选一个页面组件，让 AI：
1. 生成统一语气的文案；
2. 给出空态与 CTA；
3. 输出 a11y 修复建议。

然后比较 AI 方案与现有文案，择优合并——注意是「合并」，不是无脑全换，好的地方留、AI 更好的地方换。

**拓展一（把语气固化）**：把你选定的品牌语气写成一份 `tone.md`（三五句话即可），下次让 AI 写任何文案时都 `@tone.md` 带上，观察前后一致性有没有变好。

**拓展二（真人验 a11y）**：改完 a11y 后，把鼠标放到一边，只用键盘 Tab + Enter 走完整个组件流程。记录卡住的地方，把它原样喂回给 AI 让它修，直到能顺畅走完。

**拓展三（补齐微交互）**：给这个组件的主操作加上完整的「加载 / 成功 / 失败」三态反馈。先让 AI 画状态机，你确认没漏状态后再让它生成代码。特别检查：失败态能不能重试？加载态有没有防重复点击？
