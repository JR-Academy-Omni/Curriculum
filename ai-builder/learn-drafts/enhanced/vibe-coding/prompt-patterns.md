## 高质量 Prompt 模板

我先讲个我上周真实碰到的事。有个同事在 Cursor 里让 AI「帮我写个登录表单验证」，AI 唰唰给了一段代码，看着挺像回事——结果引了个他项目里根本没装的表单库，验证逻辑还跟他们现有的 `useToast` hook 撞了，最后他花了 40 分钟把 AI 生成的东西一点点拆掉重接。他跑来问我：「是不是 AI 变笨了？」

不是 AI 笨，是 prompt 没说清。同样这个需求，我让他换成一句结构化的写法——把「用的什么技术栈、已经有哪些能复用的组件、不许引新依赖、要什么格式的输出、怎么算做对了」全写进去，AI 第二次直接给出了能跑、能过 lint、还带了两个测试用例的代码,一次成型。

这就是这一章要讲的东西：**别再靠运气写 prompt**。你把要求写模糊，AI 就只能猜，猜错了你返工；你把要求写成固定模板，AI 每次都稳定输出你要的结果。下面 3 个模板是我自己天天在用的，直接抄。

> **术语：结构化 Prompt（structured prompt）**
> 就是不用「一句话随口说」，而是按固定的几个字段（任务/背景/约束/输出/验收）把需求填满的写法。
> 形象点说，随口 prompt 像你跟外卖备注「少放点辣」——「少」是多少？店家全靠猜；结构化 prompt 像「微辣，两根小米辣，能吃出味但不冲」——没有歧义。
> 工作中怎么用：任何要 AI 产出代码/文档/方案的场景，先花 30 秒把字段填齐，比事后返工快 10 倍。
> 最常踩的坑：以为「写详细」= 啰嗦一大段。不是。是把关键字段填全，废话反而要删——AI 读的是信息密度，不是字数。

在日常开发里，用固定模板可以快速复制出稳定结果。下面 3 个常用模式直接套用。

### 模式1：任务+背景+约束+输出+验收

（示例：[任务]实现登录表单前端校验 [背景]React+TS，已有 Form/Input 组件、可复用 hooks/useToast [约束]不引新依赖，失败返回具体字段错误 [输出]完整代码块含类型，附2个输入输出示例 [验收]通过 ESLint，说明覆盖哪些边界）适用于绝大多数功能开发，关键是写清"项目现状"和"验收标准"。

**为什么这五个字段一个都不能省？** 我见过太多人只写「任务」，剩下四个全靠 AI 脑补，结果就是开头那个 40 分钟返工的故事。逐个说下每个字段挡掉什么坑：

- **任务**——你想要它干什么。省了 AI 直接懵。这个大家一般不会漏。
- **背景**——你的项目现在长什么样。这是最容易被低估的。你不说技术栈，AI 可能用 Vue 语法写你的 React 项目；你不说「已有 useToast」，AI 就自己造一个新的 toast，跟你现有的打架。
- **约束**——什么不许做。**这是最常被漏、也最要命的一个**。「不引新依赖」这五个字，能挡掉 AI 一时兴起给你装三个 npm 包的惨案。约束就是你的护栏，不写护栏，AI 就在悬崖边自由发挥。
- **输出**——你要什么形式。要完整代码块还是只要 diff？要不要带类型？要不要示例？不说清，AI 给你一段夹在解释里的伪代码，你还得自己抠出来。
- **验收**——怎么算做对了。**这个跟「约束」并列第一容易被漏**。不写验收，AI 觉得「能编译就行」，但你心里想的是「要处理空输入、超长输入、特殊字符」。验收标准就是你脑子里那把尺，得摆到台面上。

> **术语：验收标准（acceptance criteria）**
> 一句话：判断「这活儿到底做完了没」的那把尺子——满足了就算过，没满足就打回。
> 形象比喻：像验房时的清单——「墙面无裂缝、水管不漏、开关都能用」，逐条打钩才收房。没有清单，你只能凭感觉说「差不多吧」，然后住进去才发现漏水。
> 工作中怎么用：写 prompt 时把「我怎么知道它做对了」提前列成 2-4 条可勾选的条件（通过 lint、覆盖 X 个边界、不改动 Y）。
> 最常踩的坑：写成「代码质量要高」这种没法验证的虚话。验收标准必须是能一眼判断真假的——「通过 ESLint」可验证，「代码优雅」不可验证。

**模糊 → 结构化 改写对比：**

```
❌ 模糊版
帮我写个登录表单校验

✅ 结构化版
[任务] 给登录表单加前端校验
[背景] React 19 + TypeScript 项目，已有 <Form>/<Input> 组件和 useToast hook（在 hooks/useToast.ts），
       表单字段是 email + password
[约束] 不引入任何新依赖；校验失败时返回具体到字段的错误信息，别只弹一个笼统的"输入有误"
[输出] 一个完整的可复制代码块，含 TypeScript 类型；末尾附 2 个「输入 → 输出」示例
[验收] 能通过项目现有 ESLint 配置；说明你覆盖了哪些边界情况（空值、格式错误、超长）
```

你能明显感觉到，右边那版 AI 几乎没有猜的空间了。这就是目标。

### 模式2：角色+步骤+检查表

（示例：你是代码审查助手，按步骤：1)读 @pages/login.tsx 与 @components/Form.tsx 2)找可访问性问题 3)输出修复 patch(md代码块)再列检查表。检查表：可键盘操作、字段有 label、错误信息可被屏幕阅读器读取）适合重构/Review/a11y 等需列表化检查的任务。

**为什么要给它「角色」和「步骤」？** 因为不给角色，AI 是个啥都会一点的通才；给了「你是代码审查助手」，它就切换到「挑刺模式」，标准会严很多。不给步骤，它可能读完文件直接给你一坨改动；给了 1-2-3，它会老老实实先读、再找问题、最后才动手——中间过程你能看见，也能中途叫停。

那个「检查表」尤其值钱。它把「做得好不好」这种主观判断，变成了一条条能打钩的客观项。做完 review，你拿着检查表一条条对，漏没漏一目了然。这跟模式1里的「验收」是一个思路——把标准显性化。

**模糊 → 结构化 改写对比：**

```
❌ 模糊版
帮我看看这个登录页有没有无障碍问题

✅ 结构化版
你是代码审查助手，专门查前端可访问性（a11y）。按以下步骤做：
1) 读 @pages/login.tsx 和 @components/Form.tsx
2) 找出所有可访问性问题（重点看键盘操作、label、屏幕阅读器）
3) 先输出修复的 patch（unified diff，放 md 代码块里），再附一张检查表
检查表逐条打钩：
[ ] 所有交互元素可纯键盘操作（Tab/Enter）
[ ] 每个输入框都有关联的 <label>
[ ] 错误信息能被屏幕阅读器读到（aria-live 或等价方案）
```

左边那句 AI 大概率给你一段泛泛而谈的「建议加 label、注意对比度」；右边这版你拿到的是能直接 apply 的 diff + 一张验收清单。

### 模式3：Diff/Patch 指令

（示例：生成针对 @components/Button.tsx 的 patch：primary 按钮背景改 #0f172a，hover 加阴影；保持 props 不变；输出 unified diff，不含无关文件）在已有代码上做小改动时，patch 能减少误伤。

**为什么改小地方要用 patch 而不是让 AI 重写整个文件？** 因为你让 AI「重新给我这个 Button 组件」，它很可能顺手把你的注释删了、把某个 prop 名改了、把一段它「觉得没用」的逻辑清理掉——这叫误伤，是 vibe coding 里翻车的重灾区。而 patch（diff）只描述「哪几行变成什么」，其余原封不动，AI 想动别的地方都没机会。

在 2026 年这一点尤其重要。Claude Code、Cursor 的 Agent 模式、Codex CLI 这些工具本来就是围绕「读文件 → 生成 diff → 你审 diff → apply」这个循环设计的。你主动要 unified diff，等于顺着工具的纹理来，AI 给的东西直接就能进它的 review 面板，你扫一眼就知道动了哪几行。

> **术语：unified diff（统一差异格式）**
> 就是 `git diff` 打出来的那种格式——`-` 开头的是删掉的行，`+` 开头的是加上的行，其余是没动的上下文行。
> 形象比喻：像老师批改作文时用红笔——划掉一句（`-`）、旁边补一句（`+`），整篇作文不用重抄。你一眼看得出改了哪、没动哪。
> 工作中怎么用：要 AI 在已有代码上做小修改时，明确说「输出 unified diff」，然后在 Cursor/Claude Code 里直接 apply，改动可控可回滚。
> 最常踩的坑：patch 的上下文行（没有 +/- 的那几行）跟你本地代码对不上，apply 就会失败。所以 @ 引用要指对文件，别让 AI 凭空造上下文。

**模糊 → 结构化 改写对比：**

```
❌ 模糊版
把 Button 的主按钮改深色一点，加个 hover 效果

✅ 结构化版
针对 @components/Button.tsx 生成一个 patch：
- primary 按钮背景色改成 #0f172a
- hover 时加一个柔和阴影（box-shadow: 0 2px 8px rgba(0,0,0,.15)）
约束：保持所有现有 props 和类型签名不变，不要碰其他任何文件
输出：unified diff 格式，只包含 Button.tsx 的改动
```

「深一点」是多深？「hover 效果」是啥效果？左边全是给 AI 的自由发挥题。右边把色值、阴影、边界（不碰别的文件）全钉死了，AI 只能照做。

### 常见坑

踩过的坑分享给你，能省不少来回：

1. **只写「任务」，指望 AI 猜背景。** 最高频的坑。AI 不知道你的技术栈、你有哪些现成组件，它只能按最通用的假设来，然后就跟你项目现状对不上。背景字段花 20 秒填，省你半小时。
2. **漏掉「约束」，然后惊讶 AI 装了新依赖。** 「不引新依赖」「保持 props 不变」「别碰其他文件」——这些护栏不写，AI 就当你默许它自由发挥。
3. **验收标准写成虚话。** 「代码质量要高」「要健壮」——这种 AI 没法验证，等于没写。改成「通过 ESLint」「覆盖空输入和超长输入」这种能一眼判真假的。
4. **要改三行，却让 AI 重写整个文件。** 用模式3 的 patch，别用「重新给我一份」。重写=给 AI 一张误伤的许可证。
5. **@ 引用指错文件或不指。** 2026 的工具（Claude Code / Cursor / Codex CLI）都支持 `@文件名` 把真实代码喂给 AI。你不 @，AI 就凭想象写；你 @ 错了，patch 的上下文对不上，apply 直接失败。
6. **一个 prompt 塞五个不相关的任务。** AI 会顾此失彼。宁可拆成几个结构化 prompt 分别跑，也别糊成一大坨。
7. **不给示例，却期待特定格式。** 如果你对输出格式有讲究，模式1 里那句「附 2 个输入输出示例」很关键——用例子告诉 AI 你要的样子，比用文字描述准得多。

> **术语：few-shot 示例（少样本示例）**
> 就是在 prompt 里直接塞 1-3 个「输入长这样 → 输出应该长这样」的例子，让 AI 照着葫芦画瓢。
> 形象比喻：像教新同事做表格，与其口头讲半天格式，不如直接甩他一份填好的样表——「照这个来」，秒懂。
> 工作中怎么用：当你对输出格式/风格有明确要求、又不好用文字讲清时，给 2 个例子最省事（这就是模式1 里「附 2 个输入输出示例」的作用）。
> 最常踩的坑：例子给得跟真实需求偏差太大，AI 会照着错的例子学。例子要典型、要贴近你真实场景。

### 小结

这一章其实就一句话：**别让 AI 猜，把要求写成模板**。

- 做**功能**用模式1（任务+背景+约束+输出+验收），重点是别漏「约束」和「验收」这两个最容易被跳过、又最挡坑的字段。
- 做**审查/重构/a11y**用模式2（角色+步骤+检查表），把主观判断变成可打钩的清单。
- 做**小改动**用模式3（Diff/Patch），只改该改的，避免 AI 顺手误伤。

三个模板背后是同一个道理：把你脑子里那些「默认的、没说出口的」前提（技术栈、护栏、验收尺子、输出格式）全部搬到台面上。你说得越具体，AI 猜得越少，你返工越少。刚开始会觉得「写这么多字好麻烦」，但用两周你就回不去了——因为一次成型的爽感，远超返工的痛苦。

### 练习

- 任选一个模式在 Cursor 里对当前项目跑一遍，看 AI 是否遵守格式
- 调整"约束""验收"字段观察输出变化，积累自己的模板库
- **进阶1**：拿你今天手头一个真实需求，先用「模糊版」写一遍 prompt 跑一次，再用对应模板改成「结构化版」跑一次，把两次输出并排对比——直观感受一下差距，这比看十遍讲解都管用。
- **进阶2**：故意从模式1 里删掉「约束」字段跑一次，再删掉「验收」跑一次，观察 AI 分别开始「自由发挥」什么。踩一遍你就永远记住这两个字段为什么不能省。
- **进阶3**：用模式3 让 Claude Code 或 Codex CLI 对一个真实文件生成 unified diff，然后真的 apply 一次。体会「只改该改的」跟「让它重写整个文件」在可控性上的区别，顺便熟悉 diff 的 apply/回滚流程。
- **收尾**：把你调顺的模板存进一个自己的 `prompts.md`，下次直接抄字段填内容。你的模板库就是你的生产力。
