学习目标

  • 掌握 AI 在入职后日常工作中的 4 大应用场景：编码、文档、Code Review、会议
  • 每个场景都有可以直接复制粘贴的 Prompt 模板
  • 理解 AI 辅助前后的时间对比，量化效率提升
  • 建立自己的 AI 工作流，把 AI 融入每天的工作节奏

核心概念

AI 不是单独的"工具时间"，是融入每个任务的习惯

很多人把 AI 当成一个单独的步骤："先自己想，想不出来再问 AI"。这是 2024 年的用法。2026 年的正确姿势是：每个任务开始时就打开 AI，让它参与整个过程。

就像你打开 IDE 不是一个额外步骤，AI 也不应该是。它是你的工作环境的一部分。

场景一：AI 辅助编码

日常工作流（以 Cursor 为例）：

  1. 收到一个 Jira ticket / 需求描述
  2. 在 Cursor 里 Cmd+K 打开 AI chat
  3. 把需求粘贴进去，让 AI 先给一个实现方案
  4. Review AI 的方案，调整后让它生成代码
  5. Review 生成的代码，手动修正细节
  6. 让 AI 生成 unit test
  7. 跑测试，修 bug

时间对比：

任务 不用 AI 用 AI 节省
写一个 CRUD API endpoint 2-3 小时 30-45 分钟 ~70%
写 unit test（80% 覆盖） 1-2 小时 15-20 分钟 ~80%
调试一个不熟悉的报错 30 分钟 - 2 小时 5-15 分钟 ~75%
学一个新的库/框架的用法 半天看文档 1 小时边问边做 ~60%
重构一个 500 行的文件 半天 - 一天 1-2 小时 ~70%

Prompt 示例 — 日常编码：

Prompt 1 — 从需求到实现方案：

我需要实现这个功能：[粘贴 Jira ticket 描述]

项目技术栈：React + TypeScript + NestJS + MongoDB

请给我：
1. 实现方案概述（前端 + 后端各需要改哪些文件）
2. 数据模型设计（如果需要新的 collection）
3. API 接口设计（method, path, request body, response）
4. 预估工时

先不要写代码，先给方案让我确认。

Prompt 2 — 生成 Unit Test：

以下是我的 service 代码：

[粘贴代码]

请用 Jest 为每个 public method 生成 unit test。要求：
- 每个方法至少 3 个 test case（正常、边界、异常）
- Mock 所有外部依赖（数据库、第三方 API）
- 使用 describe + it 结构
- 测试名要描述行为而不是方法名（"should return 404 when user not found" 而不是 "test findById"）

Prompt 3 — Debug 报错：

我遇到这个报错：

[粘贴完整的 error stack trace]

相关代码：

[粘贴出错的文件]

环境：Node.js 20, NestJS 10, MongoDB 7

请分析可能的原因并给出修复方案。如果有多个可能原因，按概率从高到低排列。

场景二：AI 辅助文档和邮件

写文档和邮件是最容易被 AI 加速的任务，因为格式相对固定，AI 只需要你的核心信息。

时间对比：

任务 不用 AI 用 AI 节省
写 Technical Design Doc 3-4 小时 1 小时 ~70%
写 Sprint Retro 总结 30 分钟 5 分钟 ~85%
回复英文邮件 15-20 分钟（非母语） 3 分钟 ~80%
写 PR Description 10-15 分钟 2 分钟 ~85%

Prompt — 写 Technical Design Doc：

我要为以下功能写一份 Technical Design Document：

[功能描述，2-3 段]

请按这个模板生成：
1. Overview（一段话总结）
2. Background & Motivation
3. Proposed Solution（架构图用文字描述）
4. API Design（table format）
5. Data Model Changes
6. Migration Plan
7. Risks & Mitigations
8. Timeline

语气：技术文档风格，简洁直接，避免废话。

Prompt — 专业英文邮件：

帮我用英文写一封邮件给团队。核心内容：
- 这周的 deployment 推迟到下周二
- 原因是发现了一个 payment 模块的 edge case bug
- 需要 QA 团队周五前完成回归测试
- 如果有问题找我

语气：专业但友好，不要太 formal。收件人是同事不是客户。

场景三：AI 辅助 Code Review

你可以在 review 别人代码时让 AI 先跑一遍，帮你找出明显问题。这不是偷懒 — 是让你把时间花在 AI 找不到的架构问题上，而不是格式和命名问题上。

时间对比：

任务 不用 AI 用 AI 节省
Review 一个 200 行 PR 20-30 分钟 10 分钟（AI 先扫 + 你看重点） ~60%
Review 一个 1000+ 行 PR 1-2 小时 30-40 分钟 ~60%

Prompt — Code Review 辅助：

请 review 以下代码变更。关注以下几点：
1. 有没有潜在的 bug 或 edge case？
2. 有没有性能问题（N+1 查询、不必要的循环、大数据量处理）？
3. 错误处理是否完善？
4. 命名是否清晰、代码是否可读？
5. 有没有安全隐患（SQL injection、XSS、敏感信息泄露）？

对于每个问题，给出具体的行号和修改建议。如果代码没问题就说没问题，不要为了凑数量硬挑毛病。

[粘贴代码 diff]

进阶用法 — Review 自己的代码：

提交 PR 之前先让 AI review 一遍自己的代码，提前修掉问题，减少 review round-trip。

Prompt — Self Review：

我准备提交这个 PR。请以一个严格的 Senior Engineer 的角度 review，假设你不了解背景上下文。

PR 描述：[简述改了什么]

变更文件：
[粘贴 git diff 或者各文件的代码]

请指出所有你会在 PR review 里提 comment 的问题。

场景四：AI 辅助会议准备

开会前 5 分钟用 AI 准备，比开会时毫无准备强 10 倍。

时间对比：

任务 不用 AI 用 AI 节省
准备 Sprint Planning 的估时 30 分钟自己想 10 分钟 AI 拆解 + 你调整 ~65%
准备 1:1 谈话要点 15 分钟 3 分钟 ~80%
写 Stand-up update 5 分钟想措辞 1 分钟生成 ~80%

Prompt — Sprint Planning 估时：

以下是下个 Sprint 的 tickets：

[粘贴 ticket 列表，每个包含标题和简要描述]

请帮我：
1. 把每个 ticket 拆成具体的子任务
2. 每个子任务估时（小时）
3. 标出哪些任务有依赖关系、需要按顺序做
4. 标出哪些任务风险高（可能超时）
5. 假设 Sprint 10 天、每天有效编码 5 小时，这些 ticket 能不能都做完？

Prompt — 1:1 准备：

我明天跟 Manager 有 1:1。过去两周我做了：
- [列举 3-4 件事]

我想讨论的问题：
- [列举 1-2 个]

请帮我组织一个 1:1 谈话提纲，包括：
1. 开场（30 秒，总结过去两周进展）
2. 我想讨论的问题的表达方式（怎么说不会让 manager 觉得我在抱怨）
3. 我应该主动问 manager 什么问题

操作步骤

Step 1：盘点你一天的工作任务

列出你今天（或这周）要做的所有工作任务。对每个任务标记：

  • ✅ AI 能显著加速（编码、文档、邮件）
  • ⚠️ AI 能辅助但需要大量人工判断（架构设计、需求讨论）
  • ❌ AI 帮不上忙（人际沟通、团队管理、创意决策）

Step 2：选 3 个 ✅ 任务，今天全程用 AI 完成

用上面的 Prompt 模板。记录每个任务的实际耗时，跟你平时不用 AI 的耗时对比。

Step 3：建立你的个人 Prompt 库

把你常用的 Prompt 保存下来。推荐用 Notion 或者 GitHub 私有 repo，按场景分类：

  • 编码 / Coding
  • 测试 / Testing
  • 文档 / Documentation
  • 邮件 / Communication
  • 会议 / Meetings
  • Debug / Troubleshooting

动手练习

练习 1：AI 编码实战

选一个你熟悉的技术栈，用 AI 从零完成一个小功能：

  • 一个 Todo API（CRUD）+ 简单前端
  • 全程使用 Cursor 或 Copilot
  • 记录总耗时，目标：1 小时内完成
  • Review AI 生成的每一行代码 — 至少找出 3 个你会手动修改的地方

练习 2：AI 文档实战

用 AI 生成以下文档之一：

  • 一个你做过的项目的 Technical Design Doc
  • 一封向 Manager 申请 remote 工作的邮件
  • 一份 Sprint Retro 总结（自己编一个场景也行）

生成后 review：哪些部分直接能用？哪些需要修改？AI 有没有编造事实？

练习 3：AI Code Review 实战

去你的 GitHub（或者随便找一个开源 PR），把 diff 粘贴给 AI 让它 review。对比 AI 的 review 和实际的 reviewer comments — 重合度有多少？AI 有没有找到人类没注意到的问题？

常见问题

Q: 用 AI 写的代码出了 bug，谁负责？

你负责。AI 是工具，你是使用工具的人。你用 Stack Overflow 复制的代码出了 bug 也是你负责，AI 一样。所以永远要 review AI 生成的代码，理解它在做什么，再提交。"AI 写的"不是甩锅的理由。

Q: 公司不允许把代码粘贴到 ChatGPT，怎么办？

很多公司确实有这个安全策略。几个替代方案：

  • GitHub Copilot Business — 代码不会被用于训练，大部分公司的安全团队能接受
  • Cursor Pro — 有 Privacy Mode，代码不离开本地
  • Self-hosted LLM — 公司可以部署 Ollama + Code Llama，完全本地运行
  • 脱敏后使用 — 把公司特定的变量名、API key、业务逻辑替换成通用名称后再问 AI

Q: AI 辅助会不会让我技术退步？

不会，前提是你认真 review AI 的输出。开车用导航不会让你丧失方向感，前提是你偶尔看看路。同理，用 AI 写代码不会让你退步，前提是你理解每一行代码在做什么。真正危险的是不 review 就直接 commit — 那你确实什么都学不到。

小结

AI 辅助日常工作的核心不是"用不用"的问题，是"怎么用得好"的问题。四个高频场景 — 编码、文档、Code Review、会议准备 — 每个都能节省 60-80% 的时间。关键是把 AI 融入日常工作流，而不是当成一个偶尔用一下的工具。建立自己的 Prompt 库，记录效率数据，这些就是你在简历上写"AI-enhanced productivity"的真实证据。
