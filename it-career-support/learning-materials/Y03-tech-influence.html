学习目标

Senior 和 Staff 的核心差异不是"技术更强"，是"影响范围更大"。Senior 影响一个团队，Staff 影响一个 org 甚至整个行业。

影响力不是天生的，是经营出来的。读完这节你应该：

  • 理解 3 条建立影响力的主流路径：博客、演讲、开源
  • 有一份 2 年可执行计划，按月分解
  • 知道澳洲本地哪些真实的人是怎么从 Senior 走到"被行业认识"的

核心概念

为什么影响力是 Senior 到 Staff 的必过关

你是 Senior 的时候，价值 = 你自己能写出的代码 + 你对团队的贡献。

你要升 Staff 的时候，价值 = 你通过别人能产生的影响 + 你的决策影响多少人。

公司给 Staff 岗位的 headcount 有限，怎么决定给谁？靠"影响力证据"。没有证据的人永远排在有证据的人后面。

影响力的 3 种主流形态

形态门槛见效周期适合人群
技术博客低6-12 个月写作能力 OK，喜欢总结
会议演讲中12-18 个月不怕上台，有独特经验
开源贡献高18-24 个月代码能力强，能持续投入

不一定三个都要做，但至少要选一条深耕。最强的人三条都做——比如 Dan Abramov（React core team）博客 + 演讲 + OSS 三位一体。

操作步骤

路径 1：技术博客 — 月更一篇的两年计划

为什么博客是最容易开始的：门槛低、时间自主、复利高。一篇文章写好了，未来 3 年每个月都会给你带来曝光。

平台选择：

  • Dev.to：免费、有内置流量、适合起步。缺点：平台绑定。
  • Hashnode：可以用自己的域名，文章仍然在 Hashnode 的 feed 里分发。
  • 自建（Next.js + Vercel）：完全自主，SEO 长期收益最高。缺点：零启动流量，需要自己推广。
  • Substack：偏 newsletter，适合非技术向的"技术思考"文章。

建议新手先在 Dev.to 发 10 篇看反响，然后搬到自建。

选题三大来源：

  1. 你最近踩的坑：花了 2 天调的一个 bug，写成"How I debugged X"。这种文章流量最高，因为别人在 Google 搜同样的错误。
  2. 你做的技术决策：为什么我们从 MongoDB 换到 PostgreSQL、为什么我们不上 K8s——trade-off 类文章专业读者最喜欢。
  3. 你的工具链：My 2026 developer setup, How I use Cursor daily——个人化内容，真实感强。

两年月更计划：

# Year 1: 建立节奏
Month 1-3: 每月 1 篇，不管好坏，先发
Month 4-6: 保持月更，开始分析哪些文章反响好
Month 7-9: 针对反响好的主题写系列文章
Month 10-12: 年终总结一篇，盘点这一年学到的

# Year 2: 放大影响
Month 13-15: 开始和其他博客作者互动（引用、评论）
Month 16-18: 挑 1-2 篇最好的文章做深度版本（3000+ 字）
Month 19-21: 尝试投 InfoQ、The New Stack 这类技术媒体
Month 22-24: 你的文章被搜索到的几率应该翻 10 倍

澳洲真实例子：Sydney 的一位 Senior Frontend 叫 Mark（化名），2024 年开始在 Dev.to 月更，主题是"我在 Canva 学到的前端工程化经验"。两年后他的文章总阅读量过百万，被 Canva 破格提到 Staff——Staff 晋升评审会上他的博客被引用了 7 次。

路径 2：会议演讲 — 从内部 Tech Talk 到国际大会

为什么演讲比博客影响力更强：一次 30 分钟的演讲 = 几百人现场 + 几千人 YouTube + 几十个 LinkedIn 分享。效率比文章高。

四级进阶路径：

  1. Level 1：内部 Tech Talk（Month 1-3）
    在自己公司做一次 30 分钟的分享。主题选你最近 own 的一个项目的经验。目标：熟悉 slide 制作、克服紧张。
  2. Level 2：本地 Meetup（Month 4-9）
    去 Sydney JavaScript Meetup、Melbourne Python Meetup、AWS User Group 这种地方主动提交演讲。这些 meetup 通常求讲师。你可以把你的内部 talk 稍微脱敏后直接用。
  3. Level 3：本地会议（Month 10-18）
    投递 DDD Sydney、NDC Sydney、PyCon AU、LinuxConf AU 这类本地会议。开始做 CFP（Call for Papers）。通过率大约 20%——被拒 4 次才中一次是正常的。
  4. Level 4：国际会议（Month 19+）
    KubeCon、React Conf、Google I/O 这类国际大会。门槛很高，通常需要你在前几级已经建立了 track record。

CFP 模板（怎么写一份会被接受的演讲提案）：

# Talk Title: 尽量具体，避免 "The Future of X" 这种空洞标题
例子好: "How We Reduced Our K8s Bill by 60% Without Downtime"
例子差: "Kubernetes Cost Optimization Strategies"

# Abstract (150 词)
一段话，包含：
- 问题是什么（hook）
- 你做了什么（unique angle）
- 听众会学到什么（3 个 takeaways）

# Detailed Outline
- 00:00-05:00 问题背景
- 05:00-15:00 我们尝试过的方案和踩的坑
- 15:00-25:00 最终方案的细节
- 25:00-30:00 Q&A

# Speaker Bio (50 词)
简短介绍 + 你之前讲过什么 + 你的专业经验

# Why this talk matters
一段话解释为什么听众会关心

关键建议：投 CFP 要投"你踩过真实坑 + 有数据的主题"。会议评委最讨厌两种：(1) 产品推销式（其实是卖你们公司的 SaaS）；(2) 理论综述式（没有自己的 hands-on 经验）。

路径 3：开源贡献 — 从 good first issue 到 maintainer

为什么开源影响力最硬核：代码是真实可验证的，不像博客可以吹。Staff 晋升 committee 看到"是 X 项目的 core contributor"几乎是直接通过。

四级进阶路径：

  1. Level 1：Good First Issue（Month 1-3）
    选一个你熟悉的框架（React、Vue、NestJS、FastAPI 之类）的 github，筛 label:"good first issue"。目标：合并 3-5 个 PR，熟悉流程。
  2. Level 2：实质性贡献（Month 4-12）
    开始接 feature request 和 bug fix。不要只修 typo，修真正的问题。在 issues 里主动讨论方案，而不是等被分配。目标：合并 10+ 个有意义的 PR。
  3. Level 3：Core Contributor（Month 13-24）
    被邀请成为 committer 或 triager，有 merge 权限。这个 level 通常需要你持续活跃 12-18 个月，项目维护者开始信任你。
  4. Level 4：Maintainer（Month 24+）
    自己成为一个项目的 maintainer——要么是现有项目，要么是你自己发起的项目。发起的项目需要真实的 traction（不是 star-hunting），1000+ stars 是一个合理门槛。

选项目的建议：

  • 选你日常工作用的工具——你最懂，贡献质量最高
  • 选正在活跃开发的项目（过去 30 天有 merge 记录），不要选僵尸项目
  • 选 maintainer 友好的项目——看他们对新 PR 的响应速度和态度
  • 避开 React、Vue 这种超大型项目当起点——竞争太激烈。选中型项目更容易被看见

综合策略：2 年路线图

最有效的是三条路径组合——博客做"持续曝光"，演讲做"峰值冲击"，开源做"硬证据"。2 年路线图：

# Year 1
- 每月 1 篇博客
- 内部做 2 次 Tech Talk
- 合并 3-5 个开源 PR

# Year 2
- 每月 1 篇博客（质量升级）
- 参加 2 次本地 Meetup 演讲
- 1 次本地会议演讲
- 成为 1 个开源项目的 core contributor
- LinkedIn Follower 从 500 涨到 3000+

动手练习

  1. 从 3 条路径里选 1 条作为你的主路径，1 条作为辅助。不要三条同时上。
  2. 为主路径写一份未来 6 个月的月度计划。每个月的交付物要具体——"写博客"不算具体，"月末前发布《如何用 Postgres JSONB 做简历搜索》"才算。
  3. 这周就开始：博客路径的人，发布第一篇文章；演讲路径的人，报名下一次内部 Tech Talk；开源路径的人，找到第一个 good first issue 并开 PR。

验证：6 个月后，去 Google 搜你的名字。如果搜不到你在这个月新产生的内容，说明你的影响力为 0。

常见问题

Q: 我英语不好，能写技术博客吗？
能。用 Claude / Grammarly 做润色，你负责写初稿。两年后你的英语写作能力会大幅提升——写博客是最好的英语练习。

Q: 我怕公司不让我写博客 / 演讲公司的内部经验？
先和 manager 对齐。大多数公司鼓励员工做 public speaking（这是公司 EVP 的加分项）。规则通常是：不讲未发布的技术、不讲具体客户数据、不说竞争对手坏话——这些都好遵守。

Q: 我没有"独特经验"可以写 / 讲？
每个人都有。你每天遇到的坑、你做的每个决策、你用的每个工具——对另一个人来说都是新信息。"独特"不是"前无古人"，是"从你的视角出发"。

Q: 开源贡献和公司工作怎么平衡时间？
两种策略：(1) 找一个公司工作用到的开源项目贡献——老板会支持；(2) 每周末留 3-4 小时固定投入。第二种需要自律。

Q: 我怕第一次 meetup 演讲搞砸？
一定会搞砸。第一次紧张、slide 做得丑、问答答不上来——都是正常的。关键是做完之后复盘："如果再来一次我会改什么"。第五次的时候你就游刃有余了。

小结

Staff 不是自动升的，是自己挣来的。挣的方式就是这三条——博客、演讲、开源。选一条主路径，坚持 2 年，你的职业轨迹会被重写。

最重要的心态：不要等"准备好了再开始"，要先开始，在做的过程中准备。第一篇博客写得烂没关系，第一次演讲紧张没关系，第一个 PR 被拒没关系。这些都是必经过程。真正的失败是你从来没开始。

工具推荐：Hashnode / Substack 做博客，Notion 做演讲 slide，GitHub + git 工作流做开源。
