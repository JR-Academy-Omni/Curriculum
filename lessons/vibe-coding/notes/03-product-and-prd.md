# 03 · Digital Product 拆解 & PRD 写法

## 3.1 Digital Product · 本质就是一堆 Pages

- 一句话：**任何 Web 产品 = 一组页面 + 页面间跳转**
- 不要把"产品"想得太玄——它就是浏览器里的一堆 URL
- 拆产品的第一刀：**列出所有页面**
- 例子：电商 = 首页 + 商品列表 + 商品详情 + 购物车 + 下单 + 订单列表 + 个人中心
- 这一刀拆下去，AI 就知道要给你生成几个 Page 文件

> 讲师提示：让学员现场拆"小红书 / 钉钉 / Lark"——3 分钟能列出 8-10 个页面就算合格。

---

## 3.2 Pages CRUD · 5 种页面套路

- 任何业务系统的页面都逃不出这 5 种：
  - **List Page** — 列表（看全部）
  - **Detail Page** — 详情（看一个）
  - **Create Page** — 新建（加一个）
  - **Edit Page** — 编辑（改一个）
  - **Delete Action** — 删除（去掉一个）
- 这就是经典的 **CRUD（Create / Read / Update / Delete）**
- 90% 的内部系统 / Admin / CRM 都是 CRUD 的排列组合

> 讲师提示：告诉 AI "帮我做一个客户的 CRUD" → 它能秒出 5 个页面。这就是为什么 PRD 越结构化，AI 输出越精准。

---

## 3.3 Digital Product · Components

- 页面拆完 → 拆**组件**（Component）
- 常见组件：Header / Sidebar / Card / Table / Form / Modal / Dropdown / Tag / Avatar
- **组件 = 可复用的 UI 砖块**
- AI 喜欢组件化——告诉它"用 Card 渲染列表"比"做一个好看的列表"精准 10 倍
- 现代框架（React / Vue）的本质就是组件树

> 讲师提示：现场打开任意一个网站，按 F12，让学员指认哪些是 Card / Modal / Sidebar，建立"组件视角"。

---

## 3.4 Digital Product · Flows

- 光有页面不够，还要有**流转（Flow）**
- Flow = 用户从 Page A 到 Page B 的路径 + 触发条件
- 典型 Flow：
  - **登录 Flow**：未登录 → Login Page → 输入 → 验证 → 跳首页
  - **下单 Flow**：商品详情 → 加购 → 购物车 → 结算 → 支付 → 订单成功
  - **审批 Flow**：提交 → 待审批 → 审批通过 / 驳回 → 通知
- 给 AI 写 PRD 时**必须画 Flow**，否则它只会做静态页面，不会做"会动的产品"

> 讲师提示：流程图工具推荐 Mermaid（Markdown 直接写）——AI 看得懂、人也看得懂。

---

## 3.5 PRD · 写 PRD 的 8 步标准动作

- PRD = **Product Spec** = 给 AI 的"任务说明书"
- 8 步流程（按顺序做，不要跳）：
  1. **BA 收集需求** — 跟用户 / Stakeholder 谈
  2. **拆解功能** — 把需求切成可独立交付的 feature
  3. **出 JSON / 结构化** — 用结构化格式写下来（不是大段文字）
  4. **画 Wireframe** — 草图也行，关键是画出来
  5. **列 User Story** — "As a X, I want Y, so that Z"
  6. **优先级** — Must-have / Nice-to-have
  7. **评审** — 跟团队 / Stakeholder 确认
  8. **AI Prompt + Markdown 生成** — 把 PRD 喂给 AI，生成代码
- 关键认知：**前 7 步是人做，第 8 步是 AI 做**

> 讲师提示：很多人想跳到第 8 步——结果 AI 给的是垃圾。原因是前 7 步没做扎实，AI 只能瞎猜。

## 3.6 PRD AI Feature · 完整 Markdown 模板

- 这就是给 AI 的"标准饭票"——按这个格式写，AI 输出质量直接翻倍
- 完整模板（可直接复制使用）：

```markdown
# Product Requirements Document (PRD)
## 1. Project Overview
## 2. Background & Objectives
### 2.1 Background
Describe the context or motivation for this feature.
> Example: Users reported it's time-consuming to read chat logs after meetings. An automatic summarization feature is needed.
### 2.2 Objective
List measurable outcomes.
- Reduce chat review time from 5 mins to under 30 seconds.
- Increase feature engagement by 20%
## 3. Feature Overview
### 3.1 Description
> Users upload chat logs and receive a concise summary generated by AI.
### 3.2 User Stories
- As a **student**, I want to summarize lecture notes so that I can review faster.
- As an **operations assistant**, I want to extract key points from group chats to write reports more easily.
## 4. Functional Requirements
### 4.1 Inputs
- Accepted formats: `.txt`, `.json`
- Max file size: 5MB
- Encoding: UTF-8
### 4.2 Outputs
- Output: Chinese text summary
- Max length: 300 words
- Display: In-browser with a Copy button
### 4.3 User Flow
1. User uploads file
2. Backend processes content
3. AI generates summary
4. Summary is shown to the user
---
## 5. UI & Interaction Design
### 5.1 Pages & Components
| Page | Description |
|------|-------------|
| Upload Page | Upload form, validation |
| Result Page | Summary output, copy button |
### 5.2 Wireframes
_Add Figma or image links here._
```

> 讲师提示：发这份模板给学员，让他们现场套自己的 idea 填一遍——填不出来的字段，就是产品没想清楚的地方。

---

## 3.7 PRD · Context + Roles + Must-have / Nice-to-have

- 模板之外还要补 3 个关键字段：
  - **Context** — 背景信息（用户是谁 / 在什么场景用 / 解决什么痛点）
  - **Roles** — 涉及哪些角色（Student / Teacher / Admin / Guest 等）
  - **Must-have / Nice-to-have** — 优先级分层
- **Must-have** = 没有就不能上线
- **Nice-to-have** = 锦上添花，时间够再做
- AI 会按优先级**先实现 Must-have**，避免一上来就做花哨功能

> 讲师提示：教学员一句话——"AI 不会替你做决定，但 AI 会忠实执行你写下来的优先级"。

## 3.8 Lab · Web App Mini CRM

- **实战 Lab**：用 AI 做一个 Mini CRM
- 范围（CRUD 三件套）：
  - **客户**（Customer） — 列表 + 新建 + 编辑 + 删除
  - **订单**（Order） — 关联客户 + CRUD
  - **跟进记录**（Note） — 关联客户 + 时间线
- 流程：
  1. 写 PRD（套 3.6 的模板）
  2. 拆 Pages（套 3.2 的 5 种 CRUD）
  3. 画 Flow（客户 → 详情 → 新建订单）
  4. 喂给 Vibe Coding 工具（Lovable / Bolt / v0 / Cursor）
  5. 拿到链接，演示给同桌看
- 时长：**90 分钟**（从 0 到能跑）

> 讲师提示：这是本课第一个完整闭环——PRD → AI → 可访问的 Web App。学员第一次"用嘴做产品"，体验感拉满。

## 3.9 RAG · 后续课预告

- **RAG（Retrieval-Augmented Generation）** = 检索增强生成
- 一句话：**让 AI 先查资料再回答**，不是凭记忆瞎编
- 用途：企业知识库问答、客服 bot、文档助手、AI 搜索
- 本节只做概念铺垫，**完整实战在后续 AI Engineer 章节展开**

> 讲师提示：留个钩子——"今天你学会了用 AI 做产品，下一阶段你将学会让产品自己'懂资料'"。
