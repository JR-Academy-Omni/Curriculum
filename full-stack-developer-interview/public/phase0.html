澳洲 IT 求职全流程

找工作不是投简历然后等电话。澳洲 IT 求职是一条有节奏的 pipeline，每个环节都有对应的准备方式。

求职渠道（按效果排名）

渠道命中率适合谁
内推 (Referral)最高（50%+ 面试率）有行业人脉的求职者
LinkedIn 直投中等Profile 优化好的求职者
Seek / Indeed中低（竞争激烈）广撒网阶段
公司官网 Careers中等有明确目标公司的求职者

面试全流程

  1. 简历筛选 (ATS + HR) — 关键词匹配，格式规范，1 页以内
  2. Phone Screen (15-30min) — HR 了解背景、签证状态、薪资期望、可开始时间
  3. Technical Assessment — 可能是 Online Test（HackerRank/Codility）或 Take-home Assignment（1-3 天完成小项目）
  4. Technical Interview (1-2 轮，各 45-60min) — 面对面技术面，前端/后端/全栈高频题 + 项目深挖
  5. System Design (45-60min) — Mid 级别以上必考，Junior 可能简化为架构思维讨论
  6. Behavioral / Culture Fit (30-45min) — STAR 格式回答，考察团队合作和价值观
  7. Reference Check + Offer — 提供 2-3 个推荐人，谈薪资 package

SOLID原则

Video · 60min · Stan Luo
面向对象设计五大原则：单一职责、开闭、里氏替换、接口隔离、依赖反转

SOLID 原则详解

SOLID 是面向对象设计的五大原则，面试中经常作为开场题考察你的工程素养。面试官不是要你背定义，而是看你能不能用真实例子讲清楚。

S — 单一职责原则 (Single Responsibility Principle)

一个类/模块只做一件事，只有一个理由需要修改它。

// ❌ 违反 SRP — UserService 又管用户又管邮件
class UserService {
  createUser(data) { /* 创建用户 */ }
  sendWelcomeEmail(user) { /* 发邮件 */ }
  generateReport(users) { /* 生成报告 */ }
}

// ✅ 遵守 SRP — 职责分离
class UserService { createUser(data) { /* 只管用户 CRUD */ } }
class EmailService { sendWelcomeEmail(user) { /* 只管邮件 */ } }
class ReportService { generateReport(users) { /* 只管报告 */ } }

O — 开闭原则 (Open/Closed Principle)

对扩展开放，对修改关闭。新增功能时不应该修改已有代码。

// ❌ 每次加新类型都要改 if-else
function calculateArea(shape) {
  if (shape.type === 'circle') return Math.PI * shape.radius ** 2;
  if (shape.type === 'rectangle') return shape.width * shape.height;
  // 加三角形？改这个函数...
}

// ✅ 用多态，新增类型不改已有代码
class Circle { area() { return Math.PI * this.radius ** 2; } }
class Rectangle { area() { return this.width * this.height; } }
class Triangle { area() { return 0.5 * this.base * this.height; } }

L — 里氏替换原则 (Liskov Substitution Principle)

子类必须能替换父类使用，不改变程序的正确性。经典反例：正方形不应该继承矩形（设置宽度时正方形会强制修改高度，破坏矩形的行为契约）。

I — 接口隔离原则 (Interface Segregation Principle)

不要让客户端依赖它不需要的接口。把大接口拆成小接口。

// ❌ 一个大接口
interface Worker {
  work(): void;
  eat(): void;  // 机器人不需要 eat
}

// ✅ 拆分接口
interface Workable { work(): void; }
interface Eatable { eat(): void; }
class Human implements Workable, Eatable { ... }
class Robot implements Workable { ... }

D — 依赖反转原则 (Dependency Inversion Principle)

高层模块不应该依赖低层模块，两者都应该依赖抽象。

// ❌ 直接依赖具体实现
class OrderService {
  private db = new MySQLDatabase(); // 紧耦合
}

// ✅ 依赖抽象接口
class OrderService {
  constructor(private db: Database) {} // 注入接口
}

面试怎么答 SOLID？

不要一条一条背定义。面试官问「说说你对 SOLID 的理解」，用这个结构回答：

  1. 一句话概括这个原则解决什么问题
  2. 举一个你项目中的真实例子（或者常见反例）
  3. 说明遵守这个原则的好处（可维护性、可测试性、可扩展性）

白板题 + 项目题

Video · 60min · Stan Luo
白板题和项目题的区别、解题流程、代码规范

白板题 vs 项目题

技术面试主要考两种代码题，准备方式完全不同。

白板题 (Whiteboard / Algorithm)

特点说明
时间15-30 分钟一道
形式面试官出题，你在白板/共享编辑器上写
考察算法思维、数据结构、代码能力、沟通
难度LeetCode Easy-Medium 为主
代表公司Canva、Rokt、Atlassian

项目题 (Take-home Assignment / Code Test)

特点说明
时间2-4 小时（有的给 1-3 天）
形式实现一个小项目或功能模块
考察代码质量、架构思维、测试、文档
评分代码规范、边界处理、测试覆盖、Git 提交历史
代表公司Employment Hero、SafetyCulture、Zendesk

白板题解题流程（每道题必须用）

  1. 读题 + 澄清 (1-2min) — 确认输入/输出类型、边界条件、约束。「Can I assume the input is always valid?」「Is the array sorted?」
  2. 说暴力解 (1min) — 先给出最简单的解法和复杂度。「The brute force approach would be O(n²) with nested loops.」
  3. 优化 (2-3min) — 讲清楚优化思路。「We can use a hash map to reduce time complexity to O(n).」
  4. 编码 (10-15min) — 边写边解释你在做什么。变量命名清晰，处理边界
  5. 测试 (2-3min) — 用示例数据 walk through 一遍代码，检查边界情况

项目题评分标准

  1. 功能完整性 (40%) — 需求是否全部实现
  2. 代码质量 (25%) — 命名规范、函数拆分、DRY 原则
  3. 测试 (15%) — 是否有单元测试和集成测试
  4. 文档 (10%) — README 是否清晰（安装、运行、设计决策）
  5. Git 提交 (10%) — 是否有合理的 commit 历史（不要一个大 commit）

代码规范检查清单

  • 变量命名有意义（userList 而不是 data）
  • 函数不超过 20 行
  • 没有 magic number（用常量代替）
  • 错误处理完善（try-catch、空值检查）
  • 有 TypeScript 类型定义
  • ESLint / Prettier 配置

Agile的由来和产品生命周期

Agile 的由来和产品生命周期

为什么需要 Agile？

传统瀑布式开发的问题：需求在项目开始时全部确定，但软件开发的特点是需求会变。两个经典案例：

  • 悉尼歌剧院 — 预算 700 万澳元，最终花了 1.02 亿，工期从 4 年变成 16 年。原因：设计反复变更，但用的是传统项目管理方式
  • 悉尼轻轨 — 类似问题，需求变更导致工期延长、成本飙升

Software Development Life Cycle (SDLC)

SDLC 是软件从概念到退役的完整生命周期：

  1. Planning — 确定项目范围、目标、资源、时间线
  2. Analysis — 需求收集和分析（和 stakeholder 对齐）
  3. Design — 系统架构设计、数据模型、UI/UX
  4. Implementation — 编码开发
  5. Testing — 单元测试、集成测试、用户验收测试 (UAT)
  6. Deployment — 上线部署
  7. Maintenance — 持续维护、bug 修复、功能迭代

Agile 宣言（2001年，17位软件开发者）

四个核心价值观：

  • 个体和互动 高于 流程和工具
  • 工作的软件 高于 详尽的文档
  • 客户合作 高于 合同谈判
  • 响应变化 高于 遵循计划

注意：右边的也有价值，但左边的更重要。

面试怎么答「什么是 Agile？」

不要背宣言。用 30 秒说清楚：Agile 是一种迭代式的软件开发方法论。核心思想是把大项目拆成小的迭代周期（Sprint），每个周期交付可工作的软件，根据反馈持续调整。相比瀑布式开发，Agile 更适合需求不确定或变化频繁的项目。

传统开发流程——瀑布式开发&V-Model

Video · 60min · Xiao Hua
Waterfall模型、V Model。项目管理具体步骤。软件开发和非软件产品开发的区别

传统开发流程：瀑布式开发 & V-Model

瀑布式开发 (Waterfall Model)

最传统的 SDLC 模型，像瀑布一样从上往下流，每个阶段完成后才能进入下一个：

需求分析 → 系统设计 → 实现编码 → 测试 → 部署 → 维护
  ↓          ↓         ↓       ↓      ↓      ↓
 不可回退    不可回退   不可回退  不可回退  不可回退

优点：流程清晰、文档完善、适合需求明确且不变的项目（如嵌入式系统、航空航天）

缺点：不灵活、发现问题太晚、不适合需求会变的项目

V-Model

瀑布模型的改进版，每个开发阶段对应一个测试阶段：

需求分析  ←————→  验收测试 (UAT)
  系统设计  ←——→  系统测试
    概要设计 ←→ 集成测试
      编码 ←→ 单元测试

优点：测试前置（每个阶段都有对应测试），质量更有保障

缺点：仍然是线性的，不支持迭代

软件开发 vs 非软件产品开发

软件开发传统产品（如建筑）
需求变化频繁变化一旦确定很少变
修改成本相对低（改代码）极高（拆了重建）
交付方式可以增量交付必须完整交付
适合方法Agile（迭代、增量）Waterfall（线性、计划驱动）

为什么瀑布式不适合大多数软件项目？

  1. 用户在看到产品前不知道自己要什么（需求不确定）
  2. 技术方案在实现过程中可能需要调整
  3. 市场和竞争环境在开发过程中会变化
  4. 等到测试阶段才发现问题，修复成本已经很高

面试考点

面试官问「Waterfall 和 Agile 的区别」时，不要只说优缺点，要说清楚什么场景用什么：需求明确不变的合规性项目用 Waterfall，需求会变的产品型项目用 Agile。

Agile理念、价值与原则

Video · 60min · Xiao Hua

Agile 理念、价值与原则

4 个核心价值观（深入理解）

1. 个体和互动 > 流程和工具

工具是辅助，人才是核心。遇到问题先走过去和同事聊 5 分钟，比在 Jira 上写半小时 comment 有效得多。但这不是说不需要工具，而是工具应该服务于人的协作，不是反过来。

2. 工作的软件 > 详尽的文档

能跑的代码比 100 页 PRD 更有说服力。MVP 先上线，根据用户反馈迭代。文档要写，但够用就行——README 能让新人跑起项目、API 文档能让前端对接，这就够了。

3. 客户合作 > 合同谈判

持续和用户/PM 对齐需求，而不是按一份固定合同交活。每个 Sprint 的 Review 就是让 stakeholder 看到进展并给反馈的机会。

4. 响应变化 > 遵循计划

计划赶不上变化。Sprint 中发现方向错了，及时调整比坚持原计划更好。Agile 团队拥抱变化，但通过 Sprint 边界来管理变化的节奏。

12 条原则精选（面试最常问的 5 条）

  1. 最高优先级是通过早期和持续交付有价值的软件来满足客户 — 每个 Sprint 交付可工作的增量
  2. 欢迎需求变化，即使在开发后期 — 变化是竞争优势的来源，不是麻烦
  3. 频繁交付可工作的软件 — 从几周到几个月，越短越好
  4. 业务人员和开发人员必须每天一起工作 — Daily Standup 就是这个原则的实践
  5. 最好的架构、需求和设计出自自组织的团队 — 团队自己决定怎么做，不是被上面安排

Agile SDLC Model

Sprint 1: 需求→设计→开发→测试→发布（2周）
    ↓ 反馈
Sprint 2: 需求→设计→开发→测试→发布（2周）
    ↓ 反馈
Sprint 3: ...（持续迭代）

每个 Sprint 都是一个完整的小瀑布，但周期短（通常 2 周），每次迭代都能根据反馈调整方向。

Scrum & Kanban

Scrum 是什么？

Scrum 是 Agile 最流行的实施框架。核心思想：把工作拆成固定长度的 Sprint（通常 2 周），每个 Sprint 交付一个可工作的产品增量。

Scrum Sprint 流程

事件时长参与者做什么
Sprint Planning2-4h全团队PO 讲优先级，团队选 story，估点，定 Sprint Goal
Daily Scrum (Standup)15min开发团队三个问题：昨天做了啥、今天做啥、有没有 blocker
Sprint Review (Demo)1-2h全团队 + stakeholder演示这个 Sprint 完成的功能，收集反馈
Sprint Retrospective1-1.5h全团队复盘：什么做得好、什么要改进、下个 Sprint 尝试什么

Scrum 团队组成

  • Product Owner (PO) — 决定做什么（管理 Product Backlog，确定优先级）
  • Scrum Master (SM) — 保证 Scrum 流程顺利运行，排除障碍（不是项目经理！）
  • Development Team — 3-9 人的跨职能团队，自己决定怎么做

Kanban 是什么？

Kanban 是另一种 Agile 方法，核心是可视化工作流 + 限制在制品 (WIP Limit)。

Kanban 看板阶段

To Do  →  In Progress  →  Code Review  →  Testing  →  Done
         (WIP: 3)        (WIP: 2)       (WIP: 2)

WIP Limit 的意义：限制每个阶段同时进行的工作数量，避免多任务切换导致效率降低。

Scrum vs Kanban

ScrumKanban
节奏固定 Sprint（2周）持续流动，无固定周期
角色PO, SM, Dev Team无固定角色
变更Sprint 内不加新需求随时可以加
适用产品开发运维、支持、持续交付

面试考点

「你们团队用 Scrum 还是 Kanban？」— 说清楚用的是什么，Sprint 多长，Daily Standup 怎么开，Retro 中发现过什么问题、怎么改进的。

Scrum

Video · 60min · Xiao Hua
Scrum框架深入讲解

Scrum 深入讲解

Sprint 的核心规则

  • Sprint 长度固定（通常 2 周），不能延长也不能缩短
  • Sprint 有明确的 Goal（不是一堆散的 task，而是一个可验收的目标）
  • Sprint 内不应该加新的需求（除非 PO 和团队协商，且替换掉等量的 story）
  • Sprint 结束时，未完成的 story 回到 Product Backlog，不算完成

User Story 怎么写

As a [用户角色],
I want to [做什么],
So that [为什么/价值].

例子：
As a job seeker,
I want to filter jobs by salary range,
So that I can focus on positions that match my expectations.

Story Points 估算

Story Points 估的是复杂度，不是时间。常用斐波那契数列：1, 2, 3, 5, 8, 13, 21。

  • 1 point — 改个文案、加个字段
  • 3 points — 实现一个标准 CRUD 功能
  • 5 points — 需要和其他模块对接的功能
  • 8 points — 涉及第三方集成或复杂业务逻辑
  • 13+ points — 太大了，应该拆分

Planning Poker

估算方式：每个人同时出牌（避免锚定效应），如果差异大则讨论后重新估。

Velocity

团队每个 Sprint 完成的 Story Points 总和。用过去 3-5 个 Sprint 的平均 Velocity 来预测未来的交付能力。

Definition of Done (DoD)

团队统一的「完成」标准，例如：

  • 代码已合并到 main 分支
  • 单元测试通过
  • Code Review 完成
  • 部署到 staging 环境
  • PO 验收通过

Scrum的构成

Video · 60min · Xiao Hua
Scrum角色分工。Scrum Artifacts（Product Backlog、Sprint Backlog、Increment）。Scrum Events。Scrum value propositions

Scrum 的构成

三个角色 (Scrum Roles)

角色职责不是什么
Product Owner管理 Product Backlog，确定优先级，代表用户和业务不是项目经理，不指挥团队怎么做
Scrum Master保证 Scrum 流程运行，排除障碍，促进团队自组织不是团队 leader，不分配任务
Development Team自组织，跨职能，自己决定怎么完成 Sprint Goal不只是程序员，包括 QA、设计等

三个产物 (Scrum Artifacts)

1. Product Backlog

所有待做工作的有序列表，由 PO 维护和排序。包含 User Story、Bug、Tech Debt 等。顶部的 item 优先级最高且足够细化，底部的可以粗略。

2. Sprint Backlog

当前 Sprint 要完成的 item 子集 + 团队为完成它们制定的计划。Sprint Backlog 属于开发团队，只有团队可以修改。

3. Increment

每个 Sprint 结束时交付的可工作的产品增量。Increment 必须满足 Definition of Done，且是可部署的状态。

五个事件 (Scrum Events)

  1. Sprint — 固定长度的迭代周期（2 周）
  2. Sprint Planning — 规划这个 Sprint 做什么、怎么做
  3. Daily Scrum — 每天 15 分钟同步进展和障碍
  4. Sprint Review — 展示成果，收集反馈
  5. Sprint Retrospective — 团队复盘，持续改进

Scrum Values

  • Commitment — 对 Sprint Goal 的承诺
  • Courage — 勇于说出问题和挑战
  • Focus — 专注于 Sprint Goal
  • Openness — 对工作和挑战保持透明
  • Respect — 尊重团队成员的能力和判断

面试常见追问

  • 「PO 和 PM 有什么区别？」— PO 关注产品价值和优先级，PM 关注项目交付和资源协调。Scrum 里没有 PM 角色
  • 「Scrum Master 和 Tech Lead 有什么区别？」— SM 关注流程和团队健康，Tech Lead 关注技术方向和代码质量
  • 「Sprint 做不完怎么办？」— 未完成的 story 回到 Product Backlog，下个 Sprint 重新评估优先级。不延长 Sprint

常见算法题解析

解题流程（面试标准流程）

  1. 理解题意 — 读题后复述一遍，确认理解正确。问清楚：输入范围？有无重复？需要排序吗？
  2. 举例子 — 用 2-3 个简单例子验证理解
  3. 暴力解 — 先说最简单的解法，分析时间和空间复杂度
  4. 优化 — 思考能否用更好的数据结构或算法降低复杂度
  5. 编码 — 写代码，边写边解释
  6. 测试 — 用例子 dry run，检查边界（空数组、单元素、极大值）

复杂度分析基础

时间复杂度

复杂度含义例子
O(1)常数时间数组取下标、哈希表查找
O(log n)对数时间二分搜索
O(n)线性时间遍历数组
O(n log n)线性对数排序（Merge Sort / Quick Sort）
O(n²)平方时间嵌套循环（冒泡排序）
O(2ⁿ)指数时间穷举所有子集

空间复杂度

额外使用的内存。如果在原数组上操作（in-place），空间 O(1)；如果创建了新数组或哈希表，空间 O(n)。

常见题型分类

  • 数组 — Two Sum、最大子数组和、合并区间
  • 字符串 — 回文判断、最长无重复子串、字母异位词
  • 链表 — 反转链表、检测环、合并排序链表
  • 树 — BFS/DFS 遍历、最大深度、验证 BST
  • 哈希表 — 频率统计、分组、LRU Cache
  • 双指针 — 有序数组求和、移除元素、盛水最多的容器
  • 滑动窗口 — 最长/最短子串/子数组
  • 动态规划 — 爬楼梯、背包、最长子序列

面试技巧

  • 先说思路再写代码，面试官想看你的思考过程
  • 不确定的地方问面试官，不要自己假设
  • 写完代码后主动 dry run 一遍
  • 如果卡住了，说出你的思路让面试官引导
