API 设计 + Node.js 后端高频题

RESTful API 设计

✅ GET    /api/users          # 获取列表
✅ GET    /api/users/:id      # 获取单个
✅ POST   /api/users          # 创建
✅ PUT    /api/users/:id      # 全量更新
✅ PATCH  /api/users/:id      # 部分更新
✅ DELETE /api/users/:id      # 删除

❌ GET    /api/getUsers       # 动词不应出现在 URL
❌ POST   /api/user/delete/1  # 用 HTTP 方法表达操作

HTTP 状态码（面试必知）

状态码含义常用场景
200成功GET/PUT/PATCH 成功
201已创建POST 创建成功
204无内容DELETE 成功
400请求错误参数验证失败
401未认证没有 token 或 token 过期
403未授权有 token 但没权限
404未找到资源不存在
429请求过多触发限流
500服务器错误未预期的异常

幂等性

同一个请求执行多次和执行一次效果一样。

  • GET — 幂等（读取不改变状态）
  • PUT — 幂等（全量替换）
  • DELETE — 幂等（删除后再删结果一样）
  • POST — 不幂等（每次创建新资源）
  • PATCH — 看实现（age = 25 幂等，age += 1 不幂等）

JWT vs Session

JWTSession
状态无状态（token 自包含）有状态（服务端存储）
扩展天然分布式需共享存储（Redis）
安全无法主动失效可主动销毁
大小较大（包含 payload）小（只有 session ID）

Node.js 核心

单线程 + Event Loop

Node.js 主线程是单线程，但通过 Event Loop + libuv 实现异步 I/O。CPU 密集型用 Worker Threads。生产环境用 Cluster 模块或 PM2 fork 多进程利用多核。

中间件模式

CORS（面试常问）

浏览器同源策略限制跨域请求。CORS 通过服务端设置 Access-Control-Allow-Origin 头允许跨域。预检请求（OPTIONS）用于复杂请求（如 PUT/DELETE 或自定义头）。

索引

B-Tree 索引原理

B-Tree 是平衡多叉搜索树，每个节点存多个键值。查找、插入、删除都是 O(log n)。数据库索引本质是 B-Tree 结构。

复合索引

-- 创建复合索引
CREATE INDEX idx_user_age_city ON users(age, city);

-- ✅ 走索引（最左前缀匹配）
SELECT * FROM users WHERE age = 25;
SELECT * FROM users WHERE age = 25 AND city = 'Sydney';

-- ❌ 不走索引（跳过了 age）
SELECT * FROM users WHERE city = 'Sydney';

为什么加了索引还是慢？

  1. 字段类型不匹配（字符串字段用数字查询）
  2. 复合索引没遵守最左前缀
  3. 查询用了函数（WHERE YEAR(created_at) = 2026）
  4. 数据量小，优化器选择全表扫描
  5. 返回字段太多，回表开销大

事务 ACID

  • Atomicity — 全成功或全回滚
  • Consistency — 从一个一致状态到另一个一致状态
  • Isolation — 并发事务互不影响
  • Durability — 提交后持久化，宕机不丢

隔离级别

级别脏读不可重复读幻读
Read Uncommitted可能可能可能
Read Committed防止可能可能
Repeatable Read防止防止可能
Serializable防止防止防止

乐观锁 vs 悲观锁

  • 悲观锁 — SELECT FOR UPDATE，锁行直到事务结束。适合高竞争
  • 乐观锁 — UPDATE WHERE version = old_version，不加锁。适合低竞争

N+1 问题

// ❌ N+1：1 条查用户 + N 条查订单
const users = await User.find();
for (const u of users) {
  u.orders = await Order.find({ userId: u._id });
}

// ✅ 解决：populate / JOIN / DataLoader
const users = await User.find().populate('orders');

并发 / 缓存 / 微服务

缓存策略

Cache-Aside（最常用）

async function getUser(id) {
  // 1. 先查缓存
  let user = await redis.get('user:' + id);
  if (user) return JSON.parse(user);

  // 2. 缓存没有，查数据库
  user = await db.findById(id);

  // 3. 写入缓存
  await redis.set('user:' + id, JSON.stringify(user), 'EX', 3600);
  return user;
}

问题原因解决方案

并发处理

Race Condition

两个请求同时读取库存=10，各扣 1，最终库存=9（应该是8）。解决方案：

  • 数据库乐观锁：UPDATE stock SET count = count - 1 WHERE count > 0
  • Redis 原子操作：DECR key
  • 分布式锁：Redis + Lua 脚本

限流算法

算法原理特点
令牌桶固定速率生成令牌，请求消耗令牌允许突发流量
漏桶请求排队，固定速率处理流量绝对平滑
滑动窗口统计过去 N 秒请求数比固定窗口更精确

微服务 vs 单体

什么时候拆微服务？

  • 团队超过 10 人，需要独立部署
  • 不同模块有不同的扩展需求
  • 需要使用不同的技术栈

什么时候不该拆？

  • 团队小（<5人），单体更高效
  • 项目早期，业务边界不清晰
  • 没有 DevOps 基础设施

服务间通信

方式同步/异步适用
REST同步简单的 CRUD 调用
gRPC同步高性能、强类型
消息队列异步事件驱动、解耦

数据一致性

微服务间不共享数据库。用 Saga 模式保证分布式事务的最终一致性。
