系统设计方法论

4 步框架（严格控制时间）

Step 1: 需求澄清 (5 min)

面试官给你一个模糊的题目，你要缩小范围到一个可讨论的子问题。

  • 核心功能有哪些？优先级？
  • 用户量级？DAU? QPS?
  • 读写比？读多写少？
  • 可用性 SLA？99.9%?
  • 数据量？增长预期？

Step 2: 高层架构 (10 min)

Client → CDN → Load Balancer → API Gateway
                                    ↓
                            Application Servers
                              ↓           ↓
                          Database    Cache (Redis)
                              ↓
                        Message Queue → Workers

Step 3: 详细设计 (20 min)

  • 数据模型（表结构、索引）
  • API 设计（endpoint、参数、返回值）
  • 关键算法或逻辑

Step 4: 扩展与权衡 (10 min)

  • 10x 流量怎么办？→ 水平扩展、数据库分片、CDN
  • 单点故障？→ 多副本、自动故障转移
  • 一致性 vs 可用性？→ CAP 取舍

核心组件速查表

组件解决什么问题常用产品
Load Balancer分配流量，避免单点Nginx, AWS ALB
CDN静态资源就近访问CloudFront, Cloudflare
Cache减少 DB 压力Redis, Memcached
Message Queue异步解耦RabbitMQ, SQS, Kafka
Search Engine全文搜索Elasticsearch
Object Storage文件/图片存储S3

容量估算速算

  • 1M DAU ≈ 60-120 QPS（平均），峰值 x3-5
  • 1 条消息 ≈ 200 bytes
  • 1 张图片 ≈ 200KB-2MB
  • 1TB = 1,000,000 MB ≈ 5亿条消息

CAP 定理

分布式系统最多同时满足两个：

  • CP（一致性优先）— 银行转账、库存扣减
  • AP（可用性优先）— 社交媒体、时间线

经典案例 1 — Chat App + Notification System

设计 Chat Application

需求澄清

支持 1v1 和群聊？消息类型（文字/图片/视频）？已读状态？离线消息？历史消息搜索？

实时通信方案

方案原理优点缺点
WebSocket全双工持久连接低延迟、双向连接管理复杂
SSE服务端单向推送简单只能服务端→客户端
Long Polling客户端挂起等待兼容性好延迟高、浪费资源

推荐：WebSocket 为主，Long Polling 为降级方案。

消息存储设计

messages: {
  messageId: UUID,
  channelId: string,
  senderId: string,
  content: string,
  type: 'text' | 'image',
  createdAt: timestamp,
  status: 'sent' | 'delivered' | 'read'
}

// 索引
{ channelId: 1, createdAt: -1 }  // 按频道查最新消息

群聊扇出策略

  • 写扩散 — 发消息时写入每个成员的收件箱。适合小群（<100人）
  • 读扩散 — 读取时实时查询。适合大群（1000+人）

设计 Notification System

架构

Event Source → Notification Service → Priority Queue
                                         ↓
                                    Channel Router
                                    ↙    ↓    ↘
                                Push  Email   SMS

关键设计决策

  • 去重 — 同一事件不重复通知（用 event_id + user_id 去重）
  • 优先级 — 安全警告 > 交易通知 > 社交通知 > 营销
  • 降级 — Push 失败降级到 Email，Email 失败降级到 SMS
  • 用户偏好 — 用户可以关闭特定类型的通知

经典案例 2 — E-commerce + Rate Limiter

设计 E-commerce 核心模块

库存扣减（面试最常考）

方案实现优点缺点
悲观锁SELECT FOR UPDATE简单强一致并发低，死锁风险
乐观锁UPDATE WHERE stock = old无锁等待高竞争时大量重试
Redis 预扣减DECR + 异步写 DB性能极高需保持一致性

推荐：Redis 预扣减 + 消息队列异步写 DB。秒杀场景用 Redis DECR 扣库存（毫秒级），成功后异步写入数据库。

订单支付流程

下单 → 创建订单（状态: pending）→ 扣减库存（Redis）
     → 跳转支付 → 支付回调 → 更新订单（状态: paid）
     → 异步：发通知、更新统计、生成发票

超时未支付 → 取消订单 → 恢复库存

设计 Rate Limiter

令牌桶 (Token Bucket)

桶以固定速率 r 生成令牌，最多存 b 个。每个请求消耗 1 个令牌。桶空则拒绝。

特点：允许突发流量（桶中有存量时）。API Gateway 最常用。

漏桶 (Leaky Bucket)

请求进入桶排队，以固定速率流出处理。桶满则丢弃。

特点：流量绝对平滑。适合需要严格控制处理速率的场景。

滑动窗口 (Sliding Window)

统计过去 N 秒内的请求数，超过阈值拒绝。

比固定窗口精确，避免窗口边界突发问题。

分布式限流（Redis 实现）

-- Lua 脚本（原子操作）
local key = KEYS[1]
local limit = tonumber(ARGV[1])
local window = tonumber(ARGV[2])
local current = redis.call('INCR', key)
if current == 1 then
  redis.call('EXPIRE', key, window)
end
if current > limit then
  return 0  -- 拒绝
end
return 1  -- 允许
