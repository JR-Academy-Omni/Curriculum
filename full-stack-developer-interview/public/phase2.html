本节是 Stan 老师直播课的预习/复习材料。直播课中 Stan 老师会带你实际操练面试的每个环节，这份教程帮你在直播前建立知识框架，直播后复习巩固。

面试轮次深度拆解

1. Phone Screen (HR 轮)

很多人不重视 HR 轮，觉得只是走流程。但 HR 轮决定你能不能进技术面。

  • 常问问题：Tell me about yourself / Why are you interested in this role? / What's your visa status? / What's your salary expectation? / When can you start?
  • 答题要点：自我介绍控制在 2 分钟内。薪资给一个范围而不是具体数字。签证问题如实回答

2. Coding Test / Online Assessment

类型时间平台准备方式
在线算法题60-90minHackerRank, CodilityLeetCode Easy-Medium 练习
Take-home4h-3天GitHub/邮件注重代码质量、测试、文档
Live Coding45-60minCoderPad, 共享屏幕练习边写边讲

3. Technical Interview (技术面)

面对面或视频，通常 2 轮，每轮 45-60 分钟。考察内容：

  • 前端：JS 核心机制、React Hooks、性能优化
  • 后端：API 设计、数据库、并发处理
  • 全栈：前后端如何协作、SSR/CSR 选型、部署流程

4. System Design

Junior 可能不考或简化为架构讨论。Mid-Senior 必考。4 步框架：需求澄清(5min) → 高层架构(10min) → 详细设计(20min) → 扩展讨论(10min)。

5. Behavioral / Culture Fit

STAR 格式答题。准备 5 个故事覆盖：团队合作、技术决策、失败复盘、冲突处理、学习成长。

前端高频突击（直播课配套教程）

本节直播课覆盖前端面试最高频的考点。以下是核心知识点的结构化整理。

JavaScript 核心机制

闭包 (Closure)

函数能记住并访问它的词法作用域，即使在作用域外执行。

function createCounter() {
  let count = 0;  // 闭包变量
  return {
    increment: () => ++count,
    getCount: () => count
  };
}
const counter = createCounter();
counter.increment(); // 1
counter.increment(); // 2

Event Loop 执行顺序

console.log('1');                    // 同步
setTimeout(() => console.log('2'), 0); // 宏任务
Promise.resolve().then(() => console.log('3')); // 微任务
console.log('4');                    // 同步
// 输出: 1 → 4 → 3 → 2

规则：同步 → 微任务队列（Promise.then）→ 宏任务队列（setTimeout）

this 绑定

调用方式this 指向
普通函数调用window（严格模式 undefined）
对象方法调用调用的对象
new 构造函数新创建的对象
箭头函数定义时的外层 this
call/apply/bind指定的对象

React 高频

组件通信方式

  • 父→子：Props
  • 子→父：回调函数
  • 跨层级：Context API
  • 复杂状态：Redux / Zustand

useState vs useReducer

简单状态用 useState，复杂状态逻辑（多个相关状态、需要 action 描述）用 useReducer。

标准答题结构

每个前端题都用这个结构回答：

  1. 结论 — 一句话直接回答
  2. 原理 — 为什么是这样
  3. 示例 — 代码或场景举例
  4. 权衡 — 什么时候不适用、替代方案

后端高频突击（直播课配套教程）

API 设计面试题

RESTful 设计原则

GET    /api/users          # 获取用户列表
GET    /api/users/:id      # 获取单个用户
POST   /api/users          # 创建用户
PUT    /api/users/:id      # 全量更新
PATCH  /api/users/:id      # 部分更新
DELETE /api/users/:id      # 删除用户

幂等性（必考）

GET/PUT/DELETE 幂等，POST 不幂等。面试追问：PATCH 幂等吗？看实现（age = 25 幂等，age += 1 不幂等）。

数据库面试题

索引原理

B-Tree 索引是一棵平衡多叉搜索树。查找复杂度 O(log n)。复合索引遵守最左前缀匹配原则。

N+1 问题

// ❌ N+1 问题：查 10 个用户发了 11 条 SQL
const users = await User.find();
for (const user of users) {
  const orders = await Order.find({ userId: user._id }); // N 条额外查询
}

// ✅ 解决方案：用 populate 或 JOIN
const users = await User.find().populate('orders'); // 只有 2 条 SQL

并发与缓存

缓存穿透 / 击穿 / 雪崩

问题原因解决方案
穿透查不存在的数据布隆过滤器 / 缓存空值
击穿热点 key 过期互斥锁 / 永不过期
雪崩大量 key 同时过期过期时间加随机偏移

Redis 数据结构选择

数据结构用途
String缓存、计数器、分布式锁
Hash对象存储（用户信息）
List消息队列、时间线
Set去重、交集（共同好友）
Sorted Set排行榜、延迟队列

系统设计入门（直播课配套教程）

系统设计面试 4 步框架

Step 1: 需求澄清 (5min)

面试官说「设计 Twitter」，你第一反应不是画架构，而是问：

  • 核心功能是什么？发推文 + 时间线 + 关注？
  • 用户量级？1M DAU 还是 100M DAU？
  • 读写比？读多写少（100:1）？
  • 可用性要求？99.9%？

Step 2: 高层架构 (10min)

Client → CDN → Load Balancer → API Gateway
                                    ↓
                            Application Servers
                              ↓           ↓
                          Database    Cache (Redis)
                              ↓
                        Message Queue → Workers

Step 3: 详细设计 (20min)

针对核心功能深入：数据模型、API 设计、关键算法

Step 4: 扩展与权衡 (10min)

10x 流量怎么办？单点故障？CAP 取舍？

核心组件速查

组件作用什么时候用
Load Balancer分配流量到多个服务器几乎所有生产系统
CDN静态资源缓存，就近分发有图片/视频/静态文件
Message Queue异步处理，解耦服务发邮件、通知、数据处理
Cache (Redis)减少数据库压力读多写少的热点数据
API Gateway统一入口，认证、限流微服务架构
Elasticsearch全文搜索商品搜索、日志搜索

容量估算

  • 1 DAU ≈ 5-10 请求/天
  • 1M DAU ≈ 60-120 QPS（平均），峰值 x3-5
  • 1 条推文 ≈ 200 bytes
  • 1 张图片 ≈ 200KB-2MB

CAP 定理

分布式系统最多同时满足两个：Consistency（一致性）、Availability（可用性）、Partition Tolerance（分区容忍）。实际选择通常是 CP（强一致，如银行）或 AP（高可用，如社交媒体）。

项目题拆解（直播课配套教程）

为什么项目题难？

不是技术不行，而是不会讲。面试官听项目，要的不是代码流水账，而是：

  1. 你在团队中的具体角色和贡献
  2. 你做了哪些技术决策，为什么
  3. 遇到了什么挑战，怎么解决的
  4. 量化结果 — 性能提升了多少？用户增长了多少？

项目表达结构（3-5 分钟）

1. 一句话概括 (30s)
   "我在 XX 公司负责了一个 XX 系统，核心解决 XX 问题"

2. 技术架构 (60s)
   前端 React + 后端 Node.js + MongoDB，部署在 AWS

3. 关键决策 (60s)
   为什么选 MongoDB 不选 PostgreSQL？
   → 数据结构灵活、快速迭代、Schema 经常变化

4. 技术挑战 (60s)
   遇到什么难题？怎么解决的？
   → API 响应慢 → 加 Redis 缓存 → 响应时间从 800ms 降到 120ms

5. 结果量化 (30s)
   → 支撑了 5000 DAU，零宕机运行 6 个月

权衡取舍的表达（高级感）

初级回答：「我们用了 React」

高级回答：「我们对比了 React 和 Vue。选 React 因为：1) 团队有经验 2) 需要 SSR，Next.js 生态成熟 3) 组件库丰富。代价是学习曲线稍高。」

常见追问及回答策略

追问考察点回答要点
如果重来你会怎么做？反思能力说一个具体改进点，展示成长
最难的部分是什么？技术深度选技术挑战，不选沟通困难
你在团队中的角色？个人贡献说具体决策和推动的改变
性能瓶颈怎么排查？问题解决能力工具 + 方法论 + 具体结果

AI Essentials + 模拟面试示范（直播课配套教程）

通过型回答的结构与节奏

面试不是考试，是对话。通过型回答有三个特征：

  1. 结论先行 — 先给答案，再展开解释
  2. 分层清晰 — 按逻辑层次讲，不跳跃
  3. 知道边界 — 不确定的地方说出来，不瞎编

真实面试流程还原

1. 读题 (30s)
   面试官出题 → 你先读/听完整 → 不要急着答

2. 澄清 (30s-1min)
   "Let me make sure I understand correctly..."
   确认边界条件、约束

3. 思考 (1-2min)
   "Give me a moment to think about this..."
   组织思路，不要沉默太久

4. 回答 (2-5min)
   结论先行 → 分层解释 → 代码/示例 → 复杂度分析

5. 追问 (1-3min)
   面试官深入追问 → 如果不知道就说 → 展示思考过程

6. 总结 (30s)
   "So to summarize, the key trade-off here is..."

卡壳救场技巧

场景救场方法
完全没思路说出你的思考方向，让面试官引导
说了一半发现有错直接说「Wait, I think there's an issue...」然后纠正
不确定答案对不对用例子验证，或说「I believe... but let me verify」
面试官追问到不会的地方说实话：「I haven't worked with that specifically, but based on what I know about X...」

常用面试英文表达

  • 开始回答：「That's a great question. Let me walk you through my approach.」
  • 需要时间：「Could I take a moment to think about this?」
  • 澄清问题：「Just to clarify, are we assuming...?」
  • 不确定：「I'm not 100% sure, but my understanding is...」
  • 总结：「To sum up, the main trade-off is between X and Y.」

AI 面试新趋势（直播课配套教程）

2026 AI Coding Interview 考什么？

澳洲越来越多的公司在面试中加入 AI 相关题目。不是要你训练模型，而是考你能不能用 LLM API 构建实际功能。

常见题型

  1. API 调用题 — 给需求，现场写调用 OpenAI/Claude API 的代码
  2. Prompt 设计题 — 给复杂任务，设计 System Prompt + User Prompt
  3. 架构设计题 — 把 AI 功能集成到现有系统
  4. 权衡讨论 — 成本 vs 质量 vs 延迟的选型

AI 编程的使用边界

面试中用 AI 辅助写代码（如 Copilot）越来越被接受，但你必须：

  • 能解释生成的代码 — 面试官会追问每一行
  • 能发现错误 — AI 生成的代码可能有 bug
  • 知道什么时候不该用 — 算法题核心逻辑要自己写

LLM API 基础（面试必知）

// OpenAI Chat Completion
const response = await openai.chat.completions.create({
  model: 'gpt-4o',
  messages: [
    { role: 'system', content: '你是一个代码审查助手' },
    { role: 'user', content: '审查这段代码: ...' }
  ],
  temperature: 0.3,  // 低温度 = 更确定性
  max_tokens: 1000
});

关键参数

参数作用面试考点
temperature控制随机性（0=确定, 1=创意）代码生成用低温度，创意写作用高温度
max_tokens限制输出长度影响成本和响应速度
top_p核采样（和 temperature 类似但不同机制）通常不同时调 temperature 和 top_p
stop停止序列用于控制输出格式

面试追问应对

  • 「AI 生成的代码有 bug 怎么办？」— 单元测试 + Code Review + 人工验证
  • 「Hallucination 怎么处理？」— RAG（检索增强）+ 事实验证 + 置信度阈值
  • 「成本怎么控制？」— 选合适的模型（不是所有任务都需要 GPT-4）+ 缓存 + Prompt 优化减少 token
