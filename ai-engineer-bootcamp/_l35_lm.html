核心概念

为什么 Prompt Engineering 不够了

Prompt Engineering 解决的是"一条消息怎么写"的问题。你精心设计 System Prompt、加 Few-shot 示例、用 CoT 引导推理——这些都很有效，但只覆盖了单次交互。

真实的 AI 应用远不止单次交互：

  • 用户问了 50 轮之后，前面的对话已经超出 context window，模型"忘了"你们讨论过什么
  • 用户问的问题需要私有数据，模型的训练数据里没有你公司的内部文档
  • 用户需要 AI 执行操作，比如查数据库、发邮件、搜网页——模型只能生成文本，做不了这些

Context Engineering 就是解决这些问题的系统方法。

Gartner 2026 预测：57% 的 AI Agent 项目失败的根本原因是上下文管理失败——不是模型不够好，而是喂给模型的信息不对。

Context Engineering 定义

Context Engineering = 为 AI 在每次交互时构建最优上下文的艺术和科学。

类比：Prompt Engineering 像写好一封邮件，Context Engineering 像管理整个信息系统。你不只是写好问题，还要决定：给 AI 看哪些文档？加载哪些工具？保留多少对话历史？这些信息怎么排列？总共不能超过多少 token？

四种上下文来源

来源说明示例特点
System InstructionsAI 的"人设"和规则角色定义、安全边界、输出格式固定，每次都加载
User Input用户当前的请求问题、上传的文件、选择的选项每次不同
Tool Results工具调用返回的数据API 响应、数据库查询结果、搜索结果按需加载，可能很大
Memory历史对话和持久化信息对话摘要、用户偏好、长期记忆需要压缩管理

这四种来源贯穿整个 Bootcamp：Phase 2 做 Knowledge Context（RAG 检索文档），Phase 3 做 Tool Context（Agent 调用工具），Phase 4 做模型层优化。

上下文窗口 = "预算"

每个模型都有 context window 限制。GPT-4o 是 128K tokens，Claude 是 200K，Gemini 是 1M。听起来很大，但实际工程中你很快就会发现不够用。

像管理预算一样管理 context window：

总预算: 128K tokens
├── System Prompt:     ~2K  (固定开销，每次都有)
├── 用户画像摘要:       ~1K  (动态，从用户数据库拉取)
├── 最近对话历史:       ~4K  (滑动窗口，保留最近 5-10 轮)
├── 知识库检索结果:     ~8K  (RAG 检索的 top-5 文档片段)
├── 工具调用结果:       ~4K  (Agent 执行工具后的返回值)
├── 当前用户输入:       ~1K  (必须保留，不能砍)
└── 预留给 AI 回答:    ~4K  (输出空间)
    ──────────────────
    已用: ~24K / 128K

预算管理四原则

  1. System Prompt 和当前任务最高优先级 — 没有 System Prompt，AI 会"忘记"自己的角色和规则。当前任务信息直接决定回答质量。考试时你会优先看题目要求和参考资料，而不是把笔记本全搬进考场。
  2. 压缩而不是丢弃 — 100 轮对话不要全塞进去（那是几万 token），让 AI 每 10 轮生成一份摘要（几百 token），关键信息保留，细节丢弃。摘要保留的是：关键决策、用户偏好、重要事实。丢弃的是：寒暄、重复、过时信息。
  3. 按需加载 — 不要预先塞入所有可能用到的文档。用户问什么再去检索相关内容。这就是 RAG 的核心思想。
  4. 预留输出空间 — 别把 context window 用满，留 10-20% 给 AI 回答。

动态上下文组装

客服 Agent 完整示例

每次用户发消息，你的代码需要动态组装上下文。不是每次都加载所有信息——订单信息只在用户提到"订单""退款"时才查询：

def build_customer_service_context(user_id, message):
    # 1. 固定部分 — 每次都加载
    system_prompt = load_system_prompt("customer_service_v3.txt")  # ~500 tokens

    # 2. 动态 — 用户画像（从数据库拉）
    profile = db.get_user_profile(user_id)
    profile_text = f"用户：{profile.name}，{profile.level}，历史问题数：{profile.ticket_count}"

    # 3. 动态 — 最近对话（滑动窗口，只保留最近 5 轮）
    recent_messages = get_recent_messages(user_id, last_n=5)

    # 4. 动态 — 知识库检索（根据用户当前问题搜索 FAQ）
    relevant_docs = rag_search(message, top_k=3)

    # 5. 条件加载 — 只在用户提到订单时才查
    order_info = ""
    if "订单" in message or "退款" in message:
        orders = db.get_recent_orders(user_id, limit=3)
        order_info = format_orders(orders)

    return {
        "system": system_prompt,
        "context": f"{profile_text}\n\n{relevant_docs}\n\n{order_info}",
        "messages": recent_messages + [{"role": "user", "content": message}]
    }

代码助手示例

对比一下不同场景的上下文组装策略——代码助手和客服完全不同：

每次用户发消息触发：
1. [固定] System Prompt (代码规范 + 技术栈约束)
2. [动态] 当前打开的文件内容（最相关的上下文）
3. [动态] 相关文件（通过 import/引用分析自动找到）
4. [动态] 最近 5 轮对话
5. [动态] 错误日志（如果在 debug 模式）
6. [输入] 用户的代码问题

关键差异：客服 Agent 需要用户画像 + 知识库，代码助手需要当前文件 + 关联文件。没有万能的上下文组装方案，每个场景都要单独设计。

用 LangChain RunnableParallel 实现

手动拼字符串到第三个数据源就崩了——f-string 嵌套、格式对齐、空值处理，全是 bug。RunnableParallel 并行获取多个上下文，代码更清晰，还比串行快 2-3 倍：

from langchain_core.runnables import RunnablePassthrough, RunnableParallel
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate

def get_user_profile(input_dict):
    user_id = input_dict["user_id"]
    return {"name": "Alice", "level": "VIP", "history": "曾投诉配送慢"}

def search_knowledge_base(input_dict):
    query = input_dict["question"]
    return "退款政策：7天无理由退款，需提供订单号..."

# 并行获取多个上下文数据源
context_chain = RunnableParallel(
    user_profile=RunnablePassthrough() | get_user_profile,
    knowledge=RunnablePassthrough() | search_knowledge_base,
    question=lambda x: x["question"],
)

prompt = ChatPromptTemplate.from_messages([
    ("system", "你是客服助手。\n用户信息：{user_profile}\n相关知识：{knowledge}"),
    ("user", "{question}")
])

chain = context_chain | prompt | ChatOpenAI(model="gpt-4o")
result = chain.invoke({"user_id": "u123", "question": "我要退款"})

我们的客服系统从串行改并行后，P99 延迟从 3.2 秒降到 1.4 秒。

量化对比：纯 Prompt vs 完整 Context Engineering

我们在客服场景上做了对比测试（50 条测试问题，3 种方案）：

方案FaithfulnessAnswer Relevancy平均延迟月成本
V1: 纯 System Prompt（无 RAG）0.450.621.2s$200
V2: System Prompt + RAG0.780.852.1s$350
V3: 完整 Context Engineering0.910.931.8s$400

V3 包括：分层 System Prompt + 用户画像注入 + RAG 检索 + 对话摘要 + 动态预算分配。比 V2 多花 15% 成本，但 Faithfulness 从 0.78 提升到 0.91。

延迟反而比 V2 低（1.8s vs 2.1s）——因为动态预算控制减少了不必要的检索内容，LLM 处理的 token 更少。

为什么 V1 这么差

纯 System Prompt 方案下，AI 只有训练数据里的通用知识。用户问"我的订单在哪"，AI 会编一个看起来合理的回答，但完全是错的（幻觉）。Faithfulness 只有 0.45 意味着超过一半的回答不可信。

V2 → V3 的关键提升

V2 加了 RAG，但每次都检索固定 5 个文档片段，不管是否相关。V3 的改进：

  • 用户画像 — 知道是 VIP 用户后，优先推快速退款通道
  • 对话摘要 — 长对话不丢上下文，不用用户重复自己说过的话
  • 动态预算 — 简单问题不做 RAG 检索（省 token），复杂问题检索更多文档

CLAUDE.md Pattern：项目级上下文注入

Claude Code 的 CLAUDE.md 是实际项目中最优雅的 Context Engineering 实践——分三层注入上下文：

项目根目录/
├── CLAUDE.md          # 项目级（技术栈、编码规范、架构约束）
├── src/
│   └── modules/
│       └── CLAUDE.md  # 模块级（该模块的特殊规则）
└── .claude/
    └── settings.json  # 用户级（个人偏好）

同样的分层逻辑适用于任何 AI 应用：

  • 你的 System Prompt = CLAUDE.md（全局规则）
  • RAG 检索结果 = 模块级上下文（领域知识）
  • 用户画像 = settings.json（用户偏好）

我们拆分成分层结构后，AI 遵循规则的准确率从 72% 提升到 91%。核心原因是：规则分层后每一层更短、更聚焦，模型更容易"记住"和遵循。

常见错误

  • ❌ 把所有规则都塞在一个巨大的 System Prompt 里 — 超过 2000 tokens 后模型对中间内容的注意力显著下降
  • ❌ 128K 的窗口就用 128K — 我们实测超过 20K tokens 后效果开始下降
  • ❌ System Prompt 里写"你是一个专业的、负责任的、有创造力的助手" — 这种废话占 token 但零信息量。2000 token 的 Prompt 拆开看，真正有用的往往不到 200 token

常见问题

Q: Context window 越大越好吗？128K 够不够？

不是越大越好。研究表明 context 超过 20K tokens 后，模型对中间内容的注意力显著下降——这就是 "Lost in the Middle" 问题。128K 的窗口不意味着你应该用 128K。实际工程中，保持有效 context 在 10-30K 范围内效果最好。关键信息放在开头或结尾，模型对这两个位置的注意力最强。

Q: 四种上下文来源都要用吗？

看场景。简单的问答机器人只需要 System Instructions + User Input。加 RAG 就有了 Knowledge Context。做 Agent 才需要 Tool Context。做长对话才需要 Memory。从简单开始，效果不够再加——跟选 AI 模式（Prompting → RAG → Agent）的思路一致。

Q: 怎么知道上下文组装的质量好不好？

三层评估，组合使用最可靠：

  1. LLM-as-Judge（快，日常迭代用）：用 GPT-4o 对两种策略的输出打分，100 条测试跑 5 分钟出结果
  2. 人工 Review（准，做校准）：每轮抽 20 条人工评分。我们发现 LLM-as-Judge 偏好长回答，加人工校准后才发现这个偏差
  3. 用户反馈（最终拍板）：A/B 测试跑 2 周看满意度和任务完成率

Q: 对话历史怎么管理？全部塞进去还是只保留最近几轮？

都不对。正确做法是滑动窗口 + 摘要：保留最近 5-10 轮完整对话，更早的做摘要压缩。每 10 轮让 AI 生成摘要，保留关键决策和用户偏好，丢弃寒暄和重复。这样即使 100 轮对话后，第 1 轮的关键信息仍在摘要中，token 成本只有原始的 10%。

小结

记住这几点就够了：

  1. Context Engineering 不是写 Prompt，是设计系统 — 决定给 AI 看什么、不看什么、按什么顺序看
  2. 四种来源：System Instructions / User Input / Tool Results / Memory，按需组合
  3. 上下文是有限资源 — 像管理预算一样管理 token，信号密度比信息量重要
  4. 按需加载 — 不要预先塞入所有信息，用户问什么再检索什么
  5. 量化驱动 — 每次改动都跑评估，用数据说话而不是凭感觉
